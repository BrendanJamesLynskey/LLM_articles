CONTEXT

The PPO Era and Its Pain

For three years “RLHF” was synonymous with PPO. The pipeline worked but was complicated, expensive, and finicky. Few teams outside the big labs could afford it.

Reward Model

  • Train an LLM to score responses
  • Bradley-Terry loss on preference pairs
  • Overfits, leaks, hacks
  • Reward hacking by the policy

PPO Loop

  • 4× model copies in memory
  • Policy + reference + RM + value fn
  • 10–30× SFT compute cost
  • ~12 hyperparameters to tune

Outcome

  • Worked — produced ChatGPT, Claude
  • But high barrier to entry
  • Most teams stuck with SFT
  • Pent-up demand for simpler methods

DPO

Direct Preference Optimization

Key Insight

The optimal RLHF policy has a closed form in terms of the reference policy and the reward function. Invert it: any policy implies a reward function. So you can optimize the policy directly on preference data — no reward model needed.

Loss

L = -log σ(β · [logπ(y_c|x)/π_ref(y_c|x) − logπ(y_r|x)/π_ref(y_r|x)])

Increase log-ratio for chosen, decrease for rejected.

What Disappears

  • No reward model
  • No value function
  • No on-policy rollouts
  • No PPO clipping
  • Just supervised training on pairs

What You Get

  • ~300 lines of code
  • Stable training
  • Single hyperparameter (β)
  • Quality competitive with PPO

LIMITS

Where DPO Falls Short

Offline Only

  • Learns from fixed preference dataset
  • Cannot explore new responses
  • Limited to support of training data
  • Misses opportunities outside distribution

Wrong Signal for Reasoning

  • Math/code have verifiable rewards
  • Preference pairs discard information
  • Just tell the model the answer is right
  • DPO under-uses this

The Gap GRPO Filled

For reasoning tasks — math, code, theorem proving — you don’t need a learned reward model and you don’t need preference pairs. You need an on-policy method that can explore and an objective scorer for the rollouts. That’s GRPO.

dpo-grpo / 05
GRPO

Group Relative Policy Optimization

The Setting

  • Tasks with verifiable rewards
  • Math problems with known answers
  • Code with unit tests
  • Logical reasoning with verifiers

The Procedure

  • Sample G responses per prompt (8–64)
  • Score each with verifier (0/1 or partial)
  • Compute advantage relative to group mean
  • Update policy — up if above-average, down if below

Loss

L = −E[ratio · A − β · KL]

A = (r − meangroup) / stdgroup

Innovations vs PPO

  • No value function (group mean = baseline)
  • No reward model (verifier instead)
  • Group-normalized advantage
  • ~2× less memory than PPO

DEEPSEEK

The DeepSeek-R1 Moment

DeepSeek-R1 demonstrated that GRPO with binary correctness reward could elicit complex reasoning behaviors — chain-of-thought, self-correction, multi-path exploration — with no explicit supervision of the reasoning process itself.

What Surprised People

  • No labeled CoT data needed
  • Reasoning emerged from outcome reward alone
  • Process reward models underperformed outcome-only
  • Recipe is shockingly simple

The Recipe

  • Collect math + code problems with verifiable answers
  • Run GRPO with binary reward
  • Mix in some general SFT for stability
  • Iterate

RECIPES

Practical Recipes

For Chat / Instruction

  • Use DPO (or SimPO/ORPO)
  • Preference data: UltraFeedback, HelpSteer
  • Tools: TRL, OpenRLHF, Axolotl
  • Add SFT regularization

For Reasoning / Code

  • Use GRPO
  • Verifiable tasks: math, coding, theorem proving
  • Tools: TRL, verl, OpenRLHF
  • Sample G=8–32 per prompt

For Safety

  • DPO with safety expert preferences
  • Some teams use full PPO for highest stakes
  • Constitutional AI variants

For Agents

  • GRPO with task success reward
  • Auxiliary SFT for tool format compliance
  • Long horizons need credit assignment

SUMMARY

The Bigger Picture

The Pattern

  • PPO → DPO: kill the reward model
  • DPO → GRPO: keep on-policy, kill the value function
  • Each transition removed moving parts and reduced cost
  • Yet quality has held or improved at every step

The Lesson

The methods that win in practice are usually the simplest ones that achieve the desired result. LLM alignment is, at its core, simpler than the early RLHF infrastructure suggested. Once the right loss function is found, much of the complexity falls away. Further simplifications are likely still ahead.
