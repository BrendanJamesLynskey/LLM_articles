CONTEXT

The Wasteful Uniformity

A standard transformer applies the same compute to every token. Easy tokens (“the”, “a”) get the same processing as hard tokens (“transmogrification”). The compute spent on easy tokens is mostly wasted.

Standard Transformer

  • Every token through every layer
  • O(L × N) work per sequence
  • No mechanism to redistribute compute
  • Hardware-friendly but wasteful

The Idea

  • Let each layer pick which tokens to process
  • Skip easy tokens, attend to hard ones
  • Same total compute, better-spent
  • Or: less total compute, same quality
mod / 02
MOD

The MoD Mechanism

At Each Layer

  1. Small router scores each token
  2. Top-k tokens selected for full processing
  3. Other tokens skip the layer (residual passthrough)
  4. Different layers may pick different tokens

Top-k Choice

  • k=12.5%: only 1 in 8 tokens processed per layer
  • k=50%: half processed
  • Smaller k = more savings, lower quality
  • Differentiable via soft top-k during training

Compute Saved

(1 − k)× reduction in per-layer FLOPs. For k=0.5, ~50% savings; for k=0.125, ~87.5% savings.

RESULTS

What MoD Delivers

The Claim

  • MoD models match standard transformers at the same compute budget
  • Spending compute more efficiently
  • On the tokens that need it

Subsequent Findings

  • Slight underperformance at equal token count
  • Outperformance at compute-constrained regimes
  • Crossover depends on task

CHALLENGES

Implementation Pain

Routing Overhead

  • Router itself adds compute
  • Memory bandwidth cost for router weights
  • Eats into the savings

Load Imbalance

  • Different layers select different tokens
  • GPU work distribution becomes uneven
  • Same problem as MoE

KV Cache & Causal Masking

  • Skipped tokens leave gaps in cache
  • Workarounds keep cache dense at cost of some savings
  • Custom kernels needed for full benefit

Training Stability

  • Router can collapse to degenerate routing
  • Auxiliary losses required
  • Warm-up schedules
  • Similar to MoE pain

DEPLOY

Production Status

2026 Reality

  • Still research-stage
  • No public frontier model uses MoD as primary architecture
  • Mentioned in ablations and exploration
  • Hasn’t broken through

Why Not

  • Compute savings less dramatic than alternatives
  • Engineering complexity
  • Training stability issues
  • Standard transformer is “good enough”
mod / 07
HARDWARE

The Hardware Mismatch

GPUs Want Uniformity

  • Tensor cores expect rectangular matmuls
  • Memory access best when predictable
  • Branch prediction works on consistent control flow
  • Adaptive compute fights all of this

The Translation Gap

  • 50% fewer FLOPs ≈ only 30–40% faster wall-clock
  • Irregular compute reduces hardware efficiency
  • Theoretical savings > practical savings

Future Hope

Hardware that natively supports sparse and dynamic compute would close the gap. None commercial yet.

The Big Question

Is uniform compute the right default? The success of standard transformers suggests “yes for now.” The success of MoE suggests “not entirely.” The mixed reception of MoD suggests the right kind of non-uniformity matters more than the existence of non-uniformity.

mod / 09
SUMMARY

An Important Data Point

What MoD Demonstrates

  • Adaptive compute can work for transformers
  • Engineering challenges are tractable
  • Meaningful efficiency gains achievable
  • But the standard transformer is hard to beat

Practitioner’s Lesson

Worth knowing about, not yet worth deploying. Standard inference techniques (FlashAttention, GQA, paged attention, batching) get you better cost reduction at lower complexity. MoD becomes interesting when those are exhausted and the only remaining axes of improvement are architectural. That moment may come, just not yet.
