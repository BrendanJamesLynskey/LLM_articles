Why Quantization Matters

Memory Bandwidth Bottleneck

Autoregressive decode is memory-bandwidth bound — each token requires reading the full model weights from GPU memory with little compute per byte loaded.

  • 16-bit → 8-bit: 2x effective bandwidth
  • 16-bit → 4-bit: 4x effective bandwidth

Memory Capacity & Accessibility

A 70B parameter model in FP16 requires ~140 GB — exceeding any single consumer GPU.

  • INT4 reduces this to ~35 GB
  • Fits on a single 48 GB GPU
  • Democratizes frontier-class models

Uniform vs. Non-Uniform Quantization

Uniform Quantization

  • Maps values to evenly spaced integer levels
  • Defined by a scale factor and zero-point
  • Hardware-friendly — dominates practical deployment
  • Simple dequantization logic

Non-Uniform Quantization

  • Unevenly spaced levels based on value distribution
  • Higher precision for frequently occurring values
  • Can achieve better accuracy theoretically
  • Less hardware support, more complex

In practice, uniform quantization is the dominant approach for LLM deployment due to hardware compatibility.

Weight-Only vs. Weight-and-Activation

Weight-Only Quantization

  • Quantizes only static weights
  • Activations remain in FP16/BF16
  • Careful offline calibration possible
  • Minimal quality impact
  • Most widely used for LLM inference

Weight + Activation (W8A8, W4A8)

  • Quantizes dynamic activations at runtime
  • Enables INT8 / FP8 tensor core usage
  • Additional speedups from integer matmul
  • Harder — activations have outliers
  • Requires more careful calibration

Quantization Granularity

Per-Tensor

Single scale & zero-point for the entire matrix. Simple but loses precision when ranges vary across channels.

Per-Channel

Separate parameters for each output channel. Better range capture per channel with modest overhead.

Per-Group

Independent parameters per group of 32–128 elements. Finest granularity, best accuracy, slightly more complex dequantization.

GPTQ

One-Shot Weight Quantization

  • Based on approximate second-order information
  • Processes weight matrix column by column
  • Quantizes each column, then adjusts remaining columns to compensate for error
  • Uses approximate inverse Hessian from calibration data
  • Can quantize a large model in hours on a single GPU

Key Properties

  • Produces INT4 / INT3 models
  • Remarkably low perplexity degradation
  • Standard method in open-source community
  • Optimized kernels via Marlin

AWQ — Activation-Aware Weight Quantization

Key Insight

Not all weights are equally important for model accuracy. AWQ identifies salient weight channels by examining activation magnitudes from a calibration set.

Method

  • Identify important channels via activation statistics
  • Apply per-channel scaling to protect salient channels
  • Concentrate quantization error on less important channels

AWQ vs. GPTQ

  • Better accuracy than naive RTN at same bit width
  • Lower compute cost than GPTQ
  • No second-order computation needed
  • Simpler calibration pipeline

Mixed-Precision: SqueezeLLM & SpQR

SqueezeLLM

  • Sensitivity-based non-uniform quantization
  • Uses a lookup table for level mapping
  • Outlier weights stored at higher precision (FP16)
  • Majority quantized to 3-bit or 2-bit

SpQR

  • Isolates outliers via second-order sensitivity analysis
  • Sensitive weights kept in FP16
  • Remaining weights at very low precision
  • Competitive accuracy at extremely low avg. bit widths

Both methods achieve strong results by treating outlier weights differently from the bulk distribution.

FP8 Quantization

E4M3 (Weights)

  • 4 exponent bits, 3 mantissa bits
  • Better precision for weight distributions
  • Preferred format for model weights

E5M2 (Activations)

  • 5 exponent bits, 2 mantissa bits
  • Wider dynamic range for activations/gradients
  • Handles outlier values better

Hardware Support

FP8 is natively supported on NVIDIA Hopper and Blackwell GPUs. FP8 tensor cores deliver the same throughput as INT8 while naturally handling wide dynamic ranges better than integer formats. FP8 has become the preferred 8-bit format for inference on modern NVIDIA hardware.

Quantization-Aware Training & QLoRA

Quantization-Aware Training (QAT)

  • Incorporates quantization simulation into training
  • Model learns to compensate for quantization error
  • Generally better accuracy than post-training quantization
  • Requires training data and significant compute
  • Most commonly applied during fine-tuning

QLoRA

  • Combines LoRA with 4-bit quantized base weights
  • Fine-tune large models on consumer hardware
  • Quality approaches full-precision fine-tuning
  • Uses NF4 (Normal Float 4) format

Calibration: The Underappreciated Factor

Why Calibration Matters

  • Determines scale factors and zero-points for quantization
  • Calibration data should be representative of deployment distribution
  • Using mismatched data leads to suboptimal quantization decisions

Common Pitfall

Calibrating with general web text for a model that processes code or domain-specific content.

Practical Guidelines

  • Only 128–512 samples needed
  • Curate domain-specific calibration sets
  • Match calibration to deployment distribution
  • Small investment, significant quality payoff

Per Method

  • GPTQ: Hessian from calibration data
  • AWQ: Activation magnitudes from calibration
  • FP8: Per-tensor scaling from calibration ranges

Runtime Kernel Support

Theoretical memory savings only translate to actual speedups with optimized dequantization and matmul kernels.

Library Format Key Feature
Marlin GPTQ (INT4) Optimized INT4 dequant + matmul kernels
CUTLASS FP8 NVIDIA's template library for FP8 tensor core ops
bitsandbytes NF4 Normal Float 4-bit for QLoRA fine-tuning

Warning

Choosing a quantization format without corresponding kernel support can negate much of the potential benefit. Always verify runtime support before committing to a format.
