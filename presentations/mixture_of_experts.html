What is Mixture of Experts?

MoE is an architecture paradigm that decouples model capacity from computational cost. Each token is routed to a small subset of specialized sub-networks (experts), activating only a fraction of total parameters per forward pass.

  • Scale to very large total parameter counts
  • Keep per-token compute manageable
  • Achieve dense-model quality at a fraction of FLOPS

Core Insight

Sparse activation lets you have the representational capacity of a huge model while paying the compute cost of a small one.

Structure of an MoE Layer

Standard FFN replaced with N experts + router. Only top-k experts activated per token (k=1 or 2).

mixture-of-experts / architecture
03

Sparsity & Effective Parameters

The Key Advantage

Total parameter count is very large, but active parameters per token are much smaller. Inference compute scales with active params, not total params.

Mixtral 8x7B Example

  • 47B total parameters across 8 experts
  • Each token activates 2 of 8 experts per layer
  • ~13B active parameters per token
  • Dense-13B compute, 47B capacity

Scale Precedents

Switch Transformer and GShard demonstrated this principle at even larger scales—top-1 routing across hundreds of experts, reaching up to 1.6 trillion total parameters.

04

Token-Choice Routing

The most common routing approach: each token independently selects its top-k experts based on router output probabilities.

  • Router is a simple linear layer
  • Projects token representation into per-expert scores
  • Applies softmax then top-k selection
  • Used by Mixtral, Switch Transformer, most modern MoE models

Router Function

scores = W_gate @ x    # linear
probs  = softmax(scores)
top_k  = select_top(probs, k=2)
output = sum(w_i * expert_i(x))

05

Expert-Choice Routing

How It Works

Inverts the selection: each expert selects the top-k tokens it wants to process from the full batch.

  • Perfect load balance by construction
  • Every expert processes the same number of tokens

Trade-offs

  • Some tokens may be selected by zero experts
  • Others by many experts
  • Variable processing can affect quality
  • Adds complexity to serving systems

Shared & Fine-Grained Experts

Shared Experts

Always-active experts that process every token, providing a baseline level of processing alongside routed experts.

  • Guarantees minimum computation per token
  • Used by DeepSeek-V2 and V3

Fine-Grained Experts

Smaller but more numerous experts allow more precise routing decisions and better load distribution.

  • More routing flexibility
  • Better utilization across devices

Hybrid Formula

output = shared_expert(x) + Σ wi · routed_experti(x)

Training: Load Balancing

The Problem

Without intervention, routing collapses: a few popular experts get most tokens while others are starved. This wastes capacity and creates compute bottlenecks.

The Solution

Auxiliary load-balancing loss penalizes uneven expert utilization, added to the main training objective.

  • Too little → routing collapse
  • Too much → forced uniformity, no specialization
  • Just right → balanced, meaningful experts

08

Training: Expert Specialization

Effective MoE models develop experts that specialize in different aspects of the input distribution.

Languages

Experts may specialize in particular languages or language families.

Domains

Code, math, science, or conversational text may each gravitate to different experts.

Syntax

Certain syntactic patterns or token types may preferentially route to specific experts.

Key Observations

  • Specialization varies across layers—early layers show less, later layers show more
  • The nature of specialization is difficult to control directly—it emerges from training dynamics
  • Understanding and encouraging useful specialization remains an active research area

Inference: Memory Requirements

Despite computational efficiency, MoE models have large total parameter counts that must reside in memory. All expert weights must be accessible even though only a subset is active per token.

The Asymmetry

Compute scales with active params, but memory scales with total params. This is the central deployment challenge.

mixture-of-experts / memory
10

Inference: Batching Complexity

Different tokens in the same batch may route to different experts, creating uneven workloads and complex communication patterns.

1. Dispatch

Tokens are routed to the GPU hosting their selected expert via all-to-all communication.

2. Compute

Each expert processes its assigned tokens. Workloads may be uneven across devices.

3. Gather

Results are sent back to the original token positions via another all-to-all exchange.

Expert Parallelism

Different experts placed on different devices. Efficient implementation of the dispatch–compute–gather pattern is critical for serving performance.

mixture-of-experts / batching
11

Expert Offloading

When GPU memory is insufficient, experts can be loaded on demand from CPU memory or storage.

  • Router determines needed experts before computation
  • Prefetching loads experts while other computation proceeds
  • Enables MoE inference on single consumer GPUs

Notable Systems

  • MoE-Offloading
  • Fiddler

Acceptable throughput with careful scheduling, though latency increases vs. fully GPU-resident.

Notable MoE Models

Switch Transformer

Top-1 routing at scale, up to 1.6T parameters. Proved sparse scaling works.

Mixtral 8x7B / 8x22B

Open-weight MoE by Mistral AI. Matched or exceeded much larger dense models.

DeepSeek-V2 / V3

Fine-grained experts + shared expert design. State-of-the-art efficiency.

Grok-1

Large MoE architecture by xAI for conversational AI system.

MoE has moved from research curiosity to mainstream architectural choice for frontier models.

Future Directions

Finer-Grained Experts

Larger expert counts with smaller experts improve routing precision and specialization.

Hybrid Architectures

Mix dense and MoE layers within a single model. Place MoE where sparsity helps most.

Improved Routing

Learned and RL-based routers aim to better balance specialization with load distribution.

Optimized Infrastructure

Custom kernels and communication libraries for the MoE dispatch pattern. Faster interconnects and flexible memory hierarchies.

15

Key Takeaways

  • MoE decouples model capacity from per-token computation
  • Routing (token-choice, expert-choice, shared) is the critical design axis
  • Load balancing requires careful auxiliary loss tuning
  • Memory scales with total params, compute with active params
  • Expert parallelism and offloading solve the deployment puzzle
  • Frontier-quality at a fraction of the cost of dense equivalents

Bottom Line

MoE architectures are poised to become the dominant architecture for large-scale language model deployment as infrastructure matures.
