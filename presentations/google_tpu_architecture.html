CORE ARCH

Systolic Array & Matrix Multiply Unit

Why Systolic?

  • Data reuse minimizes HBM reads
  • Pipelined: high throughput at low power
  • 128x128 = 16,384 MACs per cycle
  • BFloat16 native datatype

vs GPU SIMT Model

  • GPUs: flexible thread-level parallelism
  • TPUs: fixed dataflow, higher efficiency on matmuls
  • Trade-off: less flexibility for irregular ops
google-tpu / 03
TPU v4

TPU v4 Chip Architecture

Key v4 Specs

  • 275 TFLOPS BFloat16 per chip
  • 32 GB HBM2e memory
  • ~1.2 TB/s memory bandwidth
  • 6 ICI links per chip (3D torus)
  • 4096-chip pod = 1.1+ exaFLOPS

Pod-Level Scale

A single TPU v4 pod delivers over 1.1 exaFLOPS of aggregate BF16 compute, connected via ICI with no host-network boundary between chips.

TPU v5p

TPU v5p & SparseCore

Performance Leap

  • 459 TFLOPS BF16 per chip (+67% vs v4)
  • 95 GB HBM per chip (3x v4)
  • Enhanced memory bandwidth
  • Pods scale to 8,960 chips
  • Improved ICI bandwidth per link

SparseCore

Dedicated processing unit for embedding table lookups and sparse feature interactions. Offloads sparse ops from MXU, enabling concurrent dense + sparse compute.

  • Critical for recommendation models
  • Reduces embedding bottleneck
  • Frees MXU for dense matmuls

TRILLIUM

Trillium (TPU v6e)

4.7x Compute Gain

Per-chip compute improvement of 4.7x over TPU v5e. The largest generational leap in TPU history.

Energy Efficiency

Higher FLOPS-per-watt ratios reduce operational cost for sustained training runs at datacenter scale.

Enhanced ICI

Improved bisection bandwidth supports larger pod configurations and more efficient collective comms.

Memory Improvements

  • Increased HBM capacity and bandwidth
  • Addresses memory wall for 400B+ parameter models
  • Reduces need for aggressive model sharding

Competitive Position

  • Narrows per-chip gap with H100 and B200
  • Systolic array excels on transformer workloads
  • Early benchmarks show strong training throughput
google-tpu / 06
INTERCONNECT

ICI Mesh Topology

ICI Advantages

  • Uniform fabric across thousands of chips
  • No host-network boundary
  • 6 links/chip in 3D torus
  • Topology-aware XLA collectives

vs NVLink/NVSwitch

  • NVLink: 900 GB/s but intra-node only
  • Inter-node requires InfiniBand/RoCE
  • Bandwidth cliff at node boundary
  • TPU: uniform scaling, lower peak p2p

SCALE

TPU Pods & Multislice Training

Intra-Slice: ICI

Tensor parallelism stays within a slice where ICI provides high bandwidth, low latency. All-reduce completes in microseconds.

Inter-Slice: DCN

Data and pipeline parallelism span slices via datacenter network. Higher latency requires careful partitioning via GSPMD.

JAX / XLA / GSPMD Stack

GSPMD Partitioning

  • Logical mesh abstraction

  • Pax — reference transformer training
  • MaxText — high-performance LLM impl

LIMITS

Limitations & Trade-offs

Ecosystem Lock-in

  • JAX and TensorFlow are first-class citizens
  • PyTorch/XLA lags behind native CUDA PyTorch
  • Operator coverage gaps in PyTorch/XLA
  • Community libraries often CUDA-only
  • Smaller practitioner community

Custom Kernels

  • No CUDA kernel support
  • FlashAttention requires Pallas reimplementation
  • Many efficient attention variants unavailable
  • Pallas learning curve is steep

Deployment Constraints

  • GKE-only — no on-premises option
  • Single-cloud vendor dependency
  • Limited multi-cloud DR strategies
  • Negotiating leverage vs multi-cloud GPU

Debugging & Profiling

  • XLA profiling tools less mature than Nsight
  • HLO graph debugging requires expertise
  • Sharding propagation can be opaque
  • Fewer public training recipes & tutorials

GUIDANCE

When to Choose TPUs

TPUs Excel When

  • Training large transformers (100B+ params)
  • Team already proficient in JAX
  • Workload is dense matmul-dominated
  • Need 4000+ chip synchronous training
  • Cost-sensitive sustained training runs
  • Google Cloud is primary cloud provider
  • Recommendation models (SparseCore advantage)

GPUs Excel When

  • PyTorch-native codebase with custom CUDA kernels
  • Multi-cloud or on-premises requirements
  • Diverse workloads beyond dense matmul
  • Need FlashAttention and ecosystem libraries
  • Team expertise is CUDA-centric
  • Inference with TensorRT-LLM optimizations
  • Research requiring rapid prototyping flexibility

The decision is rarely about raw silicon performance. It hinges on software ecosystem, team skills, deployment constraints, and long-term platform strategy.

SUMMARY

Key Takeaways

Architecture

  • Systolic array (MXU) optimized for dense matmul
  • BFloat16 native with high MXU utilization
  • ICI provides uniform scaling to 8,960+ chips
  • SparseCore addresses embedding bottlenecks

Performance Trajectory

  • v4: 275 TFLOPS, 4096-chip pods
  • v5p: 459 TFLOPS, SparseCore
  • Trillium: 4.7x over v5e
  • Each generation closes gap with NVIDIA

Software

  • JAX + XLA + GSPMD = powerful compilation stack
  • Declarative sharding is ergonomic for large models
  • PyTorch support improving but not yet parity
  • SAX enables competitive inference serving

Strategic Considerations

  • Single-vendor, single-cloud commitment
  • Cost-effective for sustained large-scale training
  • Smaller community vs CUDA ecosystem
  • Best for JAX-native, GCP-committed teams
