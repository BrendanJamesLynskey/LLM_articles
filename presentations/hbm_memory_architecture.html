CONTEXT

Compute Isn’t the Bottleneck

Every product launch hypes FLOPs. But for LLM inference, the actual bottleneck is memory bandwidth. Decode is bandwidth-bound by physics — one multiply-accumulate per weight loaded.

Decode Time Floor

decode_time = (2N bytes) / HBM_BW

N = 70B, HBM = 8 TB/s → 17.5 ms/token
N = 70B, HBM = 3.35 TB/s → 42 ms/token

The Memory Wall

  • Compute grows ~2×/18 mo
  • DRAM bandwidth grows ~1.3×/18 mo
  • Gap widens every generation
  • LLM decode has zero locality to exploit

PHYSICS

Why Bandwidth Wins Over Compute

Decode Math

For each token, must read ~all weights from HBM. Compute time per weight is tens of picoseconds; memory time per weight is hundreds of picoseconds.

  • Memory-bound regime
  • Tensor cores idle most of the cycle
  • FLOPs are wasted

Real-World Impact

  • B200 has ~5× H100 compute
  • But ~2.4× H100 bandwidth
  • Decode speedup tracks bandwidth, not compute
  • ~2.4× in practice for big-model decode

The Big Lesson

Every major LLM inference optimization — KV cache quantization, GQA/MQA, prefix caching, batching, tensor parallelism, disaggregated serving — is about escaping bandwidth limits, not compute limits.

SUPPLY

Manufacturing & Supply Chain

Difficulty

  • 12 DRAM dies stacked vertically
  • Tens of thousands of TSVs at micron precision
  • Copper-to-copper hybrid bonding
  • Yield issues at any layer ruin the stack

Suppliers

  • SK hynix — dominant, first to HBM3e
  • Samsung — qualification challenges
  • Micron — recent entrant, shipping HBM3e to NVIDIA
  • Three-vendor concentration

The Constraint

Blackwell production through 2024–2025 was bottlenecked by HBM availability, not GPU silicon. A B200’s 192 GB of HBM3e represents $3,000–5,000 of memory alone.

CAPACITY

Capacity vs Bandwidth Tradeoff

Per-GPU Capacity

  • H100: 80 GB
  • H200: 141 GB
  • B200: 192 GB
  • Rubin (HBM4): ~384 GB

Per-Rack Capacity

  • NVL72: 13.5 TB
  • Holds 1.8T-param model in FP8
  • 11+ TB for KV cache
  • Enables multi-million-token contexts

Why Both Matter

Capacity decides what models you can fit. Bandwidth decides how fast you can serve them. HBM provides both, but the relative scaling is uneven across generations.

hbm / 07
POWER

Power & Thermal Cost

Per-Stack Power

  • 30–50 W per HBM3e stack at full BW
  • 8 stacks → 250–400 W of HBM alone
  • Comparable to GPU compute power
  • One reason Blackwell needs liquid cooling

Co-Packaging Constraint

  • HBM sits on the same package as the hot GPU die
  • Inherits the GPU’s thermal envelope
  • Cooling design must handle GPU + HBM together
  • Direct-to-chip liquid is the dominant solution

PIM

Compute-in-Memory Future

HBM-PIM (Samsung)

  • Compute units in HBM base die
  • Reduction ops directly on data
  • 10–20% gain on memory-bound layers
  • Limited adoption

AiM (SK hynix)

  • GDDR-AiM with MAC units
  • Research demos only
  • No commercial product yet

Why Slow

  • New programming models
  • New compilers needed
  • NVIDIA software inertia
  • Marginal vs full PIM rewrite

SUMMARY

The Bandwidth-First Mindset

What Matters

  • HBM bandwidth, not peak FLOPs, decides decode latency for big models
  • Each new HBM generation roughly doubles bandwidth and adds capacity
  • HBM is the largest single cost component in modern AI accelerators
  • Co-packaging constrains thermal and electrical design
  • PIM is an interesting alternative but stuck behind software inertia

Practitioner’s Lesson

When choosing accelerators for inference, the relevant number is HBM bandwidth, not peak FLOPs. When designing inference architecture, the relevant question is how to minimize bytes moved from HBM, not how to maximize ops performed. This produces dramatically different decisions than the compute-first mental model.
