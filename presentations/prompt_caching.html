CONTEXT

The Redundancy Problem

A surprising amount of LLM compute is wasted re-processing text the model has already seen. A coding agent re-reads the same 50k-token system prompt every turn. A document Q&A tool re-processes the same 100-page contract for every question.

Cost Without Caching

  • Every request runs full prefill
  • Prefill = O(N²) attention + O(N·d) projections
  • For 50k-token prompt + 200-token output, prefill is ~95% of compute
  • Time-to-first-token grows linearly with prefix

Cost With Caching

  • Cached prefill is reused
  • Only suffix tokens incur compute
  • Time-to-first-token drops to constant
  • Provider charges 10–50% of normal input

MECH

How It Works

The KV cache produced during prefill is exactly the data needed to skip prefill on subsequent requests. If prompt = P + S and the cache for P exists, only S needs computing.

1. First Request

  • Run full prefill on P
  • Save resulting KV cache
  • Cache key = hash(P) or trie path
  • Index for future lookup

2. Subsequent Request

  • Lookup prefix in cache
  • Find longest cached prefix matching
  • Load cached KV into GPU
  • Skip prefill for matched portion

3. Continue

  • Run prefill for new tail
  • Decode normally
  • Optionally extend cache with new tail
  • Tail becomes prefix for next request

GRANULARITY

Cache Indexing Strategies

Exact Prefix Match

  • Hash entire prefix
  • Hit only on identical prefix
  • Simple, low overhead
  • Brittle to small variations

Page-Level Match

  • KV cache split into 16–128 token blocks
  • Match per block
  • Integrates with PagedAttention
  • Used by vLLM

Trie / Radix

  • Token-level prefix tree
  • Branching for divergent suffixes
  • Highest hit rates
  • Used by SGLang RadixAttention

Cache Hit Rate Wins

SGLang reports cache hit rates above 95% on benchmark workloads with trie-based indexing — vs ~60% with naive exact-match. The choice of indexing strategy is one of the largest single factors in caching effectiveness.

DESIGN

Cache-Friendly Prompt Structure

Order matters. Cacheable content must come first; variable content must come last. Reordering breaks cache reuse.

Good Pattern

<system prompt>        <-- cache
<tool definitions>     <-- cache
<knowledge base>       <-- cache
<conversation history> <-- cache
<user’s new message>   <-- variable

Bad Pattern

<system prompt>
<user’s new message>   <-- variable, kills cache
<tool definitions>
<knowledge base>

Anthropic API Example

{
  "system": [
    {"type": "text", "text": "You are..."},
    {"type": "text", "text": "Tools: ...",
     "cache_control": {"type": "ephemeral"}}
  ],
  "messages": [...]
}

ECONOMICS

Cost Impact

Coding Agent Example

  • System + tools: 50k tokens
  • User query: 5k tokens
  • Without caching: 55k input/turn
  • With caching (Anthropic):
    50k×0.10 + 5k×1.0 = 10k effective
  • ~80% cost reduction

Document Q&A Example

  • Document: 100k tokens
  • 20 questions, 500 tokens each
  • Without caching: 20×100.5k = 2.01M input
  • With caching: 100k + 20×500.5k = 110k effective (after 1st)
  • ~95% cost reduction

Where It Doesn’t Help

Translation, ad-hoc summarization, unique-document Q&A — workloads where every prompt is novel. Caching depends on prompt repetition; without repetition, there is nothing to cache.

EVICTION

Memory Pressure & Eviction

The Constraint

  • Cached KV consumes precious GPU memory
  • ~5 MB per 1k tokens for large models
  • Memory often binding constraint on serving
  • Tradeoff: cache capacity vs concurrent users

Eviction Policies

  • LRU — standard default
  • LFU for skewed access
  • Length-aware (longer prefixes more valuable)
  • Reuse-prediction-based

Spillover Tiers

  • GPU HBM — ~0 latency overhead on hit
  • CPU memory — +1–5 ms per hit
  • NVMe — +20–100 ms per hit
  • Useful for graceful degradation

Invalidation Triggers

  • Model version changes
  • Tokenizer updates
  • Hardware migration
  • TTL expiration

ADVANCED

Advanced Topics

Cache Warming

  • Send dummy request to populate
  • Amortize write cost
  • Useful before traffic spikes
  • Common pattern for known prefixes

Cross-Tenant Caching

  • Tempting compute savings
  • Information leakage risk
  • Most providers disable
  • Per-tenant isolation default

Distributed Cache

  • Prefix-aware load balancing
  • Same prefix routes to same GPU
  • Higher hit rates at cost of routing complexity
  • Used by major hyperscalers

SUMMARY

The Bottom Line

Why It’s Important

  • 50–90% cost reduction on cached portions
  • Time-to-first-token drops dramatically
  • One of the largest single optimizations available
  • Requires no model changes
  • Available across all major providers and frameworks

Practitioner’s Checklist

1. Structure prompts with stable content first.
2. Opt into caching via API parameters where required.
3. Measure cache hit rate — expect >80% for repetitive workloads.
4. Watch out for invisible cache busters (timestamps, random IDs in early prompt).
