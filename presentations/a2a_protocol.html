a2a-protocol / 01
PROBLEM

Agent Isolation

Before A2A

  • Agents from different vendors cannot communicate
  • N platforms × N platforms = N² custom integrations
  • No standard for agent discovery or capability advertisement
  • No protocol for long-running task delegation
  • Each agent is an intelligent island

With A2A

  • Any agent can discover and communicate with any other agent
  • Each agent implements one A2A endpoint
  • Agent Cards advertise capabilities at well-known URIs
  • Tasks, streaming, and push notifications built in
  • Like USB for agent interoperability

Scope

Announced by Google April 2025. Contributed to Linux Foundation. Version 1.0 reached March 2026. 150+ supporting organizations including Google, Microsoft, AWS, Salesforce, SAP, ServiceNow, LangChain.

a2a-protocol / 02
ARCHITECTURE

Three Actors

User

  • Human operator or automated service
  • Initiates requests through a client agent
  • Interacts via chat, API, voice, etc.

Client Agent (A2A Client)

  • Initiates communication on behalf of user
  • Discovers remote agents via Agent Cards
  • Sends tasks, monitors progress, collects results
  • Typically the orchestrator in multi-agent systems

Remote Agent (A2A Server)

  • Exposes HTTP endpoint implementing A2A
  • Opaque to the client — internal workings hidden
  • Receives tasks, processes them, returns results
  • Can itself act as client to other agents
a2a-protocol / 03
DATA MODEL

Five Core Objects

Agent Card

JSON metadata “business card” describing identity, capabilities, skills, endpoint, and auth requirements.

Task

Stateful unit of work with unique ID. Progresses through defined lifecycle: submitted → working → completed.

Message

Single communication turn. Role is “user” (client) or “agent” (server). Contains one or more Parts.

Part

Smallest content unit. Holds exactly one type: text, file (inline or URI), or data (structured JSON).

Artifact

Tangible output from the remote agent — documents, images, structured data. Supports incremental streaming.

Context

Server-generated contextId grouping related Tasks into a conversational thread for multi-turn interactions.

DISCOVERY

Agent Card

  • Published at /.well-known/agent-card.json (RFC 8615)
  • Identity: name, description, version, provider, icon
  • Service endpoint: url for A2A requests
  • Capabilities: streaming, push notifications, extended card
  • Skills: ID, name, description, tags, examples, input/output MIME types
  • Security: schemes aligned with OpenAPI spec
  • Signature field for authenticity verification (v0.3+)
  • Caching via Cache-Control, ETag, conditional requests

Discovery Strategies

Well-Known URI — HTTP GET to standardized path. Curated Registries — searchable catalogs. Direct Config — hardcoded URLs.

PATTERN

Multi-Turn Interaction

The input-required Pattern

  • Client sends initial message via message/send
  • Server returns task in input-required state with a question
  • Client sends follow-up referencing same taskId and contextId
  • Server continues — may repeat cycle or transition to completed
  • Supports collaborative task completion

Example Scenarios

  • Travel agent asks for date preferences
  • Finance agent requests transaction approval
  • Research agent presents findings, asks direction
a2a-protocol / 07
TRANSPORT

Three Communication Patterns

Request/Response

  • message/send — synchronous
  • Blocking: holds connection until terminal state
  • Non-blocking: returnImmediately: true, then poll via tasks/get
  • Best for quick tasks completing in seconds

Streaming (SSE)

  • message/stream — Server-Sent Events
  • TaskStatusUpdateEvent — state transitions
  • TaskArtifactUpdateEvent — incremental output
  • append and lastChunk flags
  • Best for interactive progress display

Push Notifications

  • Webhooks via pushNotificationConfig/create
  • Server POSTs StreamResponse payloads to client URL
  • Auth via Bearer tokens or mTLS
  • Best for long-running tasks (hours/days)
  • No persistent connection needed
a2a-protocol / 08
PROTOCOL

JSON-RPC 2.0 Methods

Message Operations

  • message/send — send message, get Task response
  • message/stream — send message, get SSE stream

Task Management

  • tasks/get — retrieve task state + optional history
  • tasks/list — paginated listing with filters
  • tasks/cancel — request cancellation
  • tasks/subscribe — SSE stream for existing task

Push Notification Config

  • tasks/pushNotificationConfig/create
  • tasks/pushNotificationConfig/get
  • tasks/pushNotificationConfig/list
  • tasks/pushNotificationConfig/delete
// Request
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "message/send",
  "params": {

      "role": "user",
      "parts": [{

}

// Response

    "artifacts": [{

      "name": "analysis",

        "kind": "text",

a2a-protocol / 09
TRANSPORT

Three Protocol Bindings

JSON-RPC over HTTP

  • Primary binding — all methods via HTTP POST
  • application/json content type
  • JSON-RPC 2.0 request/response envelope
  • SSE for streaming responses
  • Widest compatibility

HTTP/REST

  • Alternative RESTful mapping
  • POST /messages — send
  • GET /tasks/{id} — retrieve
  • POST /tasks/{id}/cancel
  • Familiar to REST API developers

gRPC (v0.3+)

  • Service definition maps ops to RPCs
  • gRPC server streaming for SSE equivalents
  • Service parameters via gRPC metadata
  • High-throughput, low-latency scenarios
  • Polyglot enterprise environments

Service Parameters

A2A-Version header — protocol version (e.g., “1.0”). A2A-Extensions header — comma-separated extension URIs the client supports.

Key Principles

  • Credentials at HTTP transport layer, never in JSON-RPC payload
  • Agents treated as standard enterprise applications
  • Out-of-band credential acquisition
  • Agent Card declares security requirements
  • Aligned with OpenAPI security scheme spec

Authorization Levels

  • Skill-based — OAuth scopes per skill
  • Resource-level — backend access gating
  • Identity-based — per-client policies
a2a-protocol / 11
SECURITY

Transport & Card Integrity

Transport Security

  • HTTPS mandatory in production
  • TLS 1.2+ with strong cipher suites
  • Certificate verification against trusted CAs
  • HTTP error responses: 401 (missing/invalid creds), 403 (insufficient permissions)

Agent Card Signing (v0.3+)

  • signature field for authenticity verification
  • Canonicalized JSON for deterministic signing
  • Prevents tampering during transit or registry storage
  • Clients verify before trusting card contents

Enterprise Recommendations

  • API Management integration for externally exposed agents — centralized rate limiting, quotas, policy enforcement
  • OpenTelemetry with W3C Trace Context for distributed tracing across agent interactions
  • Comprehensive audit logging with taskId, contextId, and correlation IDs
  • Regulatory compliance awareness: GDPR, CCPA, HIPAA

ECOSYSTEM

A2A vs MCP — Complementary Protocols

MCP (Model Context Protocol)

  • Vertical — agent reaches down to tools and data
  • Stateless tool calls
  • Tool manifests for discovery
  • One agent ↔ many tools
  • Anthropic origin, now Linux Foundation

A2A (Agent-to-Agent)

  • Horizontal — agent talks to peer agents
  • Stateful tasks with lifecycle
  • Agent Cards for discovery
  • Agent ↔ agent (opaque delegation)
  • Google origin, now Linux Foundation

USE CASES

Enterprise Scenarios

Multi-Vendor Agent Orchestration

Salesforce customer agents, ServiceNow IT agents, custom LangChain finance agents, and Workday HR agents — all coordinated by a single orchestrator via A2A. “Onboard a contractor, set up their laptop, create a PO” flows through HR → IT → procurement seamlessly.

Travel & Expense Management

User requests a business trip. Orchestrator delegates via A2A to: travel booking agent (flights/hotels), finance agent (policy compliance), payment agent (transactions), currency agent (conversions). Multi-turn interactions handle preferences and approvals.

Supply Chain Optimization

Sales forecasting agent detects demand increase. Notifies supply chain planning agent via A2A, which coordinates with inventory, logistics, and supplier agents to optimize stock levels — no human intervention or custom APIs.

Collaborative Research

Research orchestrator breaks investigation into parallel workstreams. Delegates literature review, data analysis, and competitive landscape to specialists. Collects results via push notifications over hours/days, then synthesizes into a report.

DEEP DIVE

SSE Streaming

How It Works

  • Client sends message/stream
  • Server responds with text/event-stream
  • Each event is a StreamResponse wrapper
  • Contains one of: task, message, statusUpdate, artifactUpdate

Artifact Streaming

  • append: false — replace previous content
  • append: true — append to previous content
  • lastChunk: true — final piece of artifact
  • Enables real-time display of generated output

Error Handling

Error CodeMeaning
TaskNotFoundErrorReferenced task does not exist
TaskNotCancelableErrorTask in non-cancelable state
PushNotificationNotSupportedErrorServer lacks push capability
UnsupportedOperationErrorMethod not implemented
ContentTypeNotSupportedErrorMIME type not accepted
InvalidAgentResponseErrorMalformed agent response
VersionNotSupportedErrorProtocol version mismatch
ExtensionSupportRequiredErrorRequired extension missing

Error Response Format

Standard JSON-RPC 2.0 error object with code, message, and optional data field. Clients should handle errors gracefully and provide actionable feedback to the user or calling agent.

HTTP-Level Errors

  • 401 Unauthorized — missing/invalid credentials; includes WWW-Authenticate header
  • 403 Forbidden — valid credentials but insufficient permissions
  • 404 Not Found — invalid endpoint
  • 429 Too Many Requests — rate limit exceeded

CONCLUSION

Key Takeaways

What A2A Provides

  • Universal interoperability between agents regardless of vendor or framework
  • Agent Cards for capability discovery
  • Stateful tasks with full lifecycle management
  • Three communication patterns: sync, streaming, push
  • Enterprise-grade security from day one
  • Modality-agnostic: text, audio, video, files

The Complete Stack

  • MCP = how agents access tools (vertical)
  • A2A = how agents talk to each other (horizontal)
  • Together: complete interoperability layer for agentic AI
  • Both under Linux Foundation governance
  • 150+ organizations, all major clouds, v1.0 production-ready
  • The HTTP of multi-agent systems

Bottom Line

A2A fills the critical gap between agents in the same way that MCP fills the gap between agents and tools. As multi-agent systems move from prototypes to production, A2A — built on familiar web standards, governed by the Linux Foundation, and backed by every major cloud and enterprise platform — is positioned to become the standard protocol for inter-agent communication.
