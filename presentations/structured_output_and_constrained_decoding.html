The Problem

LLMs are next-token predictors over natural language. Applications need structured data: JSON, SQL, typed function arguments.

  • Prompt-only approaches fail at scale — even 0.1% malformation = thousands of daily errors
  • Trailing commas, unquoted keys, markdown fences wrapping JSON
  • Retry loops add latency and cost
  • Post-hoc repair heuristics are fragile

Failure Modes


Grammar-Based Decoding

CFG + PDA-Guided Token Masking

  • Define target format as a context-free grammar
  • Maintain a pushdown automaton tracking parse state
  • At each step, compute valid token set from PDA state
  • Set invalid token logits to −∞
  • Model samples only from grammatically valid continuations

Key Challenge

BPE tokens don't align with grammar terminals — one token may span multiple productions. Precompute token masks per grammar state to amortize cost.

GBNF Grammar Example

root   ::= object
object ::= "{" pair ("," pair)* "}"
pair   ::= string ":" value
value  ::= string | number | object
         | array | "true" | "false"
         | "null"
string ::= "\"" [^"]* "\""
number ::= [0-9]+ ("." [0-9]+)?
array  ::= "[" value ("," value)* "]"

Finite-State Machine Approaches

Regex-Constrained Generation

  • Compile regex into a deterministic finite automaton (DFA)
  • For each DFA state, precompute set of valid vocabulary tokens
  • Per-token masking becomes a table lookup — near zero overhead
  • Works well for dates, enums, phone numbers, simple formats

When to Use FSMs vs CFGs

Regular languages (FSM) are faster. Use CFGs only when you need recursion — nested JSON, matching brackets, recursive data structures.

Token Healing

The BPE Boundary Problem

When constrained decoding forces a prefix, the boundary between forced and generated tokens may produce an unnatural token sequence that the model never learned.

How Token Healing Works

  • Step 1: Back up by one token at the forced boundary
  • Step 2: Re-decode, constraining to tokens whose text starts with the backed-up character(s)
  • Step 3: Model "heals" the boundary into a natural token sequence

Without Healing

Forced: "{"         → token [123]
Model:  "name"      → token [8842]
// Boundary is unnatural split

With Healing

Forced: "{"         → back up 1
Model:  "{\"name"   → token [57234]
// Natural BPE merge preserved

Logit Processors

Types of Logit Processors

  • Logit bias — fixed per-token offsets (crudest)
  • Grammar mask — CFG/PDA state-based masking
  • Regex mask — DFA state-based masking
  • Schema mask — JSON Schema-derived constraints

Composability

Logit processors compose as a pipeline. Multiple constraints can be applied sequentially. This clean abstraction enables integration into existing inference code with minimal changes.

Performance Implications

Per-Step Overhead

  • Must compute valid token set at every decoding step
  • Complex grammars → expensive intersection of vocab with valid continuations
  • Precomputation converts per-token grammar checks to table lookups
  • Index precomputation: one-time cost, can be cached

Memory Trade-off

Precomputed index: O(states × vocab_size) bits. For large grammars and 128K+ vocabularies, this can be significant.

Speculative Decoding Interaction

The Problem

Draft model must also respect constraints. Speculative tokens valid under unconstrained decoding may be rejected for grammar violations, reducing acceptance rate.

Mitigations

  • Constraint-aware draft selection
  • Grammar state forwarding to draft model
  • Parallel constraint checking on speculative tokens

Approach Structural Validity Latency Impact Failure Mode Recovery
Prompt-only ~95–99% None Malformed output Retry loop
JSON mode ~99.9% (syntax) Minimal Wrong schema Validation + retry
Schema-constrained 100% (by construction) Low Wrong content Semantic check

At 1M requests/day, a 0.1% failure rate = 1,000 errors/day. Constrained decoding shifts the problem from "does it parse?" to "is the content correct?" — a strictly easier question.

Practical Patterns

Pattern 1: Right-Size Your Constraint

  • Regex for simple formats (dates, enums, IDs)
  • JSON Schema for structured data
  • Full CFG only when you need recursion

Narrower constraints = less overhead, better model quality.

Pattern 2: Keep Schemas Simple

  • Avoid deeply nested optional fields
  • Minimize complex union types
  • Flat structures decode faster

Complex schemas increase constraint-checking cost and can confuse content generation.

Pattern 3: Prefer Constraints Over Prompts

When the output format is known at request time, always use schema-constrained generation. Prompts are for content guidance, not format enforcement.

Pattern 4: Validate Semantics

Structural validity ≠ correctness. Always validate the content of constrained output. A perfectly formatted JSON object with hallucinated values is still wrong.

API Usage Example

vLLM with JSON Schema

from openai import OpenAI
client = OpenAI(base_url="http://localhost:8000/v1")

schema = {
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "sentiment": {
      "type": "string",
      "enum": ["positive", "negative", "neutral"]
    },
    "confidence": {
      "type": "number",
      "minimum": 0, "maximum": 1
    }
  },
  "required": ["name", "sentiment", "confidence"]
}

response = client.chat.completions.create(
  model="meta-llama/Llama-3-8B-Instruct",
  messages=[{"role": "user",
    "content": "Analyze: 'Great product!'"}],
  response_format={
    "type": "json_schema",
    "json_schema": {"schema": schema}
  }
)

Guaranteed Output

{
  "name": "Product Review",
  "sentiment": "positive",
  "confidence": 0.95
}

What's Guaranteed

  • Valid JSON syntax
  • All required fields present
  • Correct types for every field
  • Enum values from allowed set only
  • Number within specified range

What's NOT Guaranteed

  • Semantic correctness of values
  • Appropriate confidence calibration
  • Factual accuracy of content

Key Takeaways

1. Intervene at the Logit Level

Don't rely on prompts for format correctness. Constrained decoding makes invalid output impossible by construction, not by instruction.

2. Match Constraint to Complexity

Regex for simple patterns, JSON Schema for structured data, CFGs for recursive languages. Overconstraining hurts both performance and output quality.

3. Structure ≠ Correctness

Valid format does not mean valid content. Always pair structural constraints with semantic validation in your application layer.

4. Framework Convergence

vLLM, SGLang, TensorRT-LLM, and major APIs all support constrained decoding. It has transitioned from research to production infrastructure.

5. Mind the Performance Budget

Precomputation amortizes cost. Cache grammar indices. Be aware of speculative decoding interactions. Profile before optimizing.

6. Formal Languages Meet Neural Nets

Constrained decoding is where classical CS (automata theory, formal grammars) meets modern ML. The combination is more powerful than either alone.
