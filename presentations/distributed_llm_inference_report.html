The Problem Space

LLMs have grown beyond single-GPU capacity. Practitioners are turning to clusters of small, affordable GPUs (8-24 GB VRAM) instead of expensive high-end accelerators.

This introduces challenges spanning:

  • Memory management — fitting model weights + KV-cache
  • Inter-device communication — synchronizing partial results
  • Scheduling — batching and request routing
  • Numerical precision — accuracy vs. efficiency trade-offs

Core Tension

Every chip added reduces memory pressure but increases communication overhead. The engineering challenge is finding the sweet spot.

Interconnects & Inference Phases

Interconnect Bandwidth

  • NVLink (H100): 900 GB/s bidirectional
  • PCIe 4.0: 32–64 GB/s per link
  • InfiniBand: up to 400 Gb/s per port

Commodity setups use PCIe or Ethernet — making communication optimization critical.

Two-Phase Workload

  • Prefill: processes full prompt in parallel, compute-bound
  • Decode: generates tokens one at a time, memory-bandwidth-bound

A strategy ideal for prefill may be suboptimal for decode — and vice versa.

Tensor Parallelism (TP)

Splits individual weight matrices across GPUs. Each GPU computes a shard, then results are combined via all-reduce.

  • Most common with high-bandwidth links (NVLink)
  • Requires 2 all-reduces per transformer layer (attention + FFN)
  • All GPUs active on every layer → low latency
  • On PCIe, TP beyond 2-way often counterproductive

Pipeline Parallelism (PP)

Assigns different layers to different GPUs. Activations flow forward through the pipeline.

  • Point-to-point communication (not collective)
  • Tolerates lower-bandwidth links better than TP
  • Main drawback: pipeline bubbles during decode
  • Depends on request-level batching for utilization

Sequence & Expert Parallelism

Sequence Parallelism (SP)

  • Distributes the sequence dimension across GPUs
  • Each chip processes a contiguous chunk of the input
  • Ring Attention circulates KV blocks through a ring topology
  • Ideal for very long contexts where KV-cache alone exceeds chip memory
  • Communication overlaps with computation on current block

Expert Parallelism (EP)

  • For Mixture-of-Experts (MoE) models
  • Different experts placed on different GPUs
  • All-to-all communication routes tokens to correct expert
  • Well-suited to small GPUs: each chip holds only a subset of experts

Communication Bottleneck & Solutions

The Bottleneck

For a model with L layers, decode involves 2L all-reduce ops per token. At TP=8 on PCIe 4.0, transfer time can exceed compute time by 10× at small batch sizes.

Overlap Comm. & Compute

Schedule NCCL ops on separate CUDA streams. While one chunk is being reduced, the GPU computes the next. Used by DeepSpeed, Megatron-LM, vLLM.

Quantized Communication

Send activations in FP8 or INT8 instead of FP16 — halving bytes on the wire with minimal accuracy loss. Separate quantization for comm. vs. compute paths.

Topology-Aware Placement

Assign TP groups to chips sharing fastest links. Confine PP stages to inter-node boundaries. Can reduce cross-node traffic by 50–90%.

KV-Cache Management

For a 70B model with 2048-token sequences at FP16, KV-cache per request can exceed 2–3 GB.

PagedAttention

Borrows virtual memory paging from OS design. KV blocks allocated on demand in fixed-size pages. Eliminates internal fragmentation. De facto standard (vLLM).

Prefix Caching

Shared system prompts → compute KV-cache once and reuse. Reduces redundant computation for chatbot workloads with common prefixes.

KV-Cache Compression

GQA / MQA reduce cache at architecture level. Post-hoc quantization to FP8 / INT4 halves or quarters memory with modest accuracy trade-off.

Weight Quantization & Offloading

Weight Quantization

  • Formats: GPTQ, AWQ, GGUF — compress to 4-bit or lower
  • 70B model in 4-bit ≈ 35 GB (2 × 24 GB GPUs vs. 6–8 at FP16)
  • Accuracy trade-off at extreme quantization, especially for reasoning tasks
  • Mixed-precision: keep first/last blocks at higher precision, aggressively quantize middle layers

Offloading to CPU / NVMe

  • When GPU memory insufficient even after quantization
  • FlexGen: schedules prefetch + compute + offload across GPU-CPU-SSD hierarchy
  • Reasonable throughput for offline inference; latency increases
  • Partial offloading: hot layers on GPU, cold layers in CPU RAM
  • Works best when PCIe bandwidth can overlap fetches with computation

Scheduling & Batching

Continuous Batching

New requests join the batch at every decode step (not waiting for the whole batch to finish). Short sequences no longer block long ones. Pioneered by Orca, adopted by vLLM and TensorRT-LLM.

Disaggregated Prefill/Decode

Separate chip pools for each phase. Prefill pool is compute-optimized, decode pool is bandwidth-optimized. KV-cache handed off between pools. Used by Splitwise and DistServe.

Speculative Decoding

Small draft model generates candidate tokens. Large model verifies in one batched pass. Multiple tokens accepted per step, reducing expensive large-model forward passes and inter-chip communication rounds.

Challenges Specific to Small-Chip Systems

Higher Parallelism Degree

70B FP16 on 12 GB GPUs: 12–16 way parallelism vs. 2–4 way on 80 GB GPUs. Each additional degree adds communication ops — sometimes superlinearly.

Load Imbalance

Embedding, output projection, and first/last blocks differ in size. With many small chips, imbalances leave chips idle. Tools like Alpa and FlexFlow auto-search optimal partitions.

Fault Tolerance & Stragglers

More chips = higher failure probability. One slow chip bottlenecks everything. Solutions: redundant compute, async pipelines, checkpoint-restart, routing around failures.

Interconnect Heterogeneity

Mixed NVLink + PCIe intra-node, Ethernet/IB inter-node. Parallelism strategy must be co-designed with physical topology to avoid high-comm workloads on slow links.

Performance Engineering

Profiling

  • Profile both compute and communication
  • NVIDIA Nsight Systems visualizes kernels + NCCL on a timeline
  • Goal: communication fully overlapped with useful computation

Flash Attention

  • Tiled, online attention computation
  • Memory: O(n²) → O(n) in sequence length
  • Enables larger batch/sequence on small chips
  • Critical for distributed inference throughput

Configuration Heuristic

  • TP within highest-BW domain (intra-node)
  • PP across lower-BW boundary (inter-node)
  • Tune batch size to fill pipeline bubbles
  • Automated search tools (Alpa, FlexFlow)

Key Recommendations

1. Start with Quantization

4–8 bit weights dramatically reduce chip count and communication overhead.

2. Match Parallelism to Topology

TP on high-BW links only; PP where bandwidth is limited.

3. Adopt PagedAttention

Efficient KV-cache management is non-negotiable. vLLM is the baseline.

4. Use Continuous Batching

Static batching leaves enormous throughput on the table.

5. Profile Relentlessly

Measure TTFT and inter-token latency separately. Optimize each phase independently.

6. Consider Disaggregation

Separate prefill and decode onto specialized chip pools for high-throughput serving.

7. Plan for Failure

More chips = more failure surface. Build monitoring, health checks, and graceful degradation from the start.

Looking Ahead

The trade-off landscape continues to shift with emerging innovations:

  • Linear attention and state-space models (SSMs) that eliminate the KV-cache entirely
  • New quantization methods pushing to sub-4-bit with minimal quality loss
  • Evolving frameworks with better auto-configuration and topology awareness
  • Architectural innovations reducing communication requirements

A system designed with modularity and configurability will be best positioned to adopt these advances as they mature.

Core Takeaway

Distributing LLM inference across many small GPU chips is not only feasible but increasingly practical. The core engineering challenge is managing the tension between per-chip memory limits and inter-chip communication overhead.

Design Principle

No one-size-fits-all answer. The optimal configuration is hardware- and workload-specific. Measure, profile, and iterate.
