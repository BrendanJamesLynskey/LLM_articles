Why the KV Cache Matters

During autoregressive decoding, every attention head at every layer computes key and value projections for each token. Caching these avoids redundant recomputation — but the cache rapidly becomes the dominant consumer of GPU memory.

  • Often exceeds model weight memory for long sequences
  • Directly determines concurrent batch size
  • Limits maximum supported context length
  • Impacts cost per generated token

Core Trade-off

Caching keys and values eliminates redundant compute but creates a memory bottleneck that constrains throughput, concurrency, and context length.

The Fragmentation Problem

Naive Pre-allocation

Traditional systems allocate contiguous memory for the maximum possible sequence length. Most sequences finish early, leaving vast unused memory.

  • 60–80% of allocated KV cache memory wasted
  • Drastically reduces effective batch size
  • Memory is reserved but never used

PagedAttention & Virtual Memory

Introduced in vLLM, PagedAttention applies OS virtual memory concepts to KV cache management.

Key Properties

  • Fixed-size pages (~16 tokens each)
  • Page table maps logical → physical
  • On-demand allocation & freeing
  • Eliminates internal fragmentation

PagedAttention Adoption

PagedAttention has been widely adopted across the LLM inference ecosystem:

Framework Support

Benefits

  • Dynamic allocation matches actual usage
  • No quality impact — purely system-level

Enabling Capability

Serves as the foundation layer for combining with other optimization techniques like quantization and prefix caching.

KV Cache Quantization

Reduce cached K/V tensor precision from FP16/BF16 to lower-bit formats. KV cache is read-heavy, so quantize/dequantize cost amortizes well.

Token Eviction & Pruning

Not all cached tokens matter equally. Eviction strategies remove low-importance tokens to reclaim memory.

H2O (Heavy Hitter Oracle)

Observes that a small subset of tokens consistently receive high attention weights. Evicts tokens with low cumulative attention scores.

Scissorhands & Similar

Uses attention pattern analysis to identify and preserve tokens most likely to be attended to in future steps.

Reduction: 2–5x

50–80% cache size reduction with minimal quality impact on many tasks.

GQA & MQA: Architectural Approaches

Multi-Head Attention

Baseline: separate K/V per head

KV cache: 1x (full)

Grouped Query (GQA)

Groups share K/V heads

KV cache: 4–8x smaller

Multi-Query (MQA)

All queries share 1 K/V

KV cache: 32–128x smaller

Modern models (Llama 3, Mistral, Gemma 2) use GQA as default — best quality-efficiency trade-off.

Cross-Layer KV Cache Sharing

K/V representations are often highly similar between neighboring layers. Cross-layer sharing exploits this redundancy.

YOCO (You Only Cache Once)

Assigns KV caches to only a subset of layers; other layers reference them.

CLA (Cross-Layer Attention)

Adjacent layers share KV caches with modest quality trade-offs.

Result

Reduces total KV cache memory by 2x or more.

Prefix Caching & Radix Trees

Many workloads involve repeated prefixes — system prompts, few-shot examples, shared documents.

How It Works

  • Store KV cache for common prefixes
  • Reuse across multiple requests
  • Eliminates redundant prefill computation
  • Radix tree enables partial prefix matching

SGLang RadixAttention

Reduces time-to-first-token by 3–5x for workloads with high prefix overlap.

Offloading & Hierarchical Storage

When GPU memory is insufficient, spill KV cache pages to slower memory tiers.

Memory Hierarchy

  • GPU HBM — fastest, limited
  • CPU DRAM — larger, slower
  • NVMe SSD — vast, slowest

Prefetching

Predict which pages are needed next and overlap transfer with computation to hide latency.

Future Directions

Learned Compression

Lightweight neural networks trained to compress and decompress KV cache entries, going beyond fixed quantization schemes.

Hardware-Aware Design

Co-design cache formats with GPU memory hierarchy — exploiting L2 cache, shared memory, and register files for lower latency.

Beyond KV Caches

Linear attention variants and state-space models (Mamba) replace KV caches entirely with fixed-size recurrent states — fundamentally changing memory scaling.

As models and context windows grow, continued innovation in KV cache management remains essential for practical and cost-effective LLM deployment.

Key Takeaways

Multi-Layered Challenge

KV cache optimization spans model architecture, runtime systems, and hardware utilization — no single technique is sufficient.

Proven Combination

  • PagedAttention eliminates fragmentation
  • Quantization + eviction for compression
  • GQA for structural reduction
  • Prefix caching for workload reuse

Impact

These techniques collectively enable LLM serving systems to handle much larger batch sizes and context lengths than would otherwise be possible.

Looking Ahead

Learned compression, hardware co-design, and alternative architectures (SSMs, linear attention) promise to push efficiency further as context windows scale to millions of tokens.
