PROBLEM

The KV Cache Memory Crisis

Every decoded token requires stored key and value projections across all layers and heads. The KV cache grows rapidly and dominates GPU memory.

  • Single token in Llama 2 70B: ~320 KB of KV cache
  • 4,096-token sequence: ~1.3 GB
  • Batch of 64 sequences: ~80 GB — exceeds A100 capacity
  • Memory per token: 2 × L × H × d × 2 bytes

PROBLEM

Contiguous Pre-Allocation Waste

Internal Fragmentation

System reserves max-length buffer per sequence. If max = 2,048 but avg = 256 tokens, 87.5% of allocated memory sits unused.

External Fragmentation

As variable-length sequences arrive and depart, GPU memory fills with gaps too small individually for new requests, even if total free memory suffices.

Reservation Waste

Full buffer is locked from request arrival to completion. Other requests cannot use reserved-but-unused space during generation.

Prior systems utilised only 20–38% of allocated KV cache memory effectively — the rest was dead weight.

INSIGHT

The Virtual Memory Analogy

OS Virtual Memory

  • Processes see contiguous virtual address space
  • Mapped to non-contiguous physical pages via page table
  • Pages allocated on demand (page faults)
  • Eliminates internal + external fragmentation
  • Enables sharing via copy-on-write

PagedAttention

  • Sequences see logical KV cache positions (0, 1, ..., n)
  • Mapped to physical GPU blocks via block table
  • Blocks allocated on demand as tokens are generated
  • Eliminates internal + external fragmentation
  • Enables sharing via copy-on-write

Blocks = pages  •  Tokens = bytes  •  Sequences = processes  •  Block table = page table

MECHANISM

Block Allocation & Memory Layout

  • GPU memory pool divided into uniform physical blocks (typically 16 tokens each)
  • Each block stores K and V tensors for block_size tokens across all layers and heads
  • Block table: simple int32 array mapping logical → physical blocks
  • New request: allocate blocks only for prompt tokens
  • Each new token fills current block; when full, allocate next from free pool
  • On completion: all blocks return to free pool immediately

Max waste per sequence: block_size − 1 tokens (the unfilled last block).

MECHANISM

The PagedAttention Kernel

Modified Attention Computation

  • Kernel takes query vector, block table, and physical KV blocks as input
  • Iterates over block table entries, fetching each physical block
  • Computes dot products between query and keys per block
  • Accumulates weighted output with online softmax
  • Functionally identical to standard MHA, but follows block table indirection

Overhead Trade-off

Per-Operation Cost

Block table lookups and non-contiguous access add 2–5% overhead (decode phase is memory-bandwidth bound regardless)

Net System Benefit

Dramatically higher batch sizes from memory savings yield 2–4x overall throughput improvement

MECHANISM

Copy-on-Write for Parallel Sampling

When multiple sequences share a prefix (beam search, parallel sampling), their block tables point to the same physical blocks.

  • Reference count tracks how many sequences use each block
  • On divergence (refcount > 1 and write needed): copy block to new location, update only that sequence's table
  • Other sequences continue referencing the original

Memory Savings

  • Beam search (width 8, 10:1 prompt ratio): ~7x reduction vs. full duplication
  • vLLM reported 55% memory reduction for beam search workloads

ALTERNATIVE

vAttention: Hardware-Level Paging

The Problem with Software Indirection

PagedAttention's block table adds software overhead that prevents use of highly optimized kernels like FlashAttention.

vAttention's Approach (2024)

  • Uses CUDA Virtual Memory Management API
  • GPU's hardware MMU handles virtual-to-physical translation
  • KV cache appears contiguous to the attention kernel
  • Standard optimized kernels (FlashAttention, FlashDecoding) work unmodified
  • CoW via OS/driver-level virtual memory remapping

LIMITATIONS

Trade-Offs & Constraints

Block Size Sensitivity

No single block size is ideal for mixed workloads. Smaller blocks reduce waste but increase metadata and scatter. Default of 16 works for most cases.

Metadata at Extreme Scale

Millions of blocks across thousands of sequences require free lists, reference counts, and hash tables. Block management can bottleneck at >10K req/s.

Attention Variant Compatibility

Sliding window, sparse attention, cross-attention in encoder-decoder models all need custom kernel modifications to work with paged storage.

Does Not Reduce KV Cache Size

PagedAttention eliminates waste, not inherent memory requirements. A 128K-token sequence still needs tens of GB. Architectural solutions (GQA, MQA) and quantization remain necessary.

OPERATIONS

Production Monitoring Metrics

Block Utilization Rate

Proportion of allocated slots with active token data. Healthy: >96%. Below 90% suggests block size is too large for workload.

Free Pool Size

Unallocated blocks available. Alert when below 10–15% of total — onset of preemption-related latency spikes.

Preemption Rate

Sustained rate > 0 means GPU memory is over-subscribed. Degrades tail latency; signals need for more capacity or KV compression.

Copy-on-Write Frequency

High during beam search / parallel sampling = healthy. Unexpectedly high during single-sample = possible prefix caching misconfiguration.

Shared-to-Unique Block Ratio

Quantifies CoW and prefix caching savings. Low ratio on expected high-prefix workloads may indicate broken hash matching or caching disabled.

Together, these metrics guide fleet sizing, block size tuning, and workload routing decisions.
