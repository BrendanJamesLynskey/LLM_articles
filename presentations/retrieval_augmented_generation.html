Why RAG?

LLM Limitations

  • Knowledge frozen at training cutoff
  • Hallucination on factual queries
  • No source attribution
  • Retraining is expensive & slow

RAG Solution

  • External knowledge at inference time
  • Grounded, verifiable answers
  • Traceable citations to sources
  • Update knowledge without retraining

First formalized by Lewis et al. (2020), RAG decouples knowledge storage from reasoning, letting the generator focus on synthesis.

Chunking Strategies

Fixed-Length

512-token windows with overlap. Simple but splits semantic units mid-thought.

Recursive Splitting

Respects document hierarchy: headings → paragraphs → sentences. Better boundary placement.

Semantic Chunking

Embedding similarity between consecutive sentences; split where similarity drops. Preserves topic coherence.

Parent-Child Strategy

Retrieve on small child chunks for precision, then expand to parent section for generation context. Best of both worlds.

Retrieval Methods

Dense Retrieval

  • Bi-encoder similarity
  • Strong on paraphrases
  • Weak on rare terms

Sparse Retrieval

  • BM25, SPLADE
  • Exact keyword matching
  • Strong on entities

Hybrid Retrieval

  • Combines dense + sparse
  • RRF or learned fusion
  • Best overall accuracy

Empirically, hybrid retrieval with reciprocal rank fusion consistently outperforms either method alone. Most production systems default to this pattern.

Multi-Hop & Iterative RAG

Complex questions often require synthesizing information across multiple documents. Single-round retrieval is insufficient.

Iterative Pattern

  • Decompose query into sub-questions
  • Retrieve evidence for each sub-question
  • Interleave CoT reasoning with retrieval
  • Refine context at each step

IRCoT interleaves chain-of-thought with retrieval, dramatically improving multi-step tasks (HotpotQA, MuSiQue).

Agentic RAG

The agent treats retrieval as one tool among many, deciding dynamically:

  • When to retrieve (skip for simple queries)
  • What query to issue (reformulate if needed)
  • Which index to search (multi-source)
  • Whether to re-retrieve on poor results

Frameworks

LangGraph, LlamaIndex Agents, CrewAI — all support tool-use patterns with retrieval as a callable action.

Agentic Loop

  • Receive query
  • Plan: do I need external info?
  • Act: call retrieval / API / calculator
  • Observe: evaluate result quality
  • Decide: answer or iterate

Risks

Runaway tool loops, high latency, unpredictable costs. Guardrails and max-iteration limits are essential.

Self-RAG & Corrective RAG

Self-RAG (Asai et al. 2023)

  • Model emits reflection tokens
  • [Retrieve] — is retrieval needed?
  • [IsRel] — is passage relevant?
  • [IsSup] — is output supported?
  • Reduces unnecessary retrieval calls
  • Improves factual grounding without external verifier

CRAG (Corrective RAG)

  • External evaluator scores retrieval quality
  • Three actions based on confidence:
    • Correct: use retrieved docs
    • Ambiguous: refine + web search
    • Incorrect: fall back to web
  • Robust to knowledge-base gaps

Faithfulness

Are generated claims supported by retrieved context? Guards against hallucination.

Does the response address the user's actual question?

Are retrieved passages pertinent to the query? Irrelevant context misleads the generator.

Production: Latency & Caching

Latency Budget

  • Embedding query: 5–15ms
  • ANN search: 5–20ms
  • Reranking (cross-encoder): 30–80ms
  • Total retrieval overhead: 50–200ms

Semantic Caching

  • Cache results for semantically similar queries
  • Embedding similarity for cache hits
  • 40–60% retrieval call reduction
  • GPTCache, Redis vector modules

Parallel streaming: begin LLM generation while retrieval completes in parallel. Requires careful orchestration to avoid generating before context is ready.

Freshness & Index Maintenance

RAG's key advantage is updating knowledge without retraining, but this demands disciplined index maintenance.

  • Incremental indexing — handle additions, updates, deletions
  • Embedding versioning — stale vectors from old encoder silently degrade quality
  • CDC pipelines — change-data-capture from source systems
  • Periodic re-indexing — full rebuild on encoder upgrades

Silent Failures

Mixing embeddings from different encoder versions is a common production bug. Recall degrades 10–25% without any explicit error.

Best Practice

Version indices alongside the embedding model. Tag every vector with its encoder_version metadata.

Implementation Pattern

# Minimal hybrid RAG pipeline (pseudocode)
async def rag_query(query: str, top_k: int = 5) -> str:
    # 1. Encode query
    q_embedding = encoder.encode(query)

    # 2. Hybrid retrieval: dense + sparse
    dense_hits  = vector_db.search(q_embedding, limit=100)
    sparse_hits = bm25_index.search(query, limit=100)
    candidates  = reciprocal_rank_fusion(dense_hits, sparse_hits)

    # 3. Rerank with cross-encoder
    reranked = cross_encoder.rerank(query, candidates[:50])
    top_passages = reranked[:top_k]

    # 4. Assemble prompt and generate
    context = "\n---\n".join(p.text for p in top_passages)
    prompt = TEMPLATE.format(context=context, query=query)
    return await llm.generate(prompt)

Production additions: semantic cache before step 1, query rewriting between steps 1–2, guardrails after step 4, observability at every stage.

Key Takeaways

Architecture

  • Hybrid retrieval (dense + sparse) is the production default
  • Two-stage retrieve + rerank for precision
  • Semantic chunking with parent-child expansion

Advanced Patterns

  • Multi-hop for complex reasoning
  • Agentic RAG for dynamic tool selection
  • Self-RAG for adaptive retrieval decisions

Evaluation

  • Faithfulness, answer relevance, context relevance
  • RAGAS / ARES for automated assessment
  • Human eval remains gold standard

Production

  • Semantic caching reduces retrieval calls 40–60%
  • Version indices with embedding model
  • Combine RAG + fine-tuning for best results
