PROBLEM

Why Serving LLMs Is Hard

Autoregressive Bottleneck

  • Each token depends on all previous tokens
  • Sequential generation underutilizes GPU parallelism
  • Single request saturates <10% of GPU compute

KV Cache Memory Pressure

  • 70B model at 128K context: ~40GB KV cache per request
  • Contiguous allocation causes fragmentation
  • Memory, not compute, is the typical bottleneck

TECHNIQUES

Core Serving Optimizations

Continuous Batching

New requests enter the batch at each iteration, not waiting for the longest sequence to finish. Maximizes GPU utilization.

Paged KV Cache

KV cache split into fixed-size blocks mapped via a block table. Eliminates fragmentation, enables near-100% memory use.

Speculative Decoding

Small draft model proposes N tokens; large model verifies in one forward pass. 2-3x latency reduction with identical outputs.

Prefix Caching

Shared system prompts computed once, KV blocks reused across requests. Saves 20-40% prefill compute.

Kernel Fusion

FlashAttention, fused RoPE, and custom GEMM kernels reduce memory round-trips and kernel launch overhead.

Quantization

FP8, INT4, GPTQ, AWQ, GGUF -- shrink model footprint by 2-4x with minimal quality degradation.

FRAMEWORK

vLLM

PagedAttention Engine

Borrows virtual memory concepts from OS design. KV cache is split into pages mapped through a block table, eliminating internal fragmentation and enabling near-optimal memory utilization.

Key Capabilities

  • Continuous batching with preemption support
  • Prefix caching -- automatic KV block reuse
  • Multi-LoRA -- hot-swap adapters per request
  • Speculative decoding with draft models
  • Tensor parallelism across multi-GPU nodes
  • OpenAI-compatible API server

FRAMEWORK

TensorRT-LLM

Compiler-Driven Optimization

  • Compiles model into hardware-specific execution plan
  • Aggressive kernel fusion across layers
  • Native FP8 on Hopper/Blackwell GPUs
  • INT4 AWQ/GPTQ quantization support
  • Custom GEMM schedules per GPU architecture

Runtime Features

  • Inflight batching with preemption
  • Paged KV cache management
  • Tensor + pipeline parallelism
  • Speculative decoding (Medusa, Eagle)
  • Integrates with Triton Inference Server

Trade-off

Highest raw throughput on NVIDIA GPUs at the cost of portability. Requires explicit compilation steps and is locked to NVIDIA hardware. Best for stable, high-volume production deployments on NVIDIA infrastructure.

FRAMEWORK

Text Generation Inference (TGI)

Hugging Face's Production Server

Written in Rust for the serving layer with Python model backends. Prioritizes operational simplicity and ecosystem integration.

  • One-command Docker deployment for most HF Hub models
  • Continuous batching + Flash Attention
  • Quantization: GPTQ AWQ bitsandbytes
  • Grammar-constrained structured output
  • Token streaming via SSE
  • Built-in Prometheus metrics + distributed tracing
  • Watermarking support for generated text

Sweet Spot

Teams wanting a reliable, well-integrated serving layer without deep performance tuning. Sensible defaults, strong observability, and tight HF ecosystem integration.

docker run --gpus all \
  -p 8080:80 \
  ghcr.io/huggingface/\
  text-generation-inference \
  --model-id meta-llama/\
  Llama-3-70B-Instruct \
  --quantize awq

FRAMEWORK

SGLang

RadixAttention

Organizes cached KV prefixes in a radix tree. Enables automatic, fine-grained prefix matching without explicit declaration. Ideal for agentic workloads with shared system prompts.

Frontend DSL

Composable Python functions for multi-turn, branching, and constrained generation. Runtime optimizes across structured programs.

@sgl.function
def tool_agent(s, query):
    s += sgl.system("You are a tool agent.")
    s += sgl.user(query)
    s += sgl.assistant(
        sgl.gen("thought",
            max_tokens=256))
    s += sgl.assistant(
        sgl.gen("action",
            choices=["search",
                     "calculate",
                     "respond"]))

FRAMEWORK

llama.cpp & Ollama

llama.cpp

  • Pure C/C++ with minimal dependencies
  • GGUF quantization: 2-bit through 8-bit
  • Runs on CPU, Apple Silicon, CUDA, Vulkan, SYCL
  • From Raspberry Pi to MacBook Pro
  • Grammar-based constrained generation (GBNF)
  • Broad architecture support (LLaMA, Mistral, Phi, Gemma, Qwen, ...)

Ollama

  • Docker-like UX for local LLM serving
  • Model registry with one-command pull + run
  • Automatic quantization and memory management
  • REST API + native client libraries
  • Model layering with Modelfile
  • De facto standard for local LLM development
# llama.cpp server
./llama-server \
  -m llama3-70b-Q4_K_M.gguf \
  --port 8080 --n-gpu-layers 40
# Ollama
ollama run llama3:70b
# That's it. Model downloads, quantizes
# if needed, and serves automatically.

FRAMEWORK

Triton Inference Server

Multi-Model Orchestrator

Triton is not an LLM engine itself -- it orchestrates multiple backends behind a unified API. Ideal for heterogeneous model serving.

  • Model ensembles -- chain LLM + embedder + reranker
  • Dynamic batching across model types
  • A/B testing and canary deployments
  • Backends: TensorRT-LLM, vLLM, ONNX, PyTorch, TensorFlow
  • GPU resource sharing and model prioritization
  • Production-grade metrics and health checks

When to use Triton

When LLMs are one component in a larger serving infrastructure alongside embeddings, classifiers, and other models.

FEATURES

Production-Essential Capabilities

Token Streaming (SSE)

Server-sent events deliver tokens as they are generated. Universal across all frameworks. Essential for responsive UX.

Multi-LoRA Serving

Single base model serves dozens of fine-tuned variants simultaneously. Per-request adapter selection reduces GPU cost dramatically.

Speculative Decoding

Draft model proposes tokens, target model verifies in parallel. 2-3x latency reduction with mathematically identical outputs.

Structured Output

Grammar-constrained decoding, JSON schema enforcement, regex-guided generation. Critical for function calling and data extraction pipelines.

SCALING

Scaling & Load Balancing

LLM-Specific Challenges

  • Request durations vary by orders of magnitude
  • Round-robin creates hot spots -- avoid it
  • CPU-based autoscaling metrics are meaningless for GPU workloads

Effective Strategies

  • Load-aware routing: consider batch occupancy + KV cache utilization
  • Prefix-aware routing: direct similar prompts to same instance for cache hits (20-40% throughput gain)
  • GPU-aware autoscaling: scale on GPU util + queue depth, not CPU
  • Least-connections: simple but effective baseline

GUIDE

Choosing a Framework

Max Throughput on NVIDIA

Use TensorRT-LLM. Stable model, NVIDIA-only infra, willing to invest in build pipeline. The performance ceiling.

General-Purpose Serving

Use vLLM. Best balance of performance, flexibility, and community. Broad hardware support, rapid iteration.

Agentic / Structured Workloads

Use SGLang. RadixAttention and frontend DSL excel at multi-step reasoning and constrained generation.

HuggingFace Ecosystem

Use TGI. Operational simplicity, one-command deploy, strong observability. Pragmatic and reliable.

Local / Edge / Consumer HW

Use llama.cpp / Ollama. CPU, Apple Silicon, modest GPUs. Development, privacy-sensitive, and edge deployments.

Multi-Model Infrastructure

Use Triton. LLMs alongside embedders, rerankers, classifiers behind a unified API. The orchestration layer.

CONCLUSION

Key Takeaways

The Landscape Has Matured

  • Shared architectural principles: continuous batching, paged KV caches, speculative decoding
  • Implementations diverge in optimization targets and operational philosophy
  • Many organizations run multiple frameworks -- latency-sensitive interactive + throughput-optimized batch

What Matters Most

  • Understanding trade-offs is as important as understanding the models
  • Cost-per-token is the north star metric driving framework evolution
  • The best framework depends on your workload, hardware, and team

The Convergence

What began as simple model wrappers has evolved into sophisticated systems managing memory at the page level, compiling hardware-specific execution plans, and coordinating complex multi-step generation programs.

The Differentiation

vLLM: throughput generalist. TensorRT-LLM: NVIDIA performance ceiling. TGI: operational simplicity. SGLang: agentic intelligence. llama.cpp: universal accessibility. Triton: orchestration backbone.
