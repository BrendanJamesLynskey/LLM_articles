FOUNDATIONS

What Are Embeddings?

Embeddings are dense vector representations that project discrete objects — tokens, sentences, passages — into continuous high-dimensional space where geometric proximity encodes semantic similarity.

  • Compact: typically 256 – 4096 float dimensions
  • Learned end-to-end via contrastive objectives
  • Enable retrieval, clustering, classification without task-specific fine-tuning

Sparse vs Dense

  • Sparse (TF-IDF): vocab-sized, mostly zeros, exact lexical match
  • Dense (Embeddings): compact, every dim active, semantic match

ARCHITECTURE

Encoder Architectures

Bi-Encoder

Query & doc encoded independently. Single vector per input. Pre-compute doc embeddings, compare at query time.

  • Fast: O(1) per comparison
  • Used by all production models

Cross-Encoder

Query + doc concatenated, joint attention across all tokens. Highest accuracy.

  • Slow: O(n) forward passes
  • Deployed as re-ranker

Late Interaction (ColBERT)

Retains per-token embeddings. Relevance via MaxSim operation.

  • Near cross-encoder quality
  • Doc tokens pre-computable

TRAINING

Training Objectives

Contrastive Learning (InfoNCE)

Pull positive pairs together, push negatives apart. Loss over a batch of N pairs:

L = -log( exp(sim(q, d+)/τ) / Σ exp(sim(q, di)/τ) )

Negative Mining Strategies

  • In-batch negatives: free, scale with batch size
  • BM25 hard negatives: lexically similar, not relevant
  • Dense hard negatives: mined from prior retriever

Knowledge Distillation

Train a bi-encoder student to reproduce the score distribution of a cross-encoder teacher.

  • Teacher scores query-doc pairs with full attention
  • Student learns to approximate via KL divergence
  • Used in BGE, E5, GTE training pipelines

Multi-task Training

  • Combine retrieval, classification, STS objectives
  • Instruction-tuned variants (E5-Mistral, GTE-Qwen2)
  • Task-specific prefixes guide encoding

Dimensionality & Matryoshka Embeddings

Higher dimensions capture more nuance but cost more memory and compute. Matryoshka Representation Learning (MRL) trains the model so that the first d of D dimensions form a valid embedding at any truncation point.

METRICS

Vector Similarity Functions

Cosine Similarity

cos(a,b) = a·b / (||a|| ||b||)
  • Magnitude-invariant
  • Default for L2-normalized vectors
  • Range: [-1, 1]

Dot Product

dot(a,b) = Σ ai * bi
  • Equivalent to cosine when normalized
  • Captures magnitude signal
  • Fastest to compute

Euclidean (L2)

L2(a,b) = sqrt(Σ(ai-bi)²)
  • Straight-line distance
  • Equivalent to cosine for normalized vecs
  • Used in some clustering tasks

For L2-normalized vectors: minimizing L2 = maximizing cosine = maximizing dot product

INDEXING

ANN Indexing Algorithms

HNSW

Multi-layer navigable small-world graph. O(log n) search, excellent recall.

  • Default in most vector DBs
  • Tunable: M, efConstruction, efSearch
  • Memory: ~1.5x raw vectors

IVF

Partition space into Voronoi cells via k-means. Search only nprobe nearest clusters.

  • Fast build time
  • Good for large static corpora
  • Recall tuned via nprobe

Product Quantization

Split vector into sub-vectors, quantize each to centroid ID. 8–32x compression.

  • Dramatic memory reduction
  • Pairs with IVF or HNSW
  • Asymmetric distance for accuracy

RETRIEVAL

Hybrid Search: Dense + Sparse

Why Hybrid?

  • Dense retrieval struggles with exact keywords, rare entities, out-of-domain terms
  • Sparse retrieval misses semantic paraphrases
  • Combination consistently outperforms either alone

Fusion Strategies

  • RRF: 1/(k+rank), simple, parameter-free
  • Learned weights: α·dense + (1-α)·sparse
  • SPLADE: learned sparse representations bridge the gap

COMPRESSION

Quantized Embeddings

Scalar Quantization (int8)

  • Each dimension: float32 → int8 (4x savings)
  • Retains 95–99% of full-precision recall
  • Native support in Qdrant, Milvus

Binary Quantization (1-bit)

  • Each dimension → single bit (32x savings)
  • Hamming distance for comparison
  • Best as first-stage filter + re-score

PRACTICE

Chunking Strategies

Fixed-Size Token

  • Split every N tokens with overlap
  • Simple, predictable chunk size
  • May split mid-sentence
  • Typical: 256–512 tokens, 50 overlap

Semantic / Structural

  • Split at paragraph, section, or heading boundaries
  • Preserves coherent units of meaning
  • Variable chunk sizes
  • Requires document structure parsing

Recursive Character

  • Try delimiters in order: \n\n, \n, sentence, word
  • Fall back to smaller splits
  • Pragmatic middle ground
  • Default in LangChain, LlamaIndex

Key trade-off: too small → lost context  |  too large → diluted embedding averaging over multiple topics

Normalization & Domain Adaptation

L2 Normalization

  • Most models output normalized vectors — verify before choosing metric
  • Normalized: cosine = dot product (faster)
  • Some models (E5, GTE) normalize by default
  • If unsure, normalize at index time for safety

Domain Adaptation

  • General-purpose embeddings underperform on specialized domains
  • Continued contrastive fine-tuning on ~1K in-domain pairs yields major gains
  • Use synthetic query generation from domain docs
  • Frameworks: sentence-transformers, Uniem
  • Risk: catastrophic forgetting — use low LR, short training
# Lightweight domain adaptation with sentence-transformers
from sentence_transformers import SentenceTransformer, losses, InputExample
model = SentenceTransformer("BAAI/bge-base-en-v1.5")
train_examples = [InputExample(texts=[query, positive_passage]) for ...]
train_loss = losses.MultipleNegativesRankingLoss(model)
model.fit(train_objectives=[(train_dataloader, train_loss)], epochs=3)

BENCHMARKS

Retrieval Performance Patterns

What Matters Most (ranked)

  1. Chunking strategy — biggest practical lever
  2. Embedding model choice — 5-15% recall spread on MTEB
  3. Hybrid search — consistent 3-8% recall lift
  4. Re-ranking — 5-10% precision improvement
  5. Index type — <1% recall difference for well-tuned HNSW
  6. Quantization — negligible if using re-scoring

Common Pitfalls

  • Wrong similarity metric for the model (cosine vs dot)
  • Chunks too large → diluted embeddings
  • No query-document asymmetry handling (missing prefixes)
  • Ignoring metadata filters — pre-filter > post-filter
  • Not evaluating on domain-specific queries
  • Skipping normalization check

SUMMARY

Key Takeaways

Architecture

  • Bi-encoders for retrieval, cross-encoders for re-ranking
  • ColBERT bridges quality-speed gap
  • Multi-stage pipelines combine all three

Efficiency

  • Matryoshka embeddings: truncate dims at deploy time
  • Binary/scalar quantization: 4-32x memory reduction
  • HNSW + PQ for billion-scale indexes

Quality

  • Hybrid search (dense + BM25) consistently wins
  • Chunking matters more than model choice
  • Domain adaptation with ~1K pairs is high-ROI

Production

  • Match similarity metric to model training
  • Start with pgvector, scale to purpose-built DB
  • Evaluate end-to-end, not just embedding quality

The cost-quality Pareto frontier keeps improving — high-quality semantic retrieval is more accessible than ever.
