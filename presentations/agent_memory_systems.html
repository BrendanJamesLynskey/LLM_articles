PROBLEM

Why Agents Need Memory

The Amnesiac Savant

  • Without memory, every interaction starts from zero
  • Context window consumed by system prompt, tools, conversation, retrieved docs
  • 128K context fills fast with 20 tool definitions + conversation history
  • "Lost in the middle" — models attend poorly to mid-context information

What Memory Enables

  • Continuity: remember previous sessions and pick up where you left off
  • Learning: remember mistakes and avoid repeating them
  • Personalisation: learn user preferences, style, and domain terminology
  • Knowledge accumulation: build understanding over time

Key Insight

Only working memory is directly accessible to the model. Everything else must be loaded into working memory (the context window) before the model can use it. Memory management is fundamentally about deciding what to load, when.

SHORT-TERM

Conversation Buffer and Sliding Window

Conversation Buffer

  • Store all messages from the current session
  • Include full history in each prompt
  • Simple and reliable — no information loss
  • Problem: hits context window limit as conversation grows
  • Works well for short conversations (<20 turns)

Sliding Window

  • Keep only the last N messages (or N tokens)
  • Older messages are dropped
  • Fixed context usage — never exceeds budget
  • Problem: loses information from early conversation
  • Model may contradict earlier statements
  • Works for casual chat; bad for complex tasks
agent-memory / 04
SHORT-TERM

Summary Memory

How It Works

  • Periodically summarise older conversation turns using the LLM
  • Replace detailed history with concise summary
  • Summary + recent messages fit in context budget
  • Progressive summarisation: summarise summaries as they grow
  • Can be triggered by token count threshold or turn count

Tradeoffs

  • Pro: retains key information in compact form
  • Pro: bounded context usage
  • Con: information loss (details discarded)
  • Con: summarisation cost (LLM call per summary)
  • Con: summary errors compound over time

LONG-TERM

RAG-Backed and Vector Store Memory

Vector Store Memory

  • Store memories as text chunks with embeddings in a vector database
  • At query time, embed the current context and retrieve top-k similar memories
  • Inject retrieved memories into the prompt
  • Scales to millions of memories
  • Retrieval quality depends on embedding model and chunking strategy

What to Store

  • User preferences: "prefers TypeScript, dislikes semicolons"
  • Facts learned: "project uses PostgreSQL 16"
  • Past interactions: "we debugged auth issue on March 15"
  • Corrections: "user corrected me: API uses v3 not v2"
  • Decisions made: "we decided to use microservices"

Retrieval Strategies

Recency: prefer recent memories. Relevance: semantic similarity to current query. Importance: score memories by significance (corrections > casual remarks). Most systems combine all three: score = α·recency + β·relevance + γ·importance.

LONG-TERM

Knowledge Graph Memory

How It Works

  • Store memories as entities and relationships in a graph
  • Entity: "FastAPI" — Relationship: "used_by" — Entity: "User's project"
  • Graph queries retrieve structured knowledge (not just similar text)
  • Supports multi-hop reasoning: "What stack does the user's project use?"
  • Better for factual / relational memory than vector search

ADVANCED

Memory Consolidation

The Consolidation Problem

  • Raw memories accumulate — retrieval degrades with volume
  • Redundant memories waste context budget
  • Contradictory memories confuse the model
  • Need to merge, update, and prune over time

Strategies

  • Deduplication: merge near-duplicate memories
  • Contradiction resolution: newer information overrides older
  • Abstraction: generalise episodic → semantic ("user corrected me 3 times about API version" → "user's project uses API v3")
  • Decay: reduce importance of memories not accessed recently
  • Periodic review: LLM reviews and cleans up memory store

SYSTEM

MemGPT and Virtual Context Management

MemGPT: Operating System for LLM Memory

MemGPT (Packer et al., 2023) treats the LLM's context window like an operating system manages RAM. The model has "main memory" (context window) and "external storage" (database). The model can explicitly issue memory operations:

  • core_memory_append: add a fact to persistent core memory (always loaded)
  • core_memory_replace: update an existing core memory fact
  • archival_memory_insert: store something in long-term archive
  • archival_memory_search: retrieve from archive by semantic search
  • conversation_search: search past conversation history

The model controls its own memory — deciding what to remember, what to search for, and what to forget. This is more flexible than application-controlled memory but requires the model to have good memory management skills.

CONCERNS

Privacy, Security, and Design

Privacy and Security

  • Memory stores contain sensitive user data
  • Must encrypt at rest and in transit
  • User should be able to view, edit, and delete memories
  • Memory poisoning: adversary injects false memories via prompt injection
  • Cross-user leakage in multi-tenant systems
  • Regulatory compliance (GDPR right to erasure)

Design Principles

  • Start simple: conversation buffer first, add complexity as needed
  • Memory budget: cap tokens allocated to retrieved memories
  • Freshness: prefer recent memories for fast-changing information
  • Transparency: show users what the agent remembers
  • Graceful degradation: agent should work without memory (just less well)

SUMMARY

Key Takeaways

Core Concepts

  • Memory bridges the gap between context window limits and long-term agent capability
  • Taxonomy mirrors cognitive science: working, episodic, semantic, procedural
  • Only working memory (context) is directly accessible; everything else must be loaded
  • Vector stores for semantic retrieval; knowledge graphs for structured facts
  • Consolidation prevents memory bloat and contradictions

Practical Guidance

  • Start with conversation buffer; add summary for longer sessions
  • Add vector store memory for cross-session personalisation
  • Combine recency + relevance + importance for retrieval scoring
  • Give users control over their memory (view, edit, delete)
  • MemGPT-style self-managed memory is powerful but requires capable models
