The Multi-Agent Multi-Model Problem

Specialised Agents, Specialised Models

Modern multi-agent systems assign different models to different roles: a large reasoning model for planning, a code-tuned model for implementation, a small fast model for tool-calling, and an embedding model for retrieval.

3–8 Concurrent Models

A realistic pipeline might load Llama 3.1 70B for the supervisor, Qwen2.5-Coder 32B for coding, Mistral 7B for tool dispatch, and nomic-embed-text for RAG — all resident in VRAM simultaneously.

Why Not Just One Model?

A single general-purpose model compromises on every task. Specialised models are smaller, faster, and more accurate per role. The trade-off: you need hardware to host all of them at once.

Why This Is Different

Single-Model Serving

  • One set of weights in VRAM
  • One KV cache pool to manage
  • Batching across identical requests
  • Well-understood memory planning
  • Mature tooling (vLLM, TGI)

Multi-Model Serving

  • Multiple weight sets competing for VRAM
  • Separate KV cache per model
  • Inter-model scheduling & routing
  • Load/unload latency on swap
  • Fragmentation from heterogeneous sizes

Key Difference

Single-model serving optimises within one model's memory space. Multi-model serving must optimise across competing models for a shared, finite GPU memory budget.

Memory Budget: Single Model Recap

The Formula

Total VRAM = Model Weights + KV Cache + Activations + Framework Overhead

Model Weights

  • Parameters × bytes per parameter
  • 70B at FP16 = 140 GB
  • 70B at Q4 (4-bit) ≈ ~38–42 GB
  • Quantisation is essential for local hosting

KV Cache

  • 2 × layers × heads × head_dim × seq_len × batch × bytes
  • Grows linearly with context length
  • Llama 70B at 4K context ≈ 2.5 GB per sequence
  • Often the limiting factor for concurrency

Activations & Overhead

Intermediate activations during forward pass: typically 0.5–2 GB. CUDA context, framework buffers, and memory allocator overhead add another 0.5–1.5 GB. Together these form the "hidden tax" on GPU memory.

KV Cache Multiplier

In single-model serving, the KV cache is one pool. In multi-model, each model maintains its own cache with different dimensions.

Per-Model KV Cache Formula

KV_bytes = 2 × n_layers × n_kv_heads
         × head_dim × seq_len
         × bytes_per_elem

The Multiplier Effect

With 4 models active, you have 4 independent KV caches. Even if each is modest, the aggregate can consume 4–8 GB of VRAM that single-model serving never needs.

VRAM Fragmentation

When models are loaded and unloaded dynamically, VRAM becomes fragmented — free memory exists but in scattered chunks too small for the next model.

Causes

  • Models have different sizes (7B vs 32B vs 70B)
  • KV caches grow and shrink dynamically
  • CUDA memory allocator rounds up to block boundaries
  • Unloading a small model leaves an unusable gap

Mitigations

  • Pre-allocate fixed memory pools per model
  • Use PagedAttention for KV caches
  • Defragment by reloading models contiguously
  • Memory-mapped model loading (mmap)

The Swap Penalty

  • When a model doesn't fit, it must be swapped from disk
  • 70B Q4 from NVMe: ~8–12 seconds
  • Each swap stalls the entire pipeline
  • Repeated swaps dominate total latency
  • Swap = death for interactive workloads

Rule of thumb: if you swap more than once per conversation turn, the user experience is unacceptable.

When It Matters

  • Tensor Parallelism — shards exchange activations every layer. PCIe becomes a severe bottleneck for 70B+ TP.
  • Model-per-GPU — interconnect matters less since models communicate through the CPU/orchestrator, not GPU-to-GPU.

Practical Rule

If you're splitting one model across GPUs (TP), NVLink is essential. If each GPU runs a different model, PCIe is fine.

NVMe is Essential

Even with NVMe Gen4, a 70B load takes ~6 seconds. This is acceptable for cold start but unacceptable mid-conversation. Keep hot models resident.

Network Loading

Loading from a model registry over 1GbE is catastrophically slow. Even 10GbE adds significant latency. Pre-stage models on local NVMe.

Model Caching and Swapping

Ollama KEEP_ALIVE

  • Controls how long a model stays in VRAM after last use
  • Default: 5 minutes
  • Set to -1 for permanent residence
  • Set per-model: keep supervisor hot, let others expire
  • OLLAMA_NUM_PARALLEL controls concurrent slots

Hot vs Cold Models

  • Hot: resident in VRAM, instant inference
  • Warm: in system RAM (mmap), ~1–3s reload
  • Cold: on disk only, full load penalty
  • Strategy: keep supervisor + most-used agent hot, others warm

Tiered Caching Strategy

Treat model hosting like a CPU cache hierarchy: VRAM (L1) for the 2–3 models used every turn, system RAM (L2) for models used occasionally, and NVMe (L3) for the full model library. Promote and demote based on access frequency.

Memory-Mapped Loading

Tools like llama.cpp use mmap to memory-map model files. The OS pages in data on demand, meaning a "warm" model can start generating in <1 second with most weights already in page cache.

Quantisation Strategy per Role

Not all agents need the same precision. Match quantisation to the sensitivity of the task:

Agent Role Model Recommended Quant Rationale VRAM
Reasoning / Supervisor Llama 3.1 70B Q6_K or Q8 Highest accuracy needed; planning errors cascade 54–70 GB
Code Generation Qwen2.5-Coder 32B Q5_K_M Code is syntax-sensitive; Q4 introduces errors ~23 GB
Tool-Calling Mistral 7B Q4_K_M Structured output; schema constrains errors ~4.4 GB
Embeddings nomic-embed-text FP16 Embedding quality degrades sharply with quant ~0.5 GB
Summarisation Llama 3.1 8B Q4_K_M Lossy task; slight quality loss acceptable ~4.7 GB

Key Insight

By quantising the tool-caller aggressively (Q4) while keeping the reasoning agent at Q6/Q8, you save ~3 GB from the tool-caller with minimal quality impact, freeing VRAM for the model that needs it most.

CPU Offloading for the Largest Model

How It Works

  • Split model layers between GPU and CPU
  • GPU processes its layers at full speed
  • CPU layers processed via RAM bandwidth
  • Each CPU layer adds ~5–15ms latency
  • Controlled via --n-gpu-layers in llama.cpp

Throughput Impact

70B Q4 Configtok/s
All 80 layers on GPU~15
60 GPU / 20 CPU~10
40 GPU / 40 CPU~6
20 GPU / 60 CPU~3
All CPU (DDR5)~1.5

Multi-Model Strategy

Offload 20–30 layers of the 70B to CPU, freeing ~10–15 GB of VRAM for the smaller specialist models. The 70B loses ~30% throughput, but the pipeline gains by eliminating swap latency for the other models. Net result is often faster end-to-end.

Software Orchestration

Kubernetes + GPU

  • NVIDIA GPU Operator for device plugins
  • MIG (Multi-Instance GPU) to slice H100
  • Resource requests: nvidia.com/gpu: 1
  • Node affinity for model-class placement

GPU Scheduling

  • NVIDIA MPS for GPU sharing between models
  • Time-slicing: multiple model pods share one GPU
  • Memory isolation via MIG partitions
  • Priority classes: supervisor > specialists
  • Preemption for high-priority agent requests

Health & Monitoring

  • DCGM (Data Center GPU Manager) metrics
  • VRAM utilization per model instance
  • Inference latency P50/P95/P99
  • Model load/unload event tracking
  • Alerting on OOM kills and thermal throttle

MIG for Multi-Model

An H100 can be partitioned into up to 7 MIG instances. Assign a 40 GB partition for the 70B model, a 20 GB partition for the 32B, and a 10 GB partition for small models. Each partition has isolated memory and compute — no fragmentation, no interference.

Key Recommendations

1. Budget for Total VRAM, Not Just the Largest Model

Sum all concurrent model weights + KV caches + overhead. Multi-agent pipelines need 2–4× the VRAM of single-model deployments.

2. Quantise Per-Role

Use high precision for reasoning (Q6/Q8) and aggressive quantisation for structured-output agents (Q4). Save VRAM where quality impact is lowest.

3. Keep Hot Models Resident

Swap latency kills interactive pipelines. Budget VRAM to keep your 2–3 most-used models permanently loaded.

4. Use Model-per-GPU When Possible

Avoids fragmentation, simplifies memory management, and enables per-model continuous batching through separate vLLM instances.

5. Benchmark Your Actual Pipeline

Theoretical VRAM calculations miss fragmentation, CUDA overhead, and KV cache growth. Always test with real workloads before committing to hardware.

6. Plan for the Bottleneck

The largest model dominates latency. Invest in speculative decoding, better quantisation, or a faster GPU specifically for that model.

The Future

Larger Unified Memory

  • NVIDIA Grace Hopper: 624 GB unified CPU+GPU memory
  • Grace Blackwell GB200: up to 384 GB HBM3e per GPU
  • Eliminates the multi-model VRAM squeeze
  • All models resident with room for massive KV caches

Cheaper Inference ASICs

  • Groq LPU: purpose-built for inference, extreme tok/s
  • Cerebras, SambaNova: dataflow architectures
  • Could dedicate one ASIC per model class
  • 10–100× better perf/$ for inference workloads

Model Merging & Distillation

  • Merge specialist models into one multi-talent model
  • Distill 70B reasoning into 14B with 90% capability
  • Reduce model count from 4–6 down to 2–3
  • Directly reduces VRAM and complexity

Smarter Orchestration

  • Predictive model loading: pre-fetch the next agent's model
  • Dynamic quantisation: load at Q4, promote to Q8 if needed
  • Shared-weight architectures across agents
  • Federated multi-node inference with global KV cache
