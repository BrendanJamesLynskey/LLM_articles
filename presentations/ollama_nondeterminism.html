Non-Deterministic Output in Ollama

Why temperature=0 Doesn't Guarantee Reproducibility — And What You Can Do About It

PROBLEM

The Reproducibility Expectation Gap

Common assumption: setting temperature=0 yields perfectly deterministic, repeatable output from Ollama.

Reality: different hosting instances — or even successive runs on the same instance — produce subtly different results despite identical prompts and parameters.

This report traces the root causes through the software and hardware stack and provides actionable mitigations.

Observed Symptoms

  • Same prompt, different outputs across machines
  • First run differs from subsequent runs
  • Parallel slots produce 5–8 unique completions
  • Output "slightly random" even with fixed seed

BACKGROUND

How Ollama Generates Text

Ollama wraps llama.cpp, the open-source C++ inference engine. The pipeline has four stages:

Key Insight

Temperature only controls the sampling stage. If the logits themselves differ between runs, greedy decoding will faithfully amplify that difference. Everything upstream of sampling is where non-determinism originates.

CAUSE 1

Floating-Point Non-Associativity

The most fundamental cause: IEEE 754 floating-point arithmetic is not associative.

(a + b) + c  ≠  a + (b + c)

Rounding at each intermediate step means any change in operation order shifts logits by ~1e-4 to 1e-6.

LLM inference performs billions of multiply-accumulate operations. When competing tokens have nearly equal logits, a tiny perturbation flips the argmax.

Cascade Effect

Because generation is autoregressive, a single flipped token early on cascades into a completely different continuation.

Typical Divergence

  • Logit shift: ~1e-4 to 1e-6
  • Close-call tokens: frequent in natural language
  • One flip → entirely new path

CAUSE 2

GPU Kernel Non-Determinism

cuBLAS GEMM Kernels

  • Optimised for throughput, not reproducibility
  • Use Tensor Cores with dynamic algorithm selection
  • Internal parallelism does not fix reduction order
  • Same GPU can yield different results across runs

Cross-Architecture Divergence

  • Different GPU architectures (Ampere, Ada Lovelace, Hopper) use different Tensor Core generations
  • Different PTX instruction sequences
  • Ingonyama: cuBLAS GEMM differed by ~1e-4 across RTX 3090, RTX 4080, and L4

CAUSE 3

Batch Size & Continuous Batching

Multi-Slot Experiment (H100)

Same prompt sent to 8 parallel slots with temperature=0 and all other samplers disabled produced 5 to 8 unique completions.

Prefill vs. Generation Mismatch

Even without concurrency, the batch size during prefill (all input tokens at once) differs from the single-token batch during generation. The first few generated tokens may already diverge from a fully sequential implementation.

ollama-nondeterminism • 05
CAUSE 4

Context Window Size (num_ctx)

Ollama dynamically adjusts the context window size if num_ctx is not explicitly set.

Different context sizes change:

  • Dimensions of the attention computation
  • Layout of the KV cache
  • Numerical path through the model

Community reports confirm that explicitly fixing num_ctx is essential for any level of reproducibility.

Fix: Pin num_ctx

"options": {
  "num_ctx": 4096
}

Choose a fixed value and use it consistently across all requests and instances.

Without Pinning

Output described as "slightly random" even with a fixed seed and temperature=0.

CAUSES 5–6

Quantisation & CPU/GPU Offloading

Quantisation Artefacts

  • Most models are 4-bit or 8-bit GGUF
  • Weights stored as integers but dequantised to FP at runtime
  • Normalisation layers (RMSNorm, LayerNorm) remain in full FP precision
  • FP non-associativity is not eliminated by quantisation
  • Different quant levels (Q4_K_M vs Q8_0) produce entirely different logit distributions

CPU / GPU Mixed Offloading

  • If model exceeds GPU VRAM, layers are offloaded to CPU
  • Offload count depends on available VRAM — varies between machines
  • CPU uses AVX-512, GPU uses CUDA — different instruction sets
  • Different precision modes and reduction orders
  • Same logical computation → different numerical results

Takeaway

Two instances must run the exact same GGUF file (same quant type, same file hash) with full GPU offloading (num_gpu: 999) to avoid these divergence sources.

ollama-nondeterminism • 07
CAUSES 7–8

Software Versions & First-Run Effect

Software Version Differences

  • Different Ollama versions ship different llama.cpp commits
  • Changes to kernel implementations, memory layouts, default parameters
  • Even minor updates can shift logits enough to change output
  • Tokeniser behaviour may differ between versions

Mitigation: Pin exact Ollama version across all machines.

First-Run Divergence

  • First invocation after model load differs from all subsequent runs
  • Subsequent runs are consistent with each other
  • Caused by differences in memory allocation and cache state during initial load
  • First prefill may use a different code path or memory layout

Mitigation: Discard the first response after model load.

MITIGATION

Same-Machine Reproducibility

1. Greedy Decoding

Set temperature: 0 to eliminate sampling randomness.

2. Fixed Seed

Set seed: 42 — necessary but not sufficient on its own.

3. Pin num_ctx

Set num_ctx: 4096 to prevent dynamic context resizing.

4. Single-Slot Mode

Run one request at a time to prevent continuous batching from altering the computational path.

5. Discard First Response

Issue a throwaway prompt after model load, then use subsequent responses which tend to be self-consistent.

With all measures applied, same-machine reproducibility is generally achievable for the same Ollama + llama.cpp version on the same hardware.

ollama-nondeterminism • 11
MITIGATION

Cross-Machine Reproducibility

Substantially harder and not fully possible with standard Ollama builds. These measures narrow the gap:

Environment Matching

  • Identical GPU model and CPU architecture
  • Pin exact Ollama version (same llama.cpp commit)
  • Same GGUF model file (same quant, same hash)
  • Match OS, CUDA toolkit, GPU driver version

Full GPU Offloading

  • Set num_gpu: 999 — no CPU fallback
  • All machines need sufficient VRAM
  • Eliminates CPU/GPU instruction set divergence

Remaining Gap

Even with all the above, cuBLAS kernel-level non-determinism may still introduce occasional token differences. Ingonyama found that true cross-architecture determinism required rewriting GEMM CUDA kernels to avoid Tensor Cores and enforce a deterministic reduction order.

ADVANCED

Strict Determinism Approaches

For use cases requiring bit-exact reproducibility (auditing, benchmarking, ZK proofs):

CPU-Only Inference

  • Deterministic BLAS libraries + fixed thread count
  • Eliminates GPU kernel non-determinism entirely
  • Significant performance cost

Custom Deterministic Kernels

  • Ingonyama: rewrote GEMM to use only CUDA cores
  • No Tensor Cores, fixed reduction order
  • Modest prefill penalty, negligible generation impact

Containerised Inference

  • Package full stack: OS, drivers, Ollama, model
  • Run on identical hardware
  • Maximises environmental consistency

Semantic Equivalence Testing

  • Tolerate paraphrased-but-equivalent responses
  • Use embedding similarity or structured output validation
  • Design for functional equivalence, not bit-exactness

CONFIG

Recommended API Configuration

{
  "model": "llama3:8b-q8_0",
  "messages": [
    {
      "role": "user",
      "content": "Your prompt"
    }
  ],
  "options": {
    "seed": 42,
    "temperature": 0,
    "num_ctx": 4096,
    "num_gpu": 999,
    "num_thread": 8
  }
}

Parameter Breakdown

  • temperature: 0 — greedy decoding
  • seed: 42 — fix the RNG
  • num_ctx: 4096 — prevent dynamic resizing
  • num_gpu: 999 — force full GPU offload
  • num_thread: 8 — fix CPU thread count

Remember

These settings are necessary but not sufficient for cross-machine determinism. Hardware and software environment must also match.

CONCLUSION

Key Takeaways

It's Not a Bug

Non-determinism at temperature=0 is an inherent consequence of how modern GPU hardware executes floating-point arithmetic. Temperature only controls sampling, not the millions of matrix operations that produce logits.

Same-Machine: Achievable

Fixed seed + fixed num_ctx + temperature=0 + single-slot operation is usually sufficient for same-machine reproducibility.

Cross-Machine: Hard

Hardware and software environments must match precisely. cuBLAS kernel-level non-determinism may still produce occasional token differences.

Bit-Exact: Possible But Costly

Requires rewriting inference kernels to avoid Tensor Cores and enforce deterministic reduction order — far beyond standard Ollama usage.
