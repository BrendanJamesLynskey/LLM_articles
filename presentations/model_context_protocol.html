PROBLEM

The N×M Integration Problem

Before MCP

  • N LLM apps × M tools = N×M custom integrations
  • Each app writes bespoke glue code per tool
  • Custom function definitions, provider-specific formats
  • Ad hoc serialization, inconsistent error handling
  • Duplicated effort across every new host application

With MCP

  • Each app implements one MCP client
  • Each tool implements one MCP server
  • N + M integrations instead of N × M
  • Analogous to USB for peripherals or LSP for IDEs
  • Build once, connect everywhere

Additional Pain Points Solved

Fragile context injection — manual script-based stuffing of data into prompts, with no provenance or freshness guarantees. Static tool discovery — tools fixed at prompt construction time, no runtime adaptation.

model-context-protocol / 02
ARCHITECTURE

Three Roles

MCP Hosts

  • LLM-facing applications that initiate connections
  • Mediate between model and external services
  • Manage connection lifecycles
  • Route tool calls, return results
  • Examples: Claude Desktop, Cursor, Cline

MCP Clients

  • Protocol-level connectors inside the host
  • 1:1 connection with a specific server
  • Handle JSON-RPC communication
  • Capability negotiation & transport
  • Host manages multiple clients simultaneously

MCP Servers

  • Lightweight programs exposing capabilities
  • Wrap databases, APIs, file systems, etc.
  • Small, focused, composable
  • One server = one coherent capability set
  • Standardized MCP interface
model-context-protocol / 03
PRIMITIVES

Six Capability Primitives

Tools

Model-controlled. Executable functions the LLM invokes to perform actions or retrieve computed results. Defined with name, description, and JSON Schema input.

Resources

Application-controlled. Data sources providing context via URIs. Can be static or dynamic with subscriptions. Like opening a file in an IDE.

Prompts

User-controlled. Reusable prompt templates for specific workflows. Appear as slash commands or menu options in the host UI.

Sampling

Bidirectional. Servers can request LLM completions through the client, enabling recursive agentic patterns. Host mediates all requests.

Roots

Informational. Declare filesystem or URI scopes a server operates within. Helps clients understand boundaries; not a hard security enforcement.

Logging

Structured log messages at debug, info, warning, error severity levels. Enables observability without separate logging infrastructure.

DEEP DIVE

Tools Primitive

  • Each tool has a name, description, and JSON Schema for inputs
  • Model decides when and how to call based on user request
  • Results returned as structured content (text, images, embedded resources)
  • Dynamic registration: servers can add/remove tools at runtime
  • Notification: notifications/tools/list_changed
  • Client re-fetches via tools/list

PRIMITIVES

Resources & Prompts

Resources

  • Identified by URIs: file:///path/to/doc.md, postgres://server/db/table
  • Static: read once at attach time
  • Dynamic: updated on change via subscriptions
  • Host decides which resources to attach (application-controlled)
  • Provide contextual information, like opening a file in an IDE
  • Discovery: resources/list, access: resources/read

Prompts

  • Reusable prompt templates for specific workflows
  • Example: a database server exposes debug_query template
  • User-controlled — selected explicitly via UI
  • Appear as slash commands or menu items in host
  • Structure natural-language requests into effective workflows
  • Discovery: prompts/list, access: prompts/get

PRIMITIVES

Sampling & Bidirectional Flow

What Is Sampling?

  • Servers can request LLM completions through the client
  • Enables agentic behaviors: tool uses model for intermediate reasoning
  • Creates bidirectional flow: model → tool → model → tool
  • All sampling requests go through the host
  • Host enforces approval policies and rate limits

Example: Code Review Server

Server uses sampling to have the model analyze each changed file, then aggregates results into a structured review.

PROTOCOL

JSON-RPC 2.0 Foundation

Requests

  • Include id, method, optional params
  • Expect a response
  • E.g., tools/call, resources/read

Responses

  • Include matching id
  • Contain result or error
  • Error has code + message

Notifications

  • Include method + optional params
  • No id — fire-and-forget
  • No response expected
// Request
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "query_database",
    "arguments": { "sql": "SELECT ..." }
  }
}
// Response
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [{
      "type": "text",
      "text": "3 rows returned..."
    }]
  }
}
model-context-protocol / 08
LIFECYCLE

Connection Lifecycle

1. Transport Establishment

Client starts server process (stdio) or connects to HTTP endpoint (Streamable HTTP).

2. Initialization

Client sends initialize with protocol version & capabilities. Server responds with its capabilities. Client confirms with initialized notification.

3. Operation

Bidirectional exchange of requests and notifications: tools/list, tools/call, resources/read, prompts/get, sampling requests.

4. Shutdown

Either side terminates. Client sends close notification, or transport is terminated directly.

Capability Negotiation

During initialization, both sides declare supported primitives. Client may offer sampling support and root declarations. Server declares which of the six primitives it supports. This ensures clear expectations before any operations begin.

model-context-protocol / 09
FLOW

Tool Invocation Sequence

  1. Host receives model's tool-call request (native format)
  2. Host identifies which MCP server exposes the tool
  3. Client sends tools/call JSON-RPC request to server
  4. Server executes the tool logic
  5. Server returns result as structured content
  6. Host formats result for the model's native format
  7. Model incorporates result and continues generating

TRANSPORT

Transport Layers

stdio

  • Client launches server as child process
  • Communicates via stdin/stdout
  • Each message = one JSON line
  • No network configuration needed
  • Inherits host process security context
  • Default for desktop integrations

HTTP + SSE (Legacy)

  • SSE endpoint for server-initiated messages
  • HTTP POST for client requests
  • Requires persistent SSE connection
  • Network-accessible for remote servers
  • Being superseded by Streamable HTTP
  • Still supported for backward compatibility

Streamable HTTP

  • Current recommended remote transport
  • Single HTTP endpoint for all communication
  • POST → JSON response or SSE stream
  • Firewall-friendly, load-balancer compatible
  • Supports stateless server architectures
  • Optional Mcp-Session-Id header

Latency Considerations

stdio: sub-millisecond round-trip overhead. Streamable HTTP: network latency dominates. Tool execution time varies from microseconds (file read) to seconds (API call).

Parallel Tool Calls

MCP hosts should execute concurrent tools/call requests in parallel across servers. Protocol is stateless at request level, so concurrent calls are safe.

SECURITY

Security Model & Threats

Threat Landscape

  • Compromised server could inject misleading context
  • Data exfiltration through tool arguments
  • Unauthorized actions on backend systems
  • Prompt injection amplified by tool access
  • Tool poisoning: hidden instructions in tool descriptions

Security Principles

  • Human-in-the-loop: user consent for side-effect operations
  • Least privilege: minimal permissions per server
  • Input validation against JSON Schema before execution
  • Output sanitization to prevent indirect prompt injection
  • Allowlists of trusted servers

Tool Poisoning Example

A malicious server advertises a “search documentation” tool whose description contains hidden instructions telling the model to exfiltrate contents of other tool calls. Mitigation: review tool descriptions before approval, host-level filtering for suspicious content, trusted server allowlists.

AUTH

Authentication & Authorization

OAuth 2.1 Framework

  • Recommended for remote MCP servers
  • Introduced in 2025-03-26 protocol revision
  • Servers declare auth requirements during connection setup
  • Clients handle the OAuth flow including PKCE for public clients
  • Enables enterprise identity providers and RBAC

Deployment Patterns

  • Local stdio: inherits OS-level process permissions
  • Team/Enterprise: OAuth-secured remote servers shared by team
  • Gateway pattern: central proxy handles auth, rate limiting, logging
  • Role-based access controls per tool or resource

Roots & Scope Boundaries

The roots primitive declares expected operational scope (e.g., file:///home/user/project). Combined with least-privilege deployment — read-only DB credentials, scoped API tokens — this limits the blast radius of any single server compromise. Note: roots are informational, not enforced by the protocol itself.

AGENTIC

Agentic Workflows with MCP

Example: “Find and fix the performance regression in the latest PR”

  1. Call GitHub MCP to list recent PRs, identify the relevant one
  2. Call GitHub MCP again to fetch the diff
  3. Call File System MCP to read affected source files
  4. Call Database MCP to query CI performance benchmarks
  5. Model reasons about the cause, generates a fix
  6. Call File System MCP to write corrected code
  7. Call GitHub MCP to create a new PR with the fix

Multi-Agent Coordination

Agents can be MCP hosts with their own servers, or expose capabilities as MCP servers consumed by other agents. Enables hierarchical systems: planning agent delegates to specialists.

Why This Works

Every tool speaks the same protocol. No custom integration between GitHub, file system, and database tools. The model orchestrates them seamlessly through a uniform interface.

MCP Sits Above Native Function Calling

MCP does not replace native tool-use APIs. It defines how the host discovers and executes tools on the server side. Tools are presented to the model through its native function-calling interface.

Provider-Agnostic

The same MCP servers work with Claude, open-source models, and any LLM supporting tool use. Build once, connect to any provider.

OUTLOOK

Limitations & Future Directions

Current Limitations

  • Stateless vs. stateful trade-off: stateless scales easily but loses session context
  • Limited guidance on retry semantics, timeouts, graceful degradation
  • Date-based versioning + rapid evolution = potential version mismatches
  • Security model still maturing: tool poisoning and prompt injection remain practical risks
  • Context window pressure from large tool results

Future Directions

  • Richer media types — streaming audio, video, interactive content
  • Standardized auth profiles — pre-built OAuth configs for common IdPs
  • Formal tool verification — cryptographic signing of tool descriptions
  • Inter-server communication — servers invoke each other directly
  • Observability standards — OpenTelemetry integration for tracing and metrics

SUMMARY

Key Takeaways

What MCP Delivers

  • Eliminates the N×M integration problem with a universal standard
  • Six primitives cover tools, resources, prompts, sampling, roots, logging
  • Dynamic discovery — models find tools at runtime
  • Bidirectional sampling enables recursive agentic patterns
  • Growing ecosystem: SDKs in 6+ languages, hundreds of pre-built servers

Practitioner Guidance

  • MCP should be the default integration mechanism for LLM tools
  • Use stdio for local development, Streamable HTTP for remote
  • Enforce least privilege, human-in-the-loop for side effects
  • Validate inputs, sanitize outputs, review tool descriptions
  • Tool providers: expose via MCP for maximum reach

Bottom Line

MCP is to LLM tool integration what USB was to peripherals and LSP was to IDEs. It provides the foundational infrastructure layer connecting AI models to external data and services — build once, connect everywhere. As the protocol matures, it is positioned to become the standard that powers the next generation of agentic AI applications.
