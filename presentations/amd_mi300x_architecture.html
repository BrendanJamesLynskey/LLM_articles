CDNA 3 Chiplet Architecture

Multi-Die Design

  • 8 XCDs (Accelerator Complex Dies) on TSMC 5nm
  • 4 IODs (I/O Dies) on TSMC 6nm
  • 3D stacking: XCDs sit atop IODs
  • 2.5D silicon interposer connects all dies
  • ~153 billion transistors total

Chiplet Advantages

  • Higher yield: smaller dies = fewer defects per die
  • Separate process optimization for compute vs I/O
  • Modular: swap GPU XCDs for CPU dies → MI300A
  • Lower cost per transistor vs monolithic design

Why Memory Matters for LLM Inference

No Quantization Needed

  • 70B FP16 = 140 GB weights
  • H100 (80 GB): must quantize or split across 2 GPUs
  • MI300X: fits on 1 GPU with 50+ GB to spare
  • 100B–180B models: single-GPU FP16 serving

Bigger KV Caches

  • Long-context (128K–256K tokens) KV caches: tens of GB
  • More memory → more concurrent requests
  • Higher throughput per GPU with continuous batching
  • vLLM PagedAttention benefits directly

Larger Batch Sizes

  • Memory headroom allows more sequences in-flight
  • Better GPU utilization at high concurrency
  • Improved cost per token in throughput-optimized serving
  • Fewer OOM errors in production

The Core Insight

LLM inference is memory-bandwidth-bound during decode: each token requires reading all model weights from HBM. More bandwidth = faster tokens. More capacity = larger models, longer contexts, bigger batches — all on fewer GPUs.

MI300A: The APU Variant

Unified CPU + GPU

  • 24 Zen 4 CPU cores + GPU CUs in one package
  • Shared 128 GB unified HBM3
  • Coherent memory: no PCIe bottleneck for CPU↔GPU
  • Some GPU XCDs replaced with CPU CCDs
  • Same chiplet platform as MI300X

Target Workloads

  • HPC applications with tight CPU-GPU coupling
  • Supercomputer deployments (El Capitan at LLNL)
  • Scientific simulations mixing CPU + GPU code
  • Unified memory simplifies programming model

ROCm Software Stack

HIP & CUDA Compatibility

  • HIP: thin translation layer mirroring CUDA API
  • hipify tool auto-converts CUDA → HIP source
  • Many CUDA programs port with minimal changes
  • Complex kernels with PTX/intrinsics need manual work

Framework Support

  • PyTorch: official ROCm support since v1.8
  • vLLM: ROCm backend for continuous batching
  • JAX, TensorFlow, DeepSpeed: varying support levels
  • Flash Attention 2: functional with competitive perf

Historical Pain Points

  • Flash Attention was CUDA-only initially
  • AMD had to develop own impl via Composable Kernel
  • Weeks-to-months lag when new techniques emerge in CUDA

Limitations

ROCm Maturity

  • Smaller ecosystem of optimized kernels
  • Less polished profiling & debugging tools
  • Fewer community resources (docs, tutorials)
  • New techniques arrive on CUDA weeks-months first

Kernel Optimization Gap

  • cuBLAS/cuDNN: decades of optimization
  • rocBLAS/MIOpen improving but behind
  • Lower % of peak hardware utilization
  • Specs impress; realized performance varies

Ecosystem Lock-In

  • Production systems deeply tied to CUDA
  • Monitoring, profiling, orchestration tooling
  • Operational switching cost beyond code porting
  • Staff expertise concentrated on NVIDIA

Flash Attention History

Original FlashAttention was CUDA-only. AMD built own impl via Composable Kernel. FA2 now functional & competitive on MI300X, but the delay illustrates the ecosystem lag pattern.

The Counterpoint

ROCm is open source. Community contributions accelerating. AMD investing heavily. Gap is narrowing with each release. vLLM + PyTorch support now solid.

Key Takeaways

What MI300X Gets Right

  • 192 GB HBM3: serve larger models on fewer GPUs
  • 5.3 TB/s: real throughput gains for memory-bound inference
  • Chiplet design: cost-effective, modular, high-yield
  • Cloud adoption: Azure, Oracle, Meta validate production use
  • Credible NVIDIA alternative for the first time

What Remains Challenging

  • ROCm ecosystem still trails CUDA in breadth & polish
  • Kernel optimization gap reduces realized performance
  • Blackwell neutralizes memory advantage
  • Operational switching costs for CUDA-native teams
  • MI350 must deliver on CDNA 4 promises

Best Fit

  • Inference where memory capacity is binding
  • Teams with ROCm engineering capacity
  • Multi-vendor strategy to reduce NVIDIA dependency
  • Workloads where fewer GPUs = operational win

Strategic Significance

  • First AMD GPU taken seriously for production AI
  • Proves memory-first strategy can challenge NVIDIA
  • Forces NVIDIA to increase HBM (H200, B200)
  • Competition benefits the entire ecosystem
