CONTEXT

Why Deployment Matters

The MCP specification defines a clean protocol for LLM–tool integration — but shipping a production-grade MCP server requires solving an entirely separate set of engineering problems.

The Gap

  • Protocol spec says nothing about how to run servers
  • No standard packaging, deployment, or lifecycle model
  • Security, scaling, and monitoring are left to implementers
  • Most tutorials stop at “npx runs locally”

Production Reality

  • Teams need multi-user access to shared servers
  • Servers must survive restarts, handle load, and report health
  • Credentials must be managed securely at scale
  • Deployment must be repeatable and auditable

Key Insight

Building an MCP server is the easy part. Operating it — reliably, securely, at scale — is where most of the engineering effort lives.

LOCAL

The stdio Deployment Model

The simplest deployment: the host application spawns the MCP server as a child process and communicates over stdin/stdout using JSON-RPC.

How It Works

  • Host reads claude_desktop_config.json or equivalent
  • Spawns each server via command + args
  • Injects env variables (API keys, tokens)
  • Server lifecycle is tied to host process
  • One server instance per host — no sharing

Limitations

  • Cannot share a server across multiple clients
  • No network access — local filesystem only
  • Secrets stored in plaintext JSON config files
  • No health checks, metrics, or restart policies

Claude Desktop config example:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/dev/projects"
      ]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxx"
      }
    }
  }
}

CONTAINERS

Containerized Deployment

Docker Pattern

  • Package MCP server + runtime in a Docker image
  • Expose HTTP endpoint (SSE or Streamable HTTP)
  • Use mcp-proxy or supergateway to bridge stdio → HTTP
  • Mount secrets via Docker Secrets or env injection
  • Health checks via HEALTHCHECK directive

Kubernetes Pattern

  • Deploy as a Deployment with readiness/liveness probes
  • Service mesh for mTLS between clients and servers
  • Horizontal Pod Autoscaler for load-based scaling
  • ConfigMaps for tool configuration, Secrets for credentials
  • Rolling updates with zero-downtime deploys

Dockerfile with stdio-to-HTTP bridge:

FROM node:22-slim
WORKDIR /app
RUN npm install -g @modelcontextprotocol/server-github \
    supergateway

ENV GITHUB_PERSONAL_ACCESS_TOKEN=""

EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=5s \
  CMD curl -f http://localhost:8080/health || exit 1

CMD ["supergateway", \
     "--stdio", "mcp-server-github", \
     "--port", "8080"]

Docker Compose for multi-server stack:

services:
  github-mcp:
    build: ./servers/github
    ports: ["8081:8080"]
    secrets: [github_token]
  postgres-mcp:
    build: ./servers/postgres
    ports: ["8082:8080"]
    secrets: [db_connection_string]
secrets:
  github_token:
    environment: "GITHUB_TOKEN"
  db_connection_string:
    environment: "DATABASE_URL"

SERVERLESS

Serverless & Cloud Deployment

AWS Lambda

  • Streamable HTTP via Function URLs
  • Cold starts: 200–800 ms (Node.js)
  • Stateless — no persistent connections
  • Use DynamoDB for session state
  • Pay-per-invocation cost model

Cloud Run / Fargate

  • Container-native — SSE supported
  • Min instances = 1 eliminates cold starts
  • Scale to zero for cost optimization
  • Long-lived connections up to 60 min
  • Best balance of serverless + flexibility

Challenges

  • Cold starts break interactive UX
  • SSE requires long-lived connections — conflicts with request-based billing
  • Session affinity hard to guarantee
  • Timeout limits (Lambda: 15 min max)
  • No stdio — HTTP transport required

Recommendation

Cloud Run or AWS Fargate with min-instance=1 is the pragmatic serverless choice for MCP servers. Pure Lambda works for Streamable HTTP (stateless request/response), but SSE-based servers need persistent compute.

TRANSPORT

Transport Layer Evolution

stdio

  • Original transport — still the default for local
  • JSON-RPC over stdin/stdout
  • Zero network overhead
  • Cannot cross machine boundaries
  • Status: Stable, not deprecated

HTTP + SSE (Legacy)

  • Added for remote servers in early MCP
  • Two endpoints: POST for requests, GET for SSE stream
  • Deprecated in spec 2025-03-26
  • Difficult to proxy, load-balance, and resume
  • Still supported in backward-compat mode

Streamable HTTP

  • New standard replacing SSE transport
  • Single POST endpoint; optional SSE upgrade
  • Stateless by default, sessions opt-in via Mcp-Session-Id
  • Works with standard HTTP infra (CDNs, load balancers)
  • Resumability via Last-Event-ID

Bridging Tools

supergateway — wraps stdio servers in SSE or Streamable HTTP. mcp-proxy — bidirectional stdio ↔ SSE bridge. mcp-remote — connects local stdio clients to remote HTTP servers with OAuth support.

CONFIG

Configuration & Discovery

JSON Configuration Files

  • Claude Desktop: claude_desktop_config.json
  • Cursor: .cursor/mcp.json (project) or global config
  • VS Code: .vscode/mcp.json with input variable support
  • Format varies by host — no universal standard yet

MCP Registry & Discovery

  • Well-known URI: /.well-known/mcp.json for server metadata
  • MCP Registry: centralized directory of published servers
  • DNS-SD and mDNS for local network discovery
  • OAuth 2.1 metadata at /.well-known/oauth-authorization-server

Environment Variables

  • Secrets injected at runtime via env block
  • Supports ${input:variable} syntax in VS Code for prompted input
  • Production: use vault references, not literals

VS Code mcp.json with inputs:

{
  "inputs": [
    {
      "id": "github-token",
      "type": "promptString",
      "description": "GitHub PAT",
      "password": true
    }
  ],
  "servers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN":
          "${input:github-token}"
      }
    }
  }
}

Well-known discovery endpoint:

GET /.well-known/mcp.json

  "endpoint": "/mcp",

LIFECYCLE

Server Lifecycle Management

Startup & Initialization

  • initialize handshake: client sends capabilities, server responds with tool list
  • Server must be ready before accepting tool calls
  • Pre-warm connections to databases, APIs on startup
  • Readiness probe should gate traffic until init completes

Health Checks

  • Liveness: is the process alive? (restart if not)
  • Readiness: can it serve requests? (remove from LB if not)
  • MCP ping method as native health check
  • Include downstream dependency checks (DB, API reachability)

Graceful Shutdown

  • Handle SIGTERM — stop accepting new requests
  • Drain in-flight tool calls (configurable timeout)
  • Close database connections, flush logs
  • K8s preStop hook for connection draining

Rolling Updates

  • maxSurge / maxUnavailable for zero-downtime
  • Version negotiation during MCP initialize handshake
  • Canary deployments: route 5% of traffic to new version
  • Automated rollback on error-rate spike

SECURITY

Security in Production

Credential Management

  • Never store secrets in config JSON or env files in repos
  • Use HashiCorp Vault, AWS Secrets Manager, or K8s Secrets
  • Rotate credentials automatically; short-lived tokens preferred
  • OAuth 2.1 for remote servers — no long-lived API keys
  • MCP spec mandates credential isolation between servers

Network Isolation

  • Run MCP servers in private subnets — no public endpoints
  • K8s NetworkPolicies: whitelist only known client pods
  • mTLS between all services via service mesh
  • Egress filtering — servers can only reach approved APIs

Sandboxing

  • Containers run as non-root with read-only filesystem
  • seccomp and AppArmor profiles to restrict syscalls
  • Resource limits (CPU, memory) to prevent abuse
  • No privilege escalation: allowPrivilegeEscalation: false
  • Filesystem servers: bind-mount only approved directories

Least Privilege

  • Each server gets minimal permissions for its function
  • Database servers: read-only access unless writes are required
  • GitHub servers: scope tokens to specific repos
  • Audit all tool invocations — log caller, tool, args, result
  • Human-in-the-loop for destructive operations

SCALING

Scaling & High Availability

Horizontal Scaling

  • MCP servers should be stateless where possible
  • Kubernetes HPA scales on CPU, memory, or custom metrics
  • Target: <200 ms p99 latency per tool call
  • Scale separately per server type (DB server needs fewer replicas than search)

Session Affinity

  • Streamable HTTP Mcp-Session-Id header ties client to session
  • Sticky sessions via load balancer cookie or header hash
  • Session state in Redis/DynamoDB for multi-replica setups
  • Stateless mode: no session required for simple tool calls

Load Balancing

  • L7 load balancer for HTTP transport
  • SSE connections: use least-connections algorithm
  • Connection draining on scale-down events
  • Circuit breakers per downstream dependency

Connection Pooling

  • Pool DB connections across concurrent tool calls
  • HTTP keep-alive to downstream APIs
  • Connection limits prevent resource exhaustion
  • MCP gateway pattern: single entry point multiplexes to N servers

OBSERVABILITY

Monitoring & Observability

Metrics

  • Tool call rate — calls/sec per tool, per server
  • Latency histograms — p50, p95, p99 per tool
  • Error rate — 4xx vs 5xx breakdown
  • Active connections / sessions
  • Resource usage: CPU, memory, FD count
  • Export via Prometheus or OTLP

Distributed Tracing

  • OpenTelemetry spans from client → server → downstream
  • Trace ID propagation through MCP request headers
  • Correlate LLM inference with tool execution
  • Identify slow tools, bottleneck dependencies
  • Visualize in Jaeger, Grafana Tempo, or Datadog

Alerting & Logging

  • Structured JSON logs with request IDs
  • Alert on: error rate > 1%, latency p99 > 2s, pod restarts
  • Log all tool invocations for audit trail
  • Centralize in ELK, Loki, or CloudWatch
  • Dashboards per server: health, throughput, errors

Development Workflow

Local Development

  • npx / uvx for instant server execution — no install
  • npx @modelcontextprotocol/server-github
  • uvx mcp-server-sqlite --db-path ./test.db
  • Hot reload with --watch or nodemon
  • TypeScript SDK: @modelcontextprotocol/sdk
  • Python SDK: mcp[cli]

MCP Inspector

  • Official debugging tool: npx @modelcontextprotocol/inspector
  • Web UI to browse tools, resources, prompts
  • Send test calls, inspect JSON-RPC messages
  • Validate server compliance before deployment

Testing Strategy

  • Unit tests: test tool handlers in isolation
  • Integration tests: spin up server, send JSON-RPC, assert responses
  • Contract tests: validate against MCP schema
  • Use in-memory transport for fast test execution
  • Mock downstream APIs for deterministic results

CI/CD Pipeline

  • Lint → type-check → unit test → build image
  • Integration test against containerized server
  • MCP Inspector validation in CI
  • Push image to registry, tag with git SHA
  • Deploy to staging → smoke test → promote to prod
  • GitOps: ArgoCD / Flux watches image tags

PATTERNS

Production Deployment Patterns

Single-Org Internal

  • All MCP servers on private infrastructure
  • Internal container registry (ECR, GCR, Harbor)
  • Network-level access control — VPN or service mesh
  • Centralized secret management via Vault
  • Shared observability stack (Grafana + Prometheus)
  • Teams own their servers, platform team owns infra

Multi-Tenant SaaS

  • Serve multiple customers from shared servers
  • Tenant isolation via OAuth scopes + request context
  • Rate limiting per tenant, per tool
  • Data isolation: tenant-scoped queries only
  • Noisy-neighbor mitigation with resource quotas
  • Compliance: SOC 2, data residency controls

Hybrid Local + Remote

  • Sensitive tools run locally via stdio (filesystem, DB)
  • Shared tools run remotely via HTTP (search, analytics)
  • MCP gateway aggregates both into single interface
  • mcp-remote bridges local clients to remote servers
  • Best of both: security for sensitive data, scale for shared tools
  • Most common pattern in enterprise adoption

Emerging: MCP Gateway as Infrastructure Layer

Products like Cloudflare MCP Gateway, Toolhouse, and MCP Hub are creating a new infrastructure category — a managed proxy layer that handles auth, rate limiting, caching, and routing for fleets of MCP servers, analogous to an API gateway for REST services.

  • ArgoCD ApplicationSets for multi-server GitOps

Automated Deployment

  • Image build triggers on git push to main
  • Progressive delivery: canary → 10% → 50% → 100%
  • Automated rollback on health check failure
  • Feature flags for new tool activation
  • Blue/green for major version upgrades

Governance

  • Approval workflows for new server deployments
  • Security scanning of server images (Trivy, Snyk)
  • Policy-as-code: OPA/Gatekeeper for deployment rules
  • Cost allocation per server, per team
  • Compliance audit trail for all tool invocations

PITFALLS

Common Deployment Pitfalls

Pitfall Impact Mitigation
Secrets in config files Credential exposure in repos or logs Use vault injection; never commit .env or config with tokens
No health checks Zombie servers serving errors silently Implement liveness + readiness probes; use MCP ping
Running as root Container escape, privilege escalation Non-root user, read-only FS, seccomp profiles
No resource limits Single server starves cluster Set CPU/memory requests and limits; use LimitRanges
Ignoring transport deprecation SSE servers stop working with new clients Migrate to Streamable HTTP; use bridge tools during transition
No graceful shutdown In-flight tool calls dropped on deploy Handle SIGTERM; drain connections; use preStop hooks
Monolithic server All tools coupled; one failure breaks everything One server per domain; independent scaling and deployment
No rate limiting Runaway agent loops exhaust downstream APIs Per-tool rate limits; circuit breakers; cost caps
Stateful assumptions Breaks when load-balanced across replicas Externalize state to Redis/DB; use session affinity headers

SUMMARY

Recommendations

Start Here

  • Begin with stdio locally for development and testing
  • Use MCP Inspector to validate before deploying
  • Containerize early — Dockerfile from day one
  • Adopt Streamable HTTP transport for any remote deployment
  • One server per bounded context — keep servers small and focused

Production Checklist

  • Health checks (liveness + readiness probes)
  • Graceful shutdown with connection draining
  • Secrets via vault, never in config files
  • Non-root containers with resource limits
  • OpenTelemetry for traces, metrics, and logs
  • Rate limiting and circuit breakers per tool

Scale Gradually

  • Phase 1: Docker Compose for team staging
  • Phase 2: Kubernetes with Helm for production
  • Phase 3: MCP gateway for fleet management
  • Do not over-engineer — most teams need 3–5 servers, not 50

Watch the Ecosystem

  • MCP Registry standardizing discovery and metadata
  • OAuth 2.1 becoming default auth for remote servers
  • Gateway products (Cloudflare, Toolhouse) maturing rapidly
  • Expect a “Kubernetes for MCP” orchestration layer to emerge

Key Takeaway

MCP server deployment is converging toward standard cloud-native patterns — containers, service meshes, observability, and GitOps. The protocol is new, but the operational playbook is well-established. Teams that treat MCP servers as first-class microservices from the start will avoid the most common pitfalls.
