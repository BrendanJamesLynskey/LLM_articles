CONTEXT

The Memory Wall Problem

Autoregressive LLM decoding is memory-bandwidth bound, not compute bound. Each token generation reads the full KV cache and model weights from memory.

GPUs address this with massive HBM bandwidth, but the von Neumann bottleneck remains: data must travel from memory to compute and back, consuming energy and time on every operation.

SambaNova asks: what if data moved through compute instead?

GPU Inference Bottleneck

  • H100: 3.35 TB/s HBM bandwidth
  • Llama 405B FP16: ~810 GB weights
  • Each decode step reads full weights
  • Arithmetic intensity < 1 op/byte
  • Compute utilization often < 30%

CHIP

SN40L Reconfigurable Dataflow Unit

Pattern Compute Units

Array of PCUs handling FP16, BF16, FP8, INT8 arithmetic. Spatially configured to match the compute graph of each neural network layer.

Pattern Memory Units

Distributed on-chip SRAM acting as software-managed scratchpad. Co-located with PCUs to minimize data movement latency.

Reconfigurable Network

On-chip interconnect whose topology is reconfigured at runtime. Routes data between PCUs/PMUs to match the model's dataflow pattern.

Unlike a GPU's fixed pipeline, the RDU's compute topology adapts to each model layer. Attention heads use a different spatial mapping than FFN layers, maximizing utilization across both compute-bound and memory-bound phases.

MEMORY

Three-Tier Memory Hierarchy

DDR capacity allows very large models on fewer chips than HBM-only designs. The compiler prefetches next-layer weights from DDR into SRAM while the current layer executes, hiding memory latency behind computation.

SOFTWARE

SambaFlow Compiler Stack

Compilation Pipeline

  1. Graph Import — PyTorch / ONNX model ingestion
  2. Graph Optimization — Operator fusion, tiling, layout transforms
  3. Spatial Mapping — Assign ops to PCUs, route data through network
  4. Memory Scheduling — Three-tier data placement & prefetch planning
  5. Code Generation — RDU configuration bitstream

Analogous to FPGA Place-and-Route

The spatial mapping step resembles FPGA toolchains but operates at a coarser granularity — mapping neural network operations rather than individual logic elements.

Key difference: the RDU fabric can be reconfigured between layers or between requests, enabling per-layer topology optimization.

RECONFIGURATION

Software-Defined Hardware Adaptation

The same physical silicon adopts different topologies for each layer type, optimizing both compute-bound and memory-bound phases of the transformer forward pass.

CoE

Composition of Experts

SambaNova's Composition of Experts (CoE) runs multiple specialized models on shared RDU hardware with near-zero switching overhead.

How It Works

  • Multiple fine-tuned expert models reside across the three-tier memory
  • Query router dispatches to the appropriate domain expert
  • RDU reconfiguration swaps model configs in milliseconds
  • Contrast with GPU: weight reload takes seconds

Use Cases

  • Enterprise with diverse workloads (code, legal, medical, finance)
  • Each expert fine-tuned for its domain, smaller than a generalist model
  • Shared infrastructure pool, dynamic allocation

COMPARISON

GPU vs. LPU vs. RDU

NVIDIA GPU

  • Pros: Mature ecosystem, CUDA libraries, universal model support
  • Cons: Von Neumann bottleneck, high power draw, memory-BW limited for decode
  • Memory: HBM only (80-192 GB)
  • Best for: Training + general inference

Groq LPU

  • Pros: Deterministic latency, extreme tok/s, SRAM-only speed
  • Cons: Limited capacity per chip, needs many chips for large models
  • Memory: SRAM only (~230 MB/chip)
  • Best for: Low-latency, smaller models

SambaNova RDU

  • Pros: Reconfigurable fabric, 3-tier memory, CoE, energy efficient
  • Cons: Ecosystem maturity, compilation complexity, narrow model support
  • Memory: SRAM + HBM + DDR (100s GB)
  • Best for: Large model inference, multi-model deployments

SCALING

Multi-Chip DataScale Systems

Rack-Scale Architecture

A DataScale rack contains 8 RDU nodes connected by a high-bandwidth chip-to-chip fabric that extends the dataflow paradigm across node boundaries.

Parallelism Strategies

  • Tensor parallelism — split individual operations across RDUs
  • Pipeline parallelism — different layers on different RDUs
  • Expert parallelism — MoE experts distributed across nodes

SambaFlow handles multi-chip partitioning automatically, enabling trillion-parameter models while maintaining dataflow latency advantages.

EFFICIENCY

Energy Efficiency Claims

Why Dataflow Is More Efficient

  • Data moves through compute, not fetched from global memory
  • Eliminates redundant memory read/write cycles
  • On-chip SRAM access: ~5 pJ/access vs. HBM ~20 pJ
  • No instruction fetch/decode overhead per operation
  • Spatial execution avoids branch prediction and speculation energy

Reported Figures

  • 2-3x better perf/watt vs. A100/H100 for large model inference
  • Particularly advantageous for decode phase where arithmetic intensity is low
  • DDR tier uses less power than equivalent HBM capacity
  • Independent verification at scale remains limited

Energy efficiency is especially relevant for always-on inference services where TCO is dominated by power costs. The dataflow model's elimination of unnecessary data movement translates directly to reduced energy per token.

TRADE-OFFS

Limitations and Challenges

Ecosystem Maturity

  • CUDA ecosystem has 20+ years of libraries, tools, community
  • SambaFlow is proprietary and closed-source
  • Fewer practitioners with hands-on RDU experience
  • Debugging and profiling tools less mature than NVIDIA's Nsight

Model Support Breadth

  • Popular architectures (Llama, Mistral, DeepSeek) well-optimized
  • Exotic or rapidly evolving architectures need additional compiler work
  • New model architectures may take weeks to support vs. days on GPU

Compilation Complexity

  • Spatial mapping is NP-hard in general
  • Compilation for new architectures can take hours
  • Performance tuning across 3-tier memory not always automatic
  • Manual intervention sometimes needed for optimal placement

Market Position

  • Small installed base limits network effects
  • Talent pool: hard to hire engineers with RDU experience
  • Training workloads still GPU-dominated
  • Vendor lock-in concerns for enterprise customers

SUMMARY

Key Takeaways

What Makes SN40L Distinctive

  1. Dataflow paradigm — data moves through compute, eliminating the von Neumann bottleneck
  2. Three-tier memory — SRAM + HBM + DDR enables large models on fewer chips
  3. Runtime reconfiguration — hardware adapts topology per layer and per model
  4. Composition of Experts — millisecond model switching on shared hardware
  5. Energy efficiency — 2-3x perf/watt advantage from reduced data movement

Open Questions

  1. Can SambaFlow keep pace with the rapid evolution of model architectures?
  2. Will the ecosystem mature fast enough to compete with CUDA's network effects?
  3. Can CoE become a standard deployment pattern beyond SambaNova hardware?
  4. How will independent benchmarks validate the efficiency claims at scale?
  5. Will training workloads ever shift to RDU, or will it remain inference-focused?

SambaNova occupies a genuine architectural niche for large-model inference. Whether the RDA paradigm scales its ecosystem to match GPU ubiquity will determine its trajectory from specialized alternative to mainstream infrastructure.
