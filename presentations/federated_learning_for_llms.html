PROBLEM

The Data-Sharing Wall

Most LLM training assumes data lives in one place. Some valuable data simply cannot be centralized: hospital records, bank transactions, classified data, trade secrets.

Use Cases Blocked

  • Hospital networks training medical LLMs
  • Banks training fraud models on combined data
  • Government agencies across classification levels
  • Pharma consortia for drug discovery

What Federated Learning Promises

  • Train shared model across silos
  • Raw data never leaves the participant
  • Only model updates flow over network
  • Coordinator aggregates updates

FEDAVG

The Basic Loop

FedAvg Procedure

  1. Coordinator sends model weights to participants
  2. Each participant trains locally for K steps
  3. Each sends weight update back
  4. Coordinator averages updates — weighted by data size
  5. Repeat

Cross-Device vs Cross-Silo

  • Cross-device: many small participants (phones)
  • Cross-silo: few large participants (orgs)
  • LLMs are cross-silo
  • Each silo has its own GPU cluster

PRACTICAL

Federated Fine-Tuning + LoRA

Pre-training is impractical for federation. Fine-tuning is the right granularity. Combined with LoRA, the communication shrinks dramatically.

The Workflow

  1. All participants start from public base model
  2. Each fine-tunes locally for some steps
  3. LoRA adapters sent to coordinator (100–1000× smaller than full weights)
  4. Coordinator averages adapters
  5. New global adapter, repeat

Communication Win

  • Full 70B fp16 update: 140 GB
  • LoRA r=16 update: ~50 MB
  • ~3000× reduction
  • Practical over modest network bandwidth

Quality

Federated LoRA fine-tuning typically reaches quality competitive with what would be achieved by centralized training on the pooled data.

PRIVACY

How Private Is It Really?

Attacks on Updates

  • Membership inference: was X in the training set?
  • Model inversion: reconstruct examples from gradients
  • Property inference: deduce dataset properties
  • LLMs especially leak via memorization

Defenses

  • Differential privacy: add noise to gradients (hurts quality)
  • Secure aggregation: cryptographic sum (hides individual updates)
  • Trusted execution environments
  • None are complete on their own

The Honest Framing

Federated learning is “more private than centralized training, less private than the marketing suggests.” Raw data is protected by org boundaries; model updates leak limited statistical information. Real protection but not equivalent to never sharing.

LIMITS

What Federation Can’t Do

Hard Cases

  • Pre-training from scratch — too expensive
  • Very heterogeneous data distributions
  • Adversarial participants (Byzantine robustness needed)
  • Continual learning over time

Alternatives

  • Differential privacy on centralized data
  • Synthetic data generation
  • Confidential computing in TEEs
  • Homomorphic encryption (slow but airtight)

SUMMARY

A Niche but Important Tool

What’s True

  • Federated LoRA fine-tuning is production-ready in 2026
  • Healthcare and pharma consortia have real deployments
  • Communication challenges solved for fine-tuning, not pre-training
  • Privacy guarantees are real but limited

Practitioner’s Lesson

If you have data that cannot be centralized but would benefit from being combined — use federated learning. If your data could be pooled centrally without legal or organizational obstacles, federation just adds complexity. It’s a tool for a specific problem, not a general improvement over centralized training.
