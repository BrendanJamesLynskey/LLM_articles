Habana Labs & Intel

Company Origins

  • Founded 2016 in Israel by David Dahan
  • Team from networking & signal processing ASIC backgrounds
  • Thesis: AI accelerators must treat compute and networking as co-equal design priorities
  • Built Gaudi (training) and Goya (inference) processors

Intel Acquisition

  • Acquired by Intel in December 2019 for ~$2B
  • Central pillar of Intel's data center AI strategy
  • Brought both silicon and SynapseAI software stack
  • Positioned as the CUDA alternative in Intel's portfolio
  • Continued development under Intel Datacenter & AI Group

Design Philosophy

Integrate RDMA-capable Ethernet NICs directly on the processor die, eliminating external network adapters and proprietary interconnects. The same Ethernet fabric handles both intra-node and inter-node communication.

Integrated Ethernet vs. Proprietary Interconnects

Gaudi: On-Die Ethernet RDMA

  • Standard Ethernet — no vendor lock-in
  • RDMA NICs integrated on the processor die
  • No external NIC cards needed
  • Direct-connect mesh within a server
  • Same fabric for intra-node & inter-node
  • Lower BoM cost for cluster builds

Scale-Out Without Switches

Eight Gaudi chips in a server connect via integrated ports in a full mesh. No NVLink bridges, no NVSwitch, no dedicated network adapters for the accelerator interconnect.

NVIDIA: Proprietary Stack

  • NVLink for intra-node (900 GB/s on H100)
  • NVSwitch chips for full bisection BW
  • InfiniBand (ConnectX) for inter-node
  • Separate network adapters required
  • Higher system complexity & cost

Gaudi 2 Architecture

Compute

  • 2 Matrix Math Engines (BF16, FP32, FP8)
  • 24 Tensor Processor Cores with VLIW ISA
  • Enhanced TPC pipelines with wider execution
  • Each TPC has local SRAM scratchpad

Memory

  • 96 GB HBM2e — fits 70B models in FP8
  • 2.45 TB/s memory bandwidth
  • 3x capacity vs Gaudi 1

Integrated Networking

  • 24× 100GbE RDMA ports on-die
  • 2.4 Tb/s aggregate bisection BW
  • 21 ports for intra-node full mesh (8 cards)
  • 3 ports for inter-node scale-out

System-Level Savings

  • No NVLink bridges needed
  • No NVSwitch chips needed
  • No dedicated NICs for accelerator fabric
  • Lower BoM vs H100 HGX configuration

Tensor Processor Core & VLIW ISA

VLIW Architecture

  • Very Long Instruction Word ISA
  • Multiple independent ops per cycle:
  •   • Vector processing unit
  •   • Scalar processing unit
  •   • Load/store unit
  •   • Special function unit
  • Compiler packs ILP explicitly into wide instructions
  • Local SRAM scratchpad per TPC

Programmability

Developers write custom TPC kernels in C-like syntax via the Habana TPC programming interface. Enables support for novel activations, attention mechanisms, and normalization schemes.

TPC vs MME Division

  • MME: GEMM operations (matmuls)
  • TPC: everything else:
  •   • Activation functions
  •   • Layer normalization
  •   • Softmax
  •   • Rotary positional embeddings
  •   • Custom fused kernels

Why It Matters

Fixed-function accelerators struggle when model architectures evolve. TPC programmability gives Gaudi flexibility to support new transformer variants without silicon changes.

Multi-Node Scaling

Integrated RDMA Advantage

  • Collective ops (all-reduce, all-gather) execute with minimal software overhead
  • No PCIe traversal to reach external NIC
  • Lower latency for gradient synchronization
  • Higher scaling efficiency at 64, 128, 256 nodes

Cluster Topology

  • Intra-node: full mesh via integrated 200GbE
  • Inter-node: remaining ports → spine switches
  • Standard Ethernet switching — commodity hardware
  • RoCE v2 for low-latency collectives

DeepSpeed at Scale

ZeRO + integrated RDMA enables efficient gradient sharding across hundreds of Gaudi accelerators with communication-efficient algorithms.

Porting from CUDA to Gaudi

What's Easy

  • Standard PyTorch models: device swap only
  • cuda → hpu in most cases
  • Hugging Face models via Optimum Habana
  • DeepSpeed training scripts
  • vLLM inference serving
# Minimal porting example
import torch
- device = torch.device('cuda')
+ import habana_frameworks.torch
+ device = torch.device('hpu')

model = model.to(device)
# ... rest of training loop unchanged

What's Hard

  • Custom CUDA kernels — must rewrite for TPC
  • TensorRT-LLM pipelines — no equivalent
  • Triton GPU kernels — not supported
  • NCCL-dependent code — use HCCL instead
  • Unsupported ops require TPC kernel development

Migration Strategy

  • Start with validated models from Optimum Habana
  • Use Tiber AI Cloud for evaluation
  • Profile with SynapseAI profiler
  • Incrementally port custom components

Limitations

Software Ecosystem Lag

  • Long tail of CUDA libraries does not exist for Gaudi
  • No TensorRT-LLM equivalent
  • No Triton kernel support
  • Custom CUDA extensions require rewrite
  • Primary barrier to adoption

Smaller Community

  • Orders of magnitude fewer developers
  • Fewer tutorials & Stack Overflow answers
  • Slower bug identification & resolution
  • Smaller catalog of optimized models
  • Novel architectures: limited support

Strategic Uncertainty

  • Intel AI strategy: multiple pivots
  • Nervana → Habana → Gaudi → Falcon Shores?
  • Organizational turbulence
  • Customers need multi-year confidence
  • Competing with Intel's own GPU line

The Core Tension

Gaudi hardware is technically competitive with NVIDIA on benchmarks and cheaper per node. But CUDA's ecosystem moat — thousands of libraries, millions of developers, a decade of tooling — means that raw hardware performance alone cannot drive adoption. The switching cost is the software porting burden.

When to Choose Gaudi

Good Fit

  • Cost-sensitive deployments at scale (100+ accelerators)
  • Standard models: Llama, Mixtral, Falcon, GPT-NeoX
  • PyTorch + DeepSpeed training workflows
  • vLLM-based inference serving
  • Organizations wanting to avoid vendor lock-in
  • Clusters where networking cost is a major factor
  • Environments that prefer standard Ethernet over InfiniBand

Poor Fit

  • Heavy reliance on custom CUDA kernels
  • TensorRT-LLM or Triton-based inference pipelines
  • Novel/experimental model architectures
  • Organizations needing large community support
  • Workloads requiring NVIDIA-specific features (NVLink BW)
  • Research teams iterating on architecture rapidly

The Economic Argument

For organizations deploying well-established open-weight models at scale, Gaudi offers a path to 30–40% lower infrastructure cost by eliminating proprietary networking, reducing per-node price, and leveraging competitive Tiber AI Cloud pricing. The porting investment pays off at scale.

Key Takeaways

Architecture Strengths

  • Integrated Ethernet RDMA eliminates NVLink/NVSwitch/external NICs
  • 128 GB HBM2e at 3.7 TB/s (exceeds H100 capacity)
  • 64 programmable TPCs with VLIW ISA
  • FP8 with 4x gen-over-gen improvement
  • Both training and inference capable

Software Ecosystem

  • SynapseAI + PyTorch bridge (torch.hpu)
  • DeepSpeed full integration
  • vLLM backend for inference serving
  • Optimum Habana for Hugging Face models
  • Growing but still far behind CUDA

Competitive Position

  • Wins on cost-performance ratio
  • Wins on networking simplicity and cost
  • Competitive on raw specs vs H100/H200
  • Tiber AI Cloud for low-friction evaluation

Key Risks

  • CUDA ecosystem moat is the primary barrier
  • Smaller developer community
  • Intel strategic uncertainty (Falcon Shores overlap)
  • Porting burden for custom workloads

Gaudi's path: competitive hardware + open networking + cost advantage. The challenge is building the software ecosystem to match.
