PROBLEM

Why Trust Is the Bottleneck

Sending data to a hosted LLM means trusting the provider, the cloud, the hypervisor, the OS, the serving software, and the GPUs. For most workloads this is fine. For some it isn’t.

Workloads That Can’t Trust

  • Healthcare (HIPAA)
  • Finance (PCI, SOX)
  • Legal (attorney-client privilege)
  • Defense / classified
  • Trade secrets / IP

What Confidential Computing Promises

  • Cryptographic confidentiality — not contractual
  • No one can read the data — not even the provider’s root user
  • Verifiable via remote attestation
  • Use frontier models on sensitive data

TEE

Trusted Execution Environments

Hardware Guarantees

  • Code outside cannot read TEE memory
  • TEE memory is encrypted in RAM
  • Code inside cannot be modified by outside
  • CPU cryptographically attests to running code

Why Hardware Matters

  • OS, hypervisor, container runtime can all be bypassed by sufficient privilege
  • TEE is designed so even root can’t peek
  • Physical attacks limited too — memory is encrypted

Implementations

  • Intel SGX — original, app-level, side-channel issues
  • Intel TDX — VM-level, dominant for AI
  • AMD SEV-SNP — AMD’s VM-level equivalent
  • ARM CCA — ARMv9 server-class

ATTEST

Remote Attestation

The Protocol

  1. User opens TLS to TEE
  2. TEE generates signed measurement of running code + fresh public key
  3. User verifies measurement against expected values
  4. User verifies signature with vendor PKI
  5. User establishes session key with verified TEE
  6. User transmits sensitive data over encrypted session

What You Verify

  • Right code is running
  • Genuine hardware (not simulator)
  • Both performed every connection

Trust Anchors

  • CPU vendor root certificate
  • Vendor attestation service
  • Public measurement registry
  • Optional third-party verifier

GPU

The GPU Problem

Why CPU TEEs Aren’t Enough

  • AI inference happens on the GPU
  • Model weights, KV cache, activations live in GPU memory
  • Naive deployment exposes everything to GPU
  • Encryption stops at PCIe bus
  • Fatal flaw for confidential AI

NVIDIA Confidential Computing

  • First on Hopper H100 (2023)
  • Encrypted PCIe transfers
  • Encrypted HBM memory
  • GPU-side attestation
  • Secure firmware boot
  • Extends TEE boundary to include GPU

DEPLOY

The Trust Model

You Trust

  • CPU and GPU vendors (Intel/AMD/NVIDIA)
  • Vendor attestation infrastructure
  • The code you approved to run inside the TEE
  • The OS, boot loader, inference server inside the TEE

You Don’t Trust

  • Cloud provider employees
  • Hypervisor or host OS
  • Anyone with physical datacenter access
  • Other tenants on the same hardware
  • Network operators between you and the TEE

Significance

The trusted computing base shrinks dramatically. Instead of trusting the entire cloud provider’s organizational security, you trust a small set of hardware vendors and a verified software image.

LIMITS

Side Channels & Vendor Trust

Side-Channel Attacks

  • SGX has long history of vulnerabilities (Foreshadow, ZombieLoad)
  • TDX/SEV-SNP newer, less studied
  • GPU-side surface poorly understood
  • Slow, expensive, not scalable for most attackers

Vendor Trust

  • Must trust silicon vendor implementation
  • No independent verification of silicon
  • Compromised vendor → broken story
  • Mitigation: choose carefully, monitor disclosures

Code Trustworthiness

Attestation verifies the right code is running. The user must still trust that the code is correctly implemented and benign. A buggy inference server inside a TEE that logs prompts to a file is just as compromising as a non-confidential one. Keep TEE-resident code small and auditable.

SUMMARY

Bridge Between Power and Privacy

Why It Matters

  • Frontier LLMs are too large for on-prem deployment
  • Without confidential computing, sensitive workloads were excluded entirely
  • With it, regulated data can use cloud frontier models with cryptographic guarantees
  • Production-ready in 2025, with mature commercial offerings

Practitioner’s Lesson

Most LLM use does not require confidential computing — the trust model of standard cloud APIs is acceptable. But for healthcare, finance, legal, defense, and other regulated workloads, it is increasingly the answer. The tooling is good enough to deploy, the compliance story is improving, and the performance overhead has dropped to a few percent. Worth knowing about even if you don’t need it today.
