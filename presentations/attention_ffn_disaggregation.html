CONTEXT

The Core Tension

One Chip, Two Workloads

  • Every transformer layer has two fundamentally different stages: attention and FFN
  • They have opposite hardware needs during decode
  • Running both on the same GPU forces a compromise neither side wins

The AFD Insight

  • Split the decode loop across two kinds of hardware
  • GPUs handle attention (stateful, memory-bound)
  • SRAM-first processors handle FFN/MoE (stateless, weight-heavy)
  • Intermediate activations shuttle between them every token

Why Now?

MoE models have gone mainstream, context lengths are growing to 128K+, and agentic workloads make per-token latency compound. The attention/FFN asymmetry is no longer an edge case — it is the default computational profile of frontier models.

ARCHITECTURE

Inside a Transformer Layer

A decoder-only transformer (GPT, LLaMA, DeepSeek) is a stack of identical layers. Each layer has two main stages:

Attention: ~1/3 of Parameters

  • Q/K/V projections, attention scores, output projection
  • Each token attends to every previous token
  • KV cache grows with context length, is per-request

FFN: ~2/3 of Parameters

  • Two or three large matrix multiplications + activation
  • Position-wise: processes each token independently
  • Stateless — no memory of previous tokens

DETAIL

Attention During Decode

  • For each new token, compute Q, K, V projections
  • Store new K, V in the KV cache
  • Dot-product query against all S cached keys
  • Softmax → weighted sum of cached values
  • Multi-head attention (MHA), GQA, or MLA variants

The Critical Detail

During decode, each new token must reference the KV cache of every previous token. The cache grows with context length and is per-request — it cannot be shared across users.

Decode Characteristics

  • Stateful — depends on full history
  • Memory footprint grows with context
  • Low arithmetic intensity (~1–2 op/B)
  • GPU tensor cores sit mostly idle
  • Needs: large HBM, high bandwidth

KV Cache Size (FP8)

2 × S × d × heads × layers

At 128K context on DeepSeek-V3: hundreds of GB per request

DETAIL

The Feed-Forward Network

Structure

FFN(x) = W₂ · activation(W₁ · x)

Modern models use SwiGLU (adds a third weight matrix + gating). Hidden dimension typically 4× model dim (~8/3× with SwiGLU).

Key Properties

  • Position-wise: each token processed independently
  • Stateless: no memory of previous tokens
  • Same fixed function applied to every input
  • Contains ~2/3 of all layer parameters

Decode Characteristics

  • Weights are same for every token in batch
  • Load weights once, reuse across batch
  • Arithmetic intensity: ~2B op/B (scales with batch)
  • At batch=1: memory-bound
  • At batch=100+: compute-bound

The Key Asymmetry

FFN arithmetic intensity is tuneable via batching. Attention arithmetic intensity is not — because the KV cache is per-request.

THEORY

Arithmetic Intensity & the Roofline Model

Arithmetic Intensity

Ratio of FLOPs to bytes moved from memory, measured in op/B. Captures how much useful work per byte of data fetched.

  • Low intensity → memory-bandwidth bound
  • High intensity → compute bound

Ridge Point

The crossover where compute and bandwidth ceilings meet.

  • H100: ~2000 TFLOPS FP8, ~3.35 TB/s HBM
  • Ridge point: ~600 op/B
  • Anything below 600 op/B is memory bound on H100

Why This Matters

Attention during decode has intensity of ~1–2 op/B — roughly 300× below the ridge point. The GPU's tensor cores are almost entirely idle during attention decode. The FFN can reach the ridge point with sufficient batching, but attention never can.

ANALYSIS

Attention Is Memory-Bound During Decode

The Problem

  • Must load entire KV cache from HBM per token
  • S tokens → load 2 × S × d bytes per head per layer
  • FLOPs: O(S × d), bytes: also O(S × d)
  • Arithmetic intensity: ~1–2 op/B
  • Ridge point on H100: ~600 op/B

Can Batching Help?

  • Batching shares weight loads (Q/K/V projections)
  • But KV caches are per-request — cannot be shared
  • As context grows, KV cache dominates traffic
  • At 128K context: batching cannot rescue attention

What Attention Needs

Large, high-bandwidth memory systems — exactly what GPUs with massive HBM pools provide. The GPU is the right hardware for attention, even though its compute units are underutilised during this phase.

attention-ffn-disaggregation / 07
ANALYSIS

FFN Is Potentially Compute-Bound

How Batching Helps FFN

  • All tokens share the same weight matrices
  • Weights loaded once, reused across batch
  • Intensity: ~2B op/B (B = batch size)
  • At B=1: intensity ~2 (memory-bound)
  • At B=300: intensity ~600 (compute-bound)

The Practical Challenge

  • Achieving large batch sizes during decode is hard
  • Latency constraints limit batching
  • KV cache memory limits concurrent requests
  • Sequential generation keeps per-user batch = 1

The Structural Insight

FFN arithmetic intensity is tuneable via batching; attention intensity is fixed regardless of batch. This asymmetry is the seed from which AFD grows — the two operations live on different sides of the roofline.

MOE

Mixture of Experts Amplifies the Split

What MoE Replaces

  • MoE applies exclusively to the FFN
  • Attention is completely untouched
  • Single FFN → multiple expert FFNs + router
  • MoE is a technique for making FFN sparse

Why MoE Exists

  • Model quality scales with total parameters
  • Inference cost scales with activated parameters
  • 671B total params, only 37B active per token
  • Knowledge of a huge model, cost of a small one

The Catch

All 671B parameters must be resident in memory, even though only 5.5% are active per token. Different tokens route to different experts, and the router's decisions are not known in advance. The FFN side becomes massively weight-heavy while attention remains unchanged.

MOE

How MoE Works: DeepSeek-V3

DeepSeek-V3 MoE Configuration

  • 671B total params, 37B active per token
  • 61 layers (first 3 use dense FFNs)
  • 256 routed experts per MoE layer
  • 1 shared expert (always active)
  • Router assigns each token to top-8 experts
  • Per token: 8 routed + 1 shared = 9 small FFNs
  • Remaining 247 experts: idle for that token

MOE

MoE: Memory & Intensity Problems

The Memory Problem

  • Dense 37B: ~37 GB in FP8
  • DeepSeek-V3 (671B): ~671 GB in FP8
  • Only 5.5% active per token, but 100% must be in memory
  • Different tokens route to different experts
  • Cannot discard unused experts in advance

The Intensity Problem

  • Dense FFN: all tokens share same weights → good reuse
  • MoE: tokens scattered across 256 experts
  • Each expert sees only B/32 tokens on average
  • Dense model: compute-bound at B ≈ 120
  • MoE: needs B ≈ 3,840 for same intensity

Expert Parallelism: All-to-All Communication

Experts distributed across many GPUs. Each token must be routed to its assigned experts on different GPUs — an all-to-all dispatch/combine pattern that is inherently cross-node. This is structurally different from attention's local KV cache access and creates a natural seam in the computation.

FRAMEWORK

Three Levels of Disaggregation

Level 1: Prefill–Decode

  • Separate prompt processing from token generation
  • Prefill: compute-bound (parallel tokens)
  • Decode: memory-bound (sequential tokens)
  • Now standard practice (DistServe, etc.)
  • Different hardware pools, independently scaled

Level 2: Attention–FFN (AFD)

  • Split within the decode phase
  • Attention → GPU (stateful, KV cache)
  • FFN/MoE → SRAM accelerator (stateless)
  • Exchange activations every token, every layer
  • The focus of this presentation

Level 3: Experts-as-a-Service

  • Fully decouple the MoE expert pool
  • Experts as stateless microservices
  • Independent scaling & load balancing
  • Finer-grained resource allocation
  • Significant orchestration complexity

Progressive Specialisation

Each level recognises a deeper structural boundary in the transformer. Level 1 separates phases. Level 2 separates operations within a phase. Level 3 separates individual experts within an operation.

HARDWARE

NVIDIA Vera Rubin + Groq 3 LPX

GPU Side: Vera Rubin NVL72

  • 72 GPUs per rack, NVLink interconnect
  • Handles prefill (compute-heavy, parallel)
  • Handles decode attention (KV cache in HBM)
  • Large HBM pools, high bandwidth
  • Does not run FFN/MoE during AFD decode

LPU Side: Groq 3 LPX Rack

  • 256 LPU accelerators per rack
  • 500 MB SRAM per chip, 128 GB at rack scale
  • 40 PB/s on-chip bandwidth (~10× GPU HBM)
  • Deterministic, compiler-orchestrated execution
  • 2.5 TB/s chip-to-chip interconnect per chip
  • Handles FFN/MoE during decode

Why This Split Works

FFN/MoE is stateless (no KV cache needed) and bandwidth-dominated at low batch. The LPU's SRAM delivers vastly more bandwidth per FLOP than GPU HBM. Expert weights pre-loaded into SRAM and reused across all requests. Meanwhile, attention stays on GPU because its KV cache needs large, flexible HBM.

MECHANISM

The AFD Decode Loop

Per-Layer Ping-Pong

This loop repeats for every layer (61 in DeepSeek-V3), for every token generated. The intermediate activation is small, keeping communication cost manageable.

NVIDIA Dynamo Orchestration

Request routing, activation transfer, KV-aware scheduling, and tail latency control. Classifies requests by latency sensitivity for optimal serving path.

SYNERGY

Speculative Decoding with AFD

LPX as Draft Engine

  • Draft model is small enough to fit in LPX SRAM
  • Deterministic, low-latency execution → fast candidate generation
  • Exactly the property you want from a draft engine

GPU as Verifier

  • Larger memory and higher compute throughput
  • Verifies candidates in parallel (prefill-like)
  • Efficient at the compute-heavy verification step

Clean Division

LPX drafts, GPU verifies. Both processors operate in their respective sweet spots — the LPU's low-latency SRAM for fast sequential generation, the GPU's parallel compute for bulk verification.

TRENDS

Why Now? Five Converging Forces

1. MoE Is Mainstream

DeepSeek-V3/R1, Qwen3, LLaMA 4 Maverick — all MoE. The attention/FFN asymmetry is now the default profile of frontier models.

2. Context Lengths Growing

128K+ tokens. KV cache grows proportionally. Attention becomes more memory-bound and the gap with FFN widens.

3. Agentic Workloads

Coding assistants chain 50+ inference calls. Per-token latency compounds 50×. AFD's latency advantage multiplies.

4. Prefix Caching

Shared prompts computed once and cached. Prefill gets cheaper → larger fraction of compute is in decode → bigger payoff from optimising decode.

5. The Economics

NVIDIA claims Vera Rubin + LPX delivers up to 35× higher tokens/s/MW at 400 tok/s per user vs. GB200 NVL72. Specialised hardware = massive efficiency gains.

CHALLENGES

Challenges & Open Questions

Communication Overhead

  • Every layer of every token: GPU ↔ LPU round trip
  • Activation tensor is small, but latency adds up across 60+ layers
  • Network jitter directly impacts token latency
  • High-bandwidth, low-latency interconnects are essential

Dense Models Are Harder

  • AFD works best with MoE models
  • MoE already has all-to-all dispatch to piggyback on
  • Dense FFN has no natural scatter/gather
  • Communication overhead is proportionally larger
  • AFD for dense models remains an open challenge

Load Balancing

  • Attention latency varies with context length
  • FFN latency is more stable
  • Mismatch → pipeline bubbles (one side idles)
  • Optimal ratio depends on model, context, batch — and changes dynamically

Scaling & Maturity

  • Bipartite topology (M attn ↔ N FFN) is more complex
  • Scheduling, routing, fault tolerance all harder
  • Production systems emerging: StepFun, ByteDance, NVIDIA LPX
  • vLLM actively implementing AFD support

SUMMARY

Key Takeaways

The Core Insight

  • The transformer is not a monolithic computation
  • Attention: stateful, memory-bound, tied to per-request KV cache
  • FFN/MoE: stateless, weight-heavy, benefits from different memory hierarchy
  • MoE amplifies this divergence to the point where disaggregation becomes necessary

The Architecture

  • GPUs handle attention (large HBM for KV cache)
  • SRAM-first processors handle FFN/MoE (bandwidth for weights)
  • Small activation tensors shuttle between them every layer
  • Vera Rubin + LPX: first major commercial AFD system

The Bottom Line

The decode loop is a relay race, not a solo sprint. Matching each half of the transformer to the hardware that serves it best is not a quirk of one vendor's strategy — it is a structural response to the transformer's own internal asymmetry. AFD will shape inference infrastructure for years to come.
