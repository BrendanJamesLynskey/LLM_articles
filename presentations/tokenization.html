INTRO

Why Tokenization Matters

Tokenization is the first and most foundational step in the LLM pipeline. Before any neural network computation occurs, raw text must be converted into discrete tokens from a fixed vocabulary.

What It Affects

  • Effective context length in characters
  • Multilingual capability
  • Arithmetic and reasoning ability
  • Code generation quality

Key Insight

Despite its fundamental importance, tokenization often receives less attention than model architecture or training methodology — yet its design decisions propagate through every aspect of model behavior.

ALGORITHM

Byte Pair Encoding (BPE)

The most widely used tokenization algorithm in modern LLMs.

  1. Start with a base vocabulary of individual characters or bytes
  2. Find the most frequent adjacent pair of tokens
  3. Merge that pair into a new token
  4. Repeat until desired vocabulary size is reached

Encoding is deterministic — merges are applied in strict priority order.

Models Using BPE

  • GPT-2, GPT-3, GPT-4
  • Llama family
  • Mistral / Mixtral

Properties

  • Deterministic encoding
  • Fast at inference time
  • Good balance of efficiency and coverage

ALGORITHM

SentencePiece & Unigram

SentencePiece Framework

  • Language-independent — treats input as raw byte stream
  • No language-specific pre-processing or whitespace handling needed
  • Supports both BPE and Unigram algorithms
  • Used by Llama, Gemma, and many multilingual models

Unigram Algorithm

  • Starts with a large candidate vocabulary
  • Iteratively prunes tokens that contribute least to corpus likelihood
  • Uses a unigram language model for evaluation
  • Can produce different segmentations for the same input
  • Enables subword regularization for data augmentation

BPE vs. Unigram: Direction of Construction

BPE builds bottom-up (merging frequent pairs)  |  Unigram prunes top-down (removing low-value tokens)

ALGORITHM

Byte-Level Tokenization

Operates on raw bytes (0–255) rather than Unicode characters, guaranteeing any input can be encoded without unknown tokens.

Advantages

  • Universal coverage — handles any language, code, or binary content
  • No <UNK> tokens needed
  • Introduced by GPT-2 as byte-level BPE

Trade-off

Non-ASCII characters (CJK languages) require multiple bytes per character, reducing effective context length for these languages.

ALGORITHM

Byte Fallback & Hybrid Approaches

Models Using Hybrid

  • Llama 3 — BPE + byte fallback
  • Gemma — BPE + byte fallback

Best of Both Worlds

Common text tokenized efficiently as subwords; rare or unusual sequences fall back to individual byte tokens for universal coverage.

DESIGN

Vocabulary Size Trade-offs

Larger Vocabularies (100K–200K)

  • More compact text encoding
  • Fewer tokens per document
  • Effectively extends context window
  • Larger embedding matrices
  • Increased memory footprint
  • May dilute training signal for rare tokens

Smaller Vocabularies (32K–50K)

  • More memory-efficient
  • Smaller embedding tables
  • Longer token sequences
  • Reduced effective context length
  • Better training signal per token
  • Simpler to manage

Industry Trend

Modern models have converged on 32K to 128K vocabulary sizes, with a trend toward larger vocabularies as model sizes increase.

DESIGN

Training Data Composition

The corpus used to train the tokenizer determines which tokens are created and how efficiently different text types are encoded.

English-Heavy Corpus

  • Efficient English tokenization
  • Verbose for other languages
  • More tokens per word in non-English
  • Wastes context window

Multilingual Corpus

  • Balanced token efficiency
  • Equitable cross-language support
  • e.g., Llama 3 vs Llama 2
  • Significant efficiency gains

Code-Inclusive Corpus

  • Tokens for indentation patterns
  • Programming keywords merged
  • Common API patterns captured
  • Fewer tokens per line of code

Special Tokens

Special tokens serve as control signals that structure the model's input and behavior.

Token TypePurpose
<BOS>Beginning of sequence marker
<EOS>End of sequence / generation stop
<SEP>Separator for multi-segment inputs
<TOOL>Tool-use and function-calling delimiters
<|system|>Role markers in chat formats

Pitfall

Poor special token design can cause:

  • Failure to recognize conversation boundaries
  • Premature generation termination
  • Broken tool integration
  • Subtle, hard-to-debug errors in multi-turn chat

IMPACT

Multilingual Performance

The Tokenization Inequality

Underrepresented languages may require 3–10× more tokens for the same semantic content, consuming proportionally more context window and computation.

The Fix

Balanced tokenizer training is one of the most impactful steps toward equitable multilingual models. Llama 3's tokenizer (128K vocab, multilingual corpus) dramatically improved non-English efficiency over Llama 2 (32K vocab).

IMPACT

Arithmetic & Numerical Reasoning

How numbers are tokenized significantly affects a model's arithmetic ability.

Problematic: Arbitrary Splits

"12345" → ["123", "45"]

Model must learn digit alignment across token boundaries — much harder than single-digit operations.

Better: Individual Digits

"12345" → ["1", "2", "3", "4", "5"]

Simpler arithmetic but longer sequences for large numbers.

Research Finding

Tokenizer design accounts for a meaningful fraction of the variance in LLM arithmetic performance. The way numbers are split into tokens determines whether the model can learn positional digit alignment.

Design Options

  • Single digits — simple arithmetic, long sequences
  • Common patterns — tokens for frequent numbers
  • Arbitrary chunks — misaligned digit boundaries

IMPACT

Code Generation

Code has fundamentally different statistical properties than natural language:

Code Properties

  • Significant whitespace
  • Frequent special characters
  • Highly repetitive patterns
  • Exact character sensitivity

Code-Aware Tokens

  • Indentation levels ( )
  • Keywords (def, class)
  • Common API calls
  • Syntax patterns (():)

Dual Benefit

Efficiency: Fewer tokens per line of code, leaving more context window for surrounding code.

Quality: Model reasons about code in larger, semantically meaningful chunks rather than character-by-character.

PITFALLS

Tokenization Edge Cases

Token Boundary Effects

Models behave differently depending on how a word is segmented, leading to inconsistencies in string processing tasks.

Prompt Injection via Tokens

Attacks exploit unusual Unicode characters or whitespace that tokenizes differently than expected.

Invisible Characters

Trailing whitespace, zero-width characters, and Unicode normalization differences can change tokenization in surprising ways.

Token Counting Errors

Context window management must use the actual tokenizer — word count approximations fail because token-to-word ratios vary across languages and content types.

FUTURE

Future Directions

Token-Free Models

Operate directly on bytes or characters, eliminating tokenization artifacts entirely. Challenges remain with sequence length and computational efficiency.

Dynamic Vocabularies

Vocabularies that adapt to input domain could improve efficiency for specialized applications without the fixed-vocabulary constraint.

Megabyte Architecture

Processes bytes at a coarse level then refines at a fine level — a middle ground between byte-level and subword approaches.

Better Multilingual Tokenization

Balanced training and larger vocabularies continue to reduce cross-language performance disparities — one of the most impactful areas of active research.

CONCLUSION

Key Takeaways

Deceptively Important

Tokenization design decisions propagate through every aspect of model behavior — from multilingual capability to arithmetic to code generation.

Critical Trade-offs

  • Vocabulary size vs. embedding cost
  • Compression efficiency vs. coverage
  • Language balance vs. dominant-language performance
  • Subword vs. byte-level granularity

Design Decisions That Matter

  • Algorithm: BPE, Unigram, byte-level, or hybrid
  • Vocab size: 32K to 256K range
  • Training data: Language and domain balance
  • Special tokens: Control signal design

Looking Ahead

The field is moving toward larger vocabularies, better multilingual coverage, and potentially tokenizer-free architectures.

The often-overlooked tokenizer deserves careful attention proportional to its outsized impact on model capability.
