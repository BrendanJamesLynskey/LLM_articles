CONTEXT

Why Conventional APM Falls Short

Standard service monitoring tracks request rate, error rate, latency, resource utilization. None of those tell you whether the LLM’s output was correct, helpful, safe, or hallucinated.

Failure Modes APM Misses

  • 200 OK with confidently-stated falsehood
  • Helpful response with wrong tone
  • Unsolicited PII leak from prompt
  • Successful jailbreak by adversarial input
  • Refusal of legitimate request
  • Wrong tool called or skipped

What LLM Observability Adds

  • Trace: full prompt, response, tools, params
  • Evaluation: quality scoring of outputs
  • Drift detection: behavior change over time
  • Cost attribution: who’s burning tokens
  • Prompt versioning: rollback on regression

TRACE

The LLM Trace

Captured Per Trace

  • System prompt + user message + history
  • Model identifier, parameters, sampling config
  • Tool calls (args + results)
  • Internal reasoning (if exposed)
  • Final response shown to user
  • Latency breakdown (prefill, decode, tools)
  • Token counts: input, output, cached, total
  • Cost in dollars
  • User feedback signals

OTEL

OpenTelemetry GenAI Conventions

Standard Attributes

  • gen_ai.system
  • gen_ai.request.model
  • gen_ai.usage.input_tokens
  • gen_ai.usage.output_tokens
  • gen_ai.response.finish_reason
  • gen_ai.prompt / gen_ai.completion

Why It Matters

  • Vendor-neutral data model
  • Switch backends without re-instrumentation
  • Multiple platforms accept OTel traces
  • Integrates with non-LLM service traces

Sampling Strategies

  • Head-based: decide at trace start
  • Tail-based: decide based on outcome
  • Adaptive: adjust by traffic level
  • Common: 100% errors/slow, 10% normal, 100% feedback-flagged

Storage

  • Trace volumes can hit GB/day
  • Prompts/responses dominate size
  • Compression + sampling essential
  • Retention typically 30–90 days

EVAL

Continuous Evaluation

Tests run in CI before deployment. Evaluation runs continuously in production — sampling traffic, scoring outputs, dashboarding quality.

Heuristic Checks

  • Format compliance (JSON parses?)
  • Required fields present
  • Forbidden content absent
  • Length within range
  • Citations present
  • Cheap, deterministic, free

LLM-as-Judge

  • Stronger LLM scores weaker LLM
  • Rubric-driven scoring
  • Reasoning + score
  • Calibrate vs human ratings
  • Most scalable approach

Human Annotation

  • Gold standard for highest stakes
  • Implicit feedback (clicks, retries)
  • Explicit thumbs/ratings
  • Annotation queues for high-priority traces

DRIFT

Drift Detection

Why LLMs Drift

  • Provider model updates (silent)
  • Prompt template changes
  • Input distribution shift over time
  • Context bloat in long conversations
  • Tool ecosystem changes

Detection Approaches

  • Threshold alerts on quality metrics
  • CUSUM and Bayesian change-point detection
  • Distributional comparison (length, topics, formats)
  • Per-user-segment monitoring

What Doesn’t Drift

Latency, error rate, throughput — the things APM tracks. The thing that drifts is quality, and you only see that if you’re measuring it.

PROMPT

Prompts & Versions as First-Class Artifacts

Prompt Registry

  • Every version stored with timestamps
  • Diff history for each change
  • Each LLM call attributed to prompt version
  • One-click rollback
  • A/B test variants in production

Model Pinning

  • Pin to specific version (e.g. claude-opus-4-6)
  • Test before upgrading to new versions
  • Compare metrics across versions during eval
  • Most teams pin and explicitly upgrade

Why It Matters

Prompt changes are the most common cause of LLM application regressions. Being able to attribute a quality drop to a specific prompt change is essential for fast diagnosis. Treating prompts as code-equivalent artifacts is non-negotiable in mature deployments.

COST

Cost Tracking & Privacy

Cost Observability

  • LLM API costs can dwarf other infra
  • Per-feature attribution
  • Per-user attribution
  • Trend analysis (linear vs runaway)
  • Outlier detection: 100× normal requests

Privacy Concerns

  • Traces capture every conversation — PII goldmine
  • Automatic redaction (names, emails, cards)
  • Tokenization for regulated data
  • RBAC + audit logging
  • Geographic data residency
  • Retention policies

SUMMARY

Practitioner’s Lesson

What Works

  • Start instrumenting from day one
  • OpenTelemetry as the data backbone
  • Heuristic checks first, LLM-as-judge second, humans last
  • Treat prompts as versioned code artifacts
  • Monitor quality metrics like you monitor latency
  • Track cost at the per-feature granularity

The Pitfall

Retrofitting observability onto an LLM application that has been running blind for months is much more expensive than building it in from the start. The organizations that operate LLM apps with rigor will out-compete the ones perpetually surprised by silent quality regressions and angry users.
