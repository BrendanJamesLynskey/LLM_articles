CONTEXT

Why ML Needs Its Own Compilers

The Information Gap

  • LLVM doesn’t know what a matmul is
  • Tensor shapes available statically — useful for optimization
  • Hardware support for whole tensor ops (tensor cores)
  • Need IRs that preserve high-level semantics

The Lowering Hierarchy

  • High level: matmul, conv, softmax
  • Mid level: loops, vectorization
  • Low level: PTX, AMDGPU IR, vector intrinsics
  • Optimize at each level where info is available

XLA

XLA: TPU’s Compiler

What XLA Does

  • Operator fusion (the big win)
  • Memory layout selection
  • Buffer assignment / reuse
  • Algebraic simplification
  • Constant folding
  • Tensor core dispatch

HLO IR

XLA’s internal representation. High-level tensor operations with shapes. Most optimizations operate on HLO.

The TPU Connection

  • Only practical path to TPU
  • TPU hardware doesn’t do CUDA-style programming
  • Tight coupling = high efficiency, less flexibility
  • Strategic for Google’s AI strategy

On NVIDIA

Often 10–30% slower than the best alternatives, but improving. Provides unified compiler across backends.

ml-compilers / 03
MLIR

MLIR: Compiler Infrastructure

Not a Compiler — a Framework

  • Built by Chris Lattner (LLVM creator) at Google
  • Now part of LLVM
  • Define IRs at multiple abstraction levels
  • Mix levels in one program
  • Progressive lowering

Dialects

  • Self-contained ops + types per abstraction level
  • linalg dialect (tensor ops)
  • affine dialect (loops)
  • vector, gpu, llvm, spirv dialects
  • Compose freely

Why It’s Foundational

XLA, IREE, Triton, torch.compile, TF-XLA bridge — all use MLIR underneath. Solves the “everyone has their own IR” problem of the previous generation.

STABLEHLO

StableHLO: The Interchange Format

The Need

  • HLO is unstable across XLA versions
  • Frameworks need a stable target
  • Want one frontend → many backends

Adoption

  • PyTorch supports as export target
  • TensorFlow supports natively
  • JAX supports as serialization format
  • ONNX has signaled cooperation
  • The de facto interchange format of 2026
ml-compilers / 05
IREE

IREE: The Universal Deployment Path

What IREE Is

  • Open-source ML compiler + runtime
  • Targets CPUs, GPUs (CUDA, Vulkan, Metal), mobile NPUs
  • MLIR-based core
  • Frontend support: TF, PyTorch (torch-mlir), JAX

The Promise

Write the model once, deploy anywhere with reasonable performance.

Where It’s Strong

  • Mobile and embedded deployment
  • Apple Silicon, AMD GPUs, Vulkan mobile GPUs
  • Recently pushing on transformer inference

Where It’s Weaker

  • Server NVIDIA — competes against TensorRT-LLM
  • Vendor-specific tooling still wins for max perf

TVM

TVM: Auto-Scheduling Pioneer

The Distinguishing Feature

  • Auto-scheduler searches kernel implementation space
  • Empirical measurement picks the best
  • Slow compile, competitive output
  • First major project to demonstrate the approach

Why It Hasn’t Won

  • Slow compilation hurts iteration
  • Tunings don’t always port across hardware generations
  • Multiple competing internal frontends
  • Overshadowed by vertically-integrated stacks

TORCH.COMPILE

The User-Facing Compiler

What It Does

  • Traces user’s Python code to a graph
  • Compiles via Inductor (PyTorch’s backend)
  • Generates Triton kernels for most ops
  • Falls back to cuBLAS/cuDNN for big GEMMs
  • Uses MLIR underneath for some paths

The Result

  • 1.5–2× speedup typical for transformers
  • Single-line decorator
  • Most users never know about XLA or MLIR
@torch.compile
def model_step(x):
    return model(x)

# fusion, codegen, dispatch.

CONVERGE

The 2026 Picture

Convergence Patterns

  • MLIR as the underlying framework everywhere
  • Triton as the kernel language
  • StableHLO as interchange format
  • Per-framework JITs as user surface
  • Vendor compilers for max perf

Pragmatic Stack

  • torch.compile for development
  • TensorRT-LLM for NVIDIA prod
  • OpenVINO for CPU/Intel fallback
  • MLIR custom passes for research

Practitioner’s Lesson

For training: use your framework’s JIT (torch.compile, jax.jit). For NVIDIA prod inference: TensorRT-LLM. For portable inference: IREE or ONNX RT. For custom kernels or research: MLIR + Triton. The choice of compiler stack is as consequential as the choice of model architecture.
