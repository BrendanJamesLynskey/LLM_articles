CONTEXT

Why Function Calling Matters

Without tool use, an LLM is a reasoning engine with no effectors. Function calling is the primitive that makes agents possible.

The Four-Step Protocol

  • 1. Define — host provides tool schemas (name, description, JSON Schema params)
  • 2. Call — model generates a structured tool call instead of text
  • 3. Execute — host validates, runs the function, gets a result
  • 4. Incorporate — result sent back to model, which continues generating

The Model's Decisions

  • Whether to use a tool (or answer from knowledge)
  • Which tool to use (matching intent to descriptions)
  • What arguments to provide (extracting from context)
  • Sequential or parallel (independent calls run concurrently)

Tool Definitions and Message Flow

Tool Definition (JSON Schema)

{"name": "get_weather",
 "description": "Get current weather",
 "parameters": {
   "type": "object",
   "properties": {
     "location": {"type": "string",
       "description": "City, country"},
     "unit": {"type": "string",
       "enum": ["celsius","fahrenheit"]}
   },
   "required": ["location"]}}

Message Flow

User: "Weather in Tokyo?"
Model → tool_call: get_weather(
  location="Tokyo, Japan")
Host executes → {"temp": 22,
  "condition": "partly cloudy"}
Model → "It's 22°C and partly
  cloudy in Tokyo."

The cycle repeats: model can call another tool based on results before producing a final response.

FEATURE

Parallel Tool Calls and Streaming

Parallel Calls

  • Model generates multiple tool calls in one response
  • Host executes concurrently — latency = max(calls) not sum
  • Each call has unique ID for result matching
  • Only works for independent calls; dependent calls must be sequential
  • Host must handle partial failures

Streaming Tool Use

  • Stop-and-resume: stream text, pause for tool exec, resume
  • Interleaved SSE: text chunks, tool_call deltas, tool results intermixed
  • Buffered JSON: partial tool call JSON accumulated until valid
  • Event types: content_block_start, input_json_delta, content_block_stop

PROVIDERS

Provider Implementations

OpenAI

  • Strict mode: constrained decoding guarantees JSON schema conformance
  • Built-in tools: web search, code interpreter, file search
  • Responses API (2025): server-side tool execution
  • Parallel calls + streaming via SSE

Anthropic (Claude)

  • Content blocks: text + tool_use intermixed
  • Extended thinking: visible reasoning about tool choice
  • Prompt caching for tool definitions
  • Computer use as specialised tool type

Google (Gemini)

  • Grounding with Google Search (built-in)
  • Built-in code execution sandbox
  • Multi-modal tool use (images, video)

Open-Weight Models

  • Llama 3.1+, Mistral, Qwen2.5: special token formats
  • Less reliable format adherence
  • Community fine-tunes improve quality

TRAINING

Training Models for Tool Use

Supervised Fine-Tuning

  • Manual annotation (high quality, expensive)
  • Model-generated data + execution-based filtering
  • Gorilla: fine-tuned on API docs, retrieval-augmented
  • ToolBench: 16K+ real APIs, multi-step trajectories

RL and Constrained Decoding

  • Outcome-based RL: reward correct final answers
  • Process-based RL: reward each tool call individually
  • Constrained decoding: mask invalid tokens at each step
  • Guarantees valid JSON — eliminates formatting errors entirely

MCP

MCP and Tool Integration

The N×M Problem

  • Before MCP: N apps × M tools = N×M custom integrations
  • MCP: universal protocol — build tool server once, use everywhere
  • JSON-RPC 2.0 over stdio or Streamable HTTP
  • Dynamic discovery via tools/list

MCP vs Native Function Calling

  • MCP sits above native function calling
  • Native FC: how model requests invocations
  • MCP: how host discovers and executes tools
  • MCP tools presented via native FC interface

PATTERNS

Multi-Step Chains and Error Handling

Multi-Step Chains

search_issues() → Issue #42
get_issue_details(42) → description
read_file("src/auth.py") → code
edit_file("src/auth.py") → fix
run_tests() → all pass
create_pr("Fix #42") → PR #87

Each step depends on previous results.

Common Failures & Mitigations

  • Premature stop → explicit "continue until done" instructions
  • Infinite loops → max iterations / dedup detection
  • Hallucinated tools → constrained decoding on tool names
  • Argument drift → structured result formats
  • Always: send errors back as tool results for self-correction

DESIGN

Schema Design Best Practices

Do

  • Descriptions: explain when to use, not just what
  • Parameter descriptions with format examples
  • Minimal schemas — fewer params = fewer errors
  • Enums over free text where possible
  • Separate read from write tools

Don't

  • Vague descriptions ("does stuff with data")
  • Overlapping tools with similar purposes
  • 50+ tools without dynamic loading
  • Complex deeply-nested schemas
  • Giant tool results (100K+ tokens)

SECURITY

Security Considerations

Injection via Args

  • SQL injection through natural language
  • Command injection via shell tools
  • Path traversal in file ops
  • Validate and sanitise all arguments

Confused Deputy

  • Indirect prompt injection hijacks tool use
  • Model uses authorised access for unauthorised purposes
  • Human-in-the-loop for sensitive actions

Data Exfiltration

  • Sensitive data encoded in tool args or URLs
  • Markdown image rendering trick
  • Output scanning and URL allowlists

SUMMARY

Key Takeaways

Core Concepts

  • Function calling is the primitive that makes agents possible
  • Four-step loop: define → call → execute → incorporate
  • Parallel calls reduce latency; streaming maintains responsiveness
  • Constrained decoding eliminates JSON formatting errors
  • MCP standardises tool discovery across providers

Practical Advice

  • Start with few, well-defined tools; add incrementally
  • Write descriptions explaining when to use each tool
  • Validate all arguments; sanitise for injection
  • Send errors back as tool results for self-correction
  • Monitor tool call accuracy and iterate on schemas
