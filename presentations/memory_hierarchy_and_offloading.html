The Core Challenge

When model weights, KV-cache, and activations collectively exceed GPU memory — even after quantization and multi-GPU distribution — the system must leverage a deeper memory hierarchy.

  • Modern servers offer a cascade of storage tiers
  • Each tier trades bandwidth for capacity
  • Strategic data placement enables inference of otherwise impossible models
  • Cost: increased latency unless carefully managed

Three Data Categories

  • Weights — static, read-only per pass
  • KV-Cache — grows with context length
  • Activations — transient, per-layer

Three Memory Tiers

  • GPU HBM/GDDR — 1–3 TB/s
  • CPU DRAM — 50–200 GB/s
  • NVMe SSD — 3–7 GB/s

GPU Memory (HBM / GDDR)

Consumer GPUs

  • 8–24 GB GDDR6/GDDR6X
  • Bandwidth: 500–900 GB/s
  • Cost-effective but memory-limited

Data Center GPUs

  • 24–80 GB HBM2e/HBM3
  • Bandwidth: 1.5–3.3 TB/s
  • Highest performance tier

The decode phase of LLM inference is memory-bandwidth-bound. GPU memory bandwidth directly determines token generation speed. The goal: keep the current layer's weights and active KV-cache resident in GPU memory.

CPU Main Memory (DRAM)

  • Capacity: 128 GB – 2 TB DDR4/DDR5
  • Bandwidth: 50–200 GB/s (depends on channel count)
  • Roughly 10–30x slower than GPU HBM
  • But 5–80x larger capacity
  • Primary overflow tier for weights

PCIe Bottleneck

CPU-GPU transfer is limited by PCIe:

  • PCIe 4.0 x16: 25–32 GB/s
  • PCIe 5.0 x16: 50–64 GB/s
Memory Hierarchy & Offloading
04

NVMe Storage

  • Capacity: terabytes
  • Sequential read: 3–7 GB/s (consumer), up to 14 GB/s (enterprise)
  • Far too slow for real-time token generation

Best Use Cases

  • Weight prefetching with overlap
  • KV-cache persistence for session resumption
  • Batch/offline workloads

Key Insight

NVMe reads can be overlapped with GPU computation on other layers, partially hiding the latency. Careful scheduling is critical.

Warning

The bandwidth cliff from DRAM to SSD is too steep for interactive inference. Reserve NVMe for offline processing only.

Weight Offloading

Keep some transformer layers' weights in CPU RAM and transfer them to GPU just before needed.

  • Transformers process layers sequentially
  • Prefetch layer N+1 weights from CPU while computing layer N on GPU
  • If transfer time ≤ compute time, offloading is essentially free

Key Parameters

  • Weight size per layer (bytes)
  • PCIe bandwidth (actual, not theoretical)
  • Per-layer compute time
Memory Hierarchy & Offloading
07

Double Buffering

When per-layer transfer time exceeds compute time, double-buffering prevents GPU stalls.

  • Maintain two weight buffers on GPU
  • Alternate: compute from buffer A while filling buffer B
  • GPU is never idle waiting for transfers

Trade-off

Double-buffering reduces effective GPU memory available for KV-cache, since two weight-layer slots are reserved.

KV-Cache Offloading

For long-context or high-batch serving, KV-cache can exceed GPU memory even when all weights fit.

PagedAttention Integration

KV-cache offloading is naturally compatible with PagedAttention's block-based memory management. Individual pages can be migrated between GPU and CPU without disrupting the overall cache structure.

Async Offloading Strategy

  • Offload KV-cache to CPU asynchronously during generation
  • Prefetch when a sequence is scheduled for next decode
  • Requires scheduler to have advance knowledge of active sequences
  • Continuous batching systems (e.g., vLLM) provide this naturally

Hybrid CPU-GPU Inference

Instead of offloading data for GPU processing, assign some layers to execute directly on CPU.

  • Eliminates transfer overhead for those layers
  • Cost: slower CPU compute
  • Works best with GPU handling the majority of layers
  • CPU handles a small number of overflow layers

llama.cpp GPU Layer Splitting

Most widely used implementation. User specifies layers per GPU; remaining layers run on CPU. Handles double-buffering and scheduling automatically.

FlexGen: Throughput-Oriented Offloading

Key Insight

Process a large batch of sequences through one layer at a time, maximizing the compute-to-transfer ratio across the GPU-CPU-SSD hierarchy.

LP-Based Scheduler

FlexGen uses linear programming to automatically determine optimal data placement and batch ordering given specific hardware bandwidth and capacity constraints.

Strengths

  • High throughput for offline workloads
  • Handles models much larger than GPU memory
  • Leverages full GPU-CPU-SSD hierarchy

Limitations

  • Latency: seconds to minutes per token
  • Unsuitable for interactive chat
  • Best for: evaluation, data processing, offline generation

Compression During Transfer

Compress data before PCIe transfer to effectively increase link throughput.

Weight Compression

Quantize from FP16 to INT4 for transfer, then dequantize on GPU after arrival. GPU compute decompresses faster than extra data could be transferred.

KV-Cache Compression

Compress KV blocks before offloading to CPU, decompress on retrieval. Overhead is small relative to transfer time savings.

12

Prefetching & Scheduling

Offloading effectiveness depends almost entirely on whether transfers can be overlapped with computation.

Requirements

  • Asynchronous DMA transfers on dedicated CUDA copy engines
  • Separate CUDA streams for compute and copy
  • Lookahead of at least one layer (weights) or one scheduling round (KV-cache)

Weight Prefetching

Lookahead of 1 layer: while computing layer N, schedule DMA for layer N+1 weights.

KV-Cache Prefetching

Lookahead of 1 scheduling round: know which sequences are active next and prefetch their KV blocks from CPU to GPU.

Critical Rule

Data must arrive in GPU memory before the compute kernel needs it. Scheduling must guarantee this timing.

Unified Memory (UM)

CUDA's Unified Memory provides a single virtual address space spanning GPU and CPU memory with automatic page migration.

Why UM Underperforms for LLMs

  • Page fault and migration overhead is unpredictable
  • Not aligned with model's access patterns
  • Explicit offloading with manual scheduling almost always wins

When UM Is Useful

  • Prototyping offloading strategies
  • Rare edge cases (e.g., exceptionally long sequences that occasionally overflow GPU KV-cache)

Practical Recommendations

1. Quantize First, Offload Second

Apply INT4/INT8 quantization before resorting to offloading. A 70B model at INT4 (~35 GB) may fit across two 24 GB GPUs with no offloading.

2. Profile Bandwidth Budget

Measure actual PCIe throughput (not theoretical) using bandwidthTest. Real throughput is often 20–30% below spec.

3. Use llama.cpp

Battle-tested GPU layer splitting. Easy to configure, handles double-buffering and scheduling automatically.

4. Prioritize KV-Cache on GPU

KV-cache is accessed repeatedly per token; weight layers only once per forward pass. Keep KV-cache on GPU over weights.

5. NVMe: Batch Only

DRAM-to-SSD bandwidth cliff is too steep for interactive inference. Use NVMe for offline processing or KV-cache persistence.

6. Monitor PCIe Contention

Multi-GPU systems may share PCIe lanes. Check topology with nvidia-smi topo -m and account for shared bandwidth.

Conclusion

Interactive Workloads

  • Quantize to minimize memory footprint
  • Modest CPU offloading (few layers)
  • Double-buffered prefetching
  • Keep KV-cache on GPU

Batch Workloads

  • Aggressive offloading across GPU + CPU + SSD
  • Large batch sizes for throughput
  • LP-based scheduling (FlexGen)
  • Prioritize throughput over latency

Key Principle

  • Measure actual bandwidth at each tier
  • Schedule transfers to hide behind computation
  • Profile real PCIe throughput
  • Monitor contention in multi-GPU setups

Memory offloading extends effective memory capacity of small-GPU systems, enabling inference of models that would otherwise be inaccessible.
