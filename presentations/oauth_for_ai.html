CONTEXT

Why AI Applications Need OAuth

The API Key Problem

  • API keys are shared secrets with no identity, scope, or expiration
  • Agent context windows, logs, and tool descriptions are all leak vectors
  • No way to limit what the agent does with the key
  • No audit trail connecting agent actions to the initiating user

What OAuth Provides

  • Scoped, time-limited tokens instead of permanent keys
  • Delegated authorization — agent acts on behalf of the user
  • Revocable access at any time
  • Full audit trail: who, what, when, through which agent

The MCP Catalyst

MCP’s specification for remote server authentication is built on OAuth 2.1, including PKCE, dynamic client registration, and token-based access control. OAuth literacy is now essential for the agent ecosystem.

FOUNDATIONS

OAuth 2.0/2.1 Core Concepts

The Four Roles

  • Resource Owner — the user who grants access
  • Client — the AI agent / MCP host
  • Authorization Server — issues tokens (Auth0, Okta, Keycloak)
  • Resource Server — the MCP server / API endpoint

Key Primitives

  • Access Tokens — short-lived (5–60 min), scoped credentials
  • Refresh Tokens — long-lived, used to obtain new access tokens
  • Scopes — permission strings (e.g. tools:execute)
  • Audiences — which resource server the token targets

FLOW

Authorization Code Flow

1. Authorize

Agent redirects user to auth server with requested scopes, redirect URI, state, and PKCE challenge.

2. Consent

User authenticates and approves permissions. Auth server returns a short-lived authorization code via redirect.

3. Exchange

Agent exchanges code + PKCE verifier for an access token and refresh token. User credentials never exposed to agent.

Client Credentials Flow

For machine-to-machine (no user present): agent authenticates directly with client ID + secret. Used for backend MCP server-to-server calls.

Implicit Flow (Removed)

OAuth 2.1 removes the implicit flow entirely. Tokens in URL fragments are exposed to browser history and referrer headers. Use auth code + PKCE instead.

SECURITY

PKCE: Proof Key for Code Exchange

Why PKCE Is Mandatory in OAuth 2.1

  • Authorization codes travel through the browser — interception risk
  • Dynamically registered MCP clients are public clients (no pre-shared secret)
  • PKCE adds proof-of-possession: only the original requester can exchange the code
  • Cost: virtually zero — one hash computation

Security Property

Even if an attacker intercepts the authorization code and the code challenge, they cannot derive the code verifier (SHA-256 is one-way).

The PKCE Steps

  • 1. Client generates random code_verifier (43–128 chars)
  • 2. Compute code_challenge = SHA256(verifier)
  • 3. Send challenge with auth request
  • 4. Auth server stores challenge alongside code
  • 5. Client sends verifier with token exchange
  • 6. Server hashes verifier, compares to stored challenge

MCP

MCP’s OAuth Integration

Server Metadata Discovery

  • MCP client fetches /.well-known/oauth-authorization-server
  • Returns auth endpoint, token endpoint, registration endpoint
  • Supported scopes, grant types, PKCE methods
  • Automated — no pre-configured knowledge needed

Dynamic Client Registration

  • RFC 7591 — client registers itself at runtime
  • POSTs client_name, redirect_uris, grant_types
  • Receives client_id (and optionally client_secret)
  • Users connect to new MCP servers without manual config

MCP

Remote MCP Server Auth Flow

Step-by-Step

  • 1. Client connects → server responds 401 Unauthorized
  • 2. Client fetches OAuth metadata from well-known endpoint
  • 3. Client performs dynamic registration (if needed)
  • 4. Client initiates auth code flow with PKCE
  • 5. User authenticates and approves scopes
  • 6. Client exchanges code for tokens
  • 7. Client retries connection with Bearer token
  • 8. Server validates token → MCP session established

Security Properties

  • PKCE required for all auth code flows
  • Redirect URI must use localhost for desktop apps
  • Metadata endpoint served over HTTPS
  • Token sent only in Authorization header
  • Never in query params or MCP message content

Ongoing Session

  • Access token included automatically on each request
  • Refresh token used transparently when token expires
  • No user re-authentication needed

TOKENS

Token Management for Agents

The Long-Running Agent Challenge

  • Agents may run for hours or days
  • Access tokens expire in 15–60 minutes
  • Requires robust refresh token mechanisms
  • Agent crash during rotation can lose access

Refresh Token Rotation

  • Each refresh invalidates the old token and issues a new one
  • Stolen token can only be used once before detection
  • Must use atomic persistence — store new token before using it
  • Recommended by OAuth 2.1

Token Storage Hierarchy (Most to Least Secure)

  • HSMs — tamper-resistant hardware, highest security
  • Secret managers (Vault, AWS Secrets Manager, GCP Secret Manager) — recommended for production
  • Encrypted filesystem — acceptable for desktop agents (how most MCP hosts store tokens today)
  • Environment variables — in-memory only, lost on restart
  • Plaintext files — unacceptable, equivalent to the API key problem

PATTERNS

Token Lifecycle Patterns

Proactive Refresh

Refresh token at 75% of lifetime elapsed — avoid failed requests and retry complexity. Never wait for a 401.

Retry with Refresh

If 401 despite valid-looking token (server-side revocation), attempt one refresh + retry. If refresh fails, surface to user.

Graceful Degradation

On refresh failure: pause workflow, notify user, resume from paused state after re-authentication.

Per-Tool Token Scoping

Separate token sets per MCP server. Each scoped to minimum permissions. Stored and refreshed independently.

SCOPES

Designing OAuth Scopes for AI Tools

Two-Level Hierarchy

  • Resource-level: database, files, github, slack
  • Action-level: :read, :write, :admin
  • Yields: database:read, files:write, github:read
  • Granular enough for least privilege, coarse enough to manage

Scope Escalation Risks

  • Transitive access: tools:execute + SQL tool = full DB access
  • Scope aggregation: read users + write email = spam all users
  • Dynamic inheritance: new tools inherit existing scopes

MCP Server Scopes

DELEGATION

The Principal Chain

Token Propagation

Agent passes its access token to MCP server. Never shares user’s original credentials. Token encodes sub, client_id, scopes.

Token Exchange (OBO)

MCP server exchanges agent token for a downstream token scoped for the backend API. RFC 8693 On-Behalf-Of flow.

Audit Trail

Each hop logs token claims: who (sub), which agent (client_id), what (scope), where (aud).

M2M

Machine-to-Machine Authentication

Client Credentials

  • No user involved — agent authenticates as itself
  • Client ID + secret (or assertion) → access token
  • Permissions configured by admin, not user consent
  • Ideal for backend pipelines

Service Accounts

  • Per-environment identities: prod, staging, dev
  • Prod: database:read only
  • Dev: admin:* on non-prod resources
  • Managed via IdP API, not hardcoded

Workload Identity

  • No static secrets — identity from runtime environment
  • AWS: IAM role → signed identity document
  • GCP: service account → Google-signed JWT
  • K8s: projected service account tokens

Recommendation

Use workload identity for cloud-deployed agents. It eliminates static secrets entirely — the agent’s identity is tied to its runtime environment, and credentials rotate automatically.

JWT

JWT Structure and AI-Specific Claims

Standard Claims

  • iss — issuer (auth server URL)
  • sub — subject (user ID)
  • aud — audience (target MCP server)
  • client_id — the agent application
  • scope — granted permissions
  • exp / iat — expiry and issued-at

AI-Specific Custom Claims

  • agent_type — coding_assistant, data_pipeline
  • session_id — links token to agent session
  • tool_restrictions — explicit tool allowlist
  • max_actions — invocation budget
  • delegation_chain — full delegation path

Example JWT Payload

  "iss": "https://auth.example.com",

VALIDATION

Token Validation & Introspection

Local JWT Validation

  • Parse JWT, extract header kid
  • Fetch signing key from JWKS endpoint (cached)
  • Verify signature (RS256 / ES256)
  • Check exp, nbf, iss, aud
  • Check scope contains required permissions
  • No network call per request — fast

Token Introspection (RFC 7662)

  • For opaque tokens or revocation checks
  • POST to /oauth/introspect with token
  • Response: {"active": true, "sub": "...", ...}
  • Adds network round-trip per request
  • Typically used at session start or periodically

Audience Restrictions in Multi-Server Agents

An agent with tokens for 3 MCP servers should hold separate tokens with distinct audiences. Token for DB server rejected by GitHub server — prevents cross-server token replay.

  • Dynamic client registration via Management API

Okta

Keycloak (OSS)

  • Realm-per-tenant for multi-tenant isolation
  • Client Policies: multi-attribute authorization
  • Built-in RFC 8693 token exchange
  • Self-hosted, containerized, edge-friendly

OIDC for Agent Identity

OpenID Connect adds authentication on top of OAuth. Enables agent identity verification (ID tokens), federated identity across organizations, and identity propagation through the agent-tool chain.

MULTI-TENANT

Multi-Tenant AI Deployments

Per-Tenant Auth Servers

  • Each tenant gets its own authorization server (or Keycloak realm)
  • Tokens signed with different keys
  • Structurally impossible to cross-access

Tenant-Aware Audiences

  • Shared auth server, tenant in audience claim
  • aud: ".../tenants/acme"
  • MCP server validates tenant match

Data Residency

  • Auth server endpoints in required region
  • Token cache respects geographic boundaries
  • Introspection routed to regional server

GATEWAY

API Gateway Patterns

Token Validation at the Edge

  • Gateway validates OAuth tokens before requests reach MCP server
  • Offloads validation from application code
  • Centralized enforcement point
  • Kong, Envoy, AWS API Gateway, Azure APIM

Rate Limiting by Token Claims

  • Per-user limits: based on sub claim
  • Per-client limits: based on client_id
  • Per-scope limits: writes stricter than reads
  • Quota budgets: monthly token-based quotas

API Key → OAuth Migration Path

  • Phase 1: Gateway accepts both API keys and OAuth tokens. Keys mapped to synthetic internal tokens.
  • Phase 2: New clients must use OAuth. Existing key clients get deprecation warnings.
  • Phase 3: API keys disabled. All access is OAuth-based. Gateway sees only OAuth tokens.

SECURITY

Security Pitfalls

Token Leakage Through Prompts

  • Tool responses may include tokens in error messages
  • LLM may echo token in response or store in history
  • Fix: sanitize errors, keep auth metadata outside model context

Bearer Tokens in Tool Descriptions

  • Some servers embed Authorization: Bearer sk-... in tool descriptions
  • Token is now in context — exfiltrable via prompt injection
  • Fix: auth at transport layer, never in prompt layer

Overly Broad Scopes

  • Requesting admin:* when only database:read is needed
  • Compromised token grants full access
  • Fix: request minimum scopes, use incremental authorization

Refresh Token Theft

  • Long-lived refresh tokens are high-value targets
  • Fix: rotation, sender-constrained tokens (DPoP/mTLS)
  • Fix: limit lifetime to expected task duration

Protecting an MCP Tool Endpoint (Python)

Token Validation

from authlib.jose import jwt, JsonWebKey
import httpx

ISSUER = "https://auth.example.com"
AUDIENCE = "https://mcp.example.com"

def get_jwks():
    resp = httpx.get(f"{ISSUER}/.well-known/jwks.json")
    return JsonWebKey.import_key_set(resp.json())

async def validate_token(token: str) -> dict:
    jwks = get_jwks()
    claims = jwt.decode(token, jwks)
    claims.validate()
    assert claims["iss"] == ISSUER
    assert claims["aud"] == AUDIENCE
    return dict(claims)

Scope Checking

def require_scope(claims, required):
    scopes = claims.get("scope", "").split()
    if required not in scopes:
        raise HTTPException(403,
          f"Requires '{required}'")
    return claims

# Tool endpoint
@app.post("/mcp/tools/call")
async def call_tool(request, claims=Depends(
    require_scope("tools:execute"))):
    tool = request.json()["name"]
    log(f"{tool} by {claims['sub']} "
        f"via {claims['client_id']}")
    # execute tool...

CODE

Protecting an MCP Tool Endpoint (TypeScript)

Token Validation Middleware

import { createRemoteJWKSet, jwtVerify } from 'jose';

const ISSUER = 'https://auth.example.com';
const AUDIENCE = 'https://mcp.example.com';
const jwks = createRemoteJWKSet(
  new URL(`${ISSUER}/.well-known/jwks.json`));

async function validateToken(req, res, next) {
  const token = req.headers.authorization?.slice(7);
  if (!token) return res.status(401).json({
    error: 'Missing bearer token' });
  const { payload } = await jwtVerify(token, jwks,
    { issuer: ISSUER, audience: AUDIENCE });
  req.claims = payload;
  next();
}

Scope + Tool Endpoint

function requireScope(scope) {
  return (req, res, next) => {
    const scopes = req.claims?.scope?.split(' ');
    if (!scopes?.includes(scope))
      return res.status(403).json({
        error: `Requires '${scope}'` });
    next();
  };
}

app.post('/mcp/tools/call',
  validateToken,
  requireScope('tools:execute'),
  async (req, res) => {
    console.log(`${req.body.name} by `
      + `${req.claims.sub}`);
    // execute tool...
  });

STANDARDS

Evolving Standards Landscape

DPoP (RFC 9449)

  • Binds token to client’s private key
  • Each request includes a signed DPoP proof
  • Stolen token is useless without the key
  • Critical for high-security agent deployments

RAR (RFC 9396)

  • Replaces flat scopes with structured authorization
  • Request: specific tools, row limits, denied operations
  • Consent: “agent wants to query users table, max 1000 rows”
  • Natural fit for tool-based AI apps

GNAP (RFC 9635)

  • Next-gen authorization — designed for agent architectures
  • Instance-based client identity (key-based, no pre-registration)
  • Multiple interaction modes negotiated per-request
  • Built-in delegation chains & ongoing access management

OAuth 2.1 (RFC 9728 Draft)

  • Consolidation of OAuth 2.0 best practices
  • Mandatory PKCE, no implicit flow
  • Refresh token rotation recommended
  • Strict redirect URI matching

SUMMARY

Key Takeaways

Why OAuth for AI

  • API keys lack scope, expiration, and identity
  • Agents need delegated, time-limited access
  • MCP mandates OAuth 2.1 for remote servers
  • Complete audit trail: user → agent → tool → API

Implementation Checklist

  • PKCE on all authorization code flows
  • Audience restrictions per MCP server
  • Minimum scopes per task, not blanket admin
  • Secret managers for token storage
  • Proactive refresh for long-running agents

The Principle

An AI agent should have exactly the permissions it needs, for exactly as long as it needs them, with a complete audit trail of everything it did with those permissions. OAuth 2.1 is the framework that makes this possible.
