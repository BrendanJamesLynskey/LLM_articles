The Cost of Full Fine-Tuning

Full fine-tuning updates every parameter, requiring memory for weights, gradients, and optimizer states.

Memory Breakdown (N params, BF16 + Adam)

  • 2N — BF16 model weights
  • 2N — BF16 gradients
  • 4N — FP32 master weights
  • 8N — Adam first & second moments
  • Total: 16N bytes

Concrete Example: 70B Model

  • Weights alone: 140 GB (BF16)
  • Full training state: ~1.12 TB
  • Exceeds 8× A100-80GB node
  • Requires ZeRO-3 or FSDP sharding

Fine-tuning updates occupy a low-dimensional subspace — PEFT methods exploit this intrinsic dimensionality to cut trainable params by 90–99%.

LoRA: Low-Rank Decomposition

Freeze all pre-trained weights W. Inject trainable low-rank matrices A (r × d_in) and B (d_out × r):

h = W·x + (α/r) · B·A·x

A ← N(0, σ²)   # random init
B ← 0             # zero init → starts as identity

Trainable params per layer: r × (d_in + d_out) vs. d_in × d_out for full.

For a 4096×4096 layer with r=16: 131K vs 16.8M params (0.78%).

Rank Selection & Alpha Scaling

Rank (r)

  • Primary capacity knob: typical range 4–256
  • Popular defaults: r=16 or r=32
  • Lower rank = stronger regularization, good for small datasets
  • Higher rank = more capacity for domain shift
  • Diminishing returns: 4→8 helps more than 16→32

Alpha (α)

  • Effective scaling: α/r
  • Common heuristic: α = 2×r
  • α=r ⇒ scaling factor of 1.0
  • Higher α amplifies adapter contribution
  • Interacts with learning rate — tune jointly

Which Layers to Adapt

LoRA can target any linear layer. The choice affects both parameter count and final quality.

TargetLayersImpact
Q, V onlyAttentionMinimal params, strong baseline
Q, K, V, OFull attentionBetter for complex tasks
All linearAttn + MLPMaximum capacity, highest param count

MLP layers contain the majority of model parameters. Including gate_proj, up_proj, down_proj substantially increases adapter size but also representational capacity.

Recommendation

For most tasks, start with all attention + MLP projections at a moderate rank (16). This gives the adapter access to all representational pathways while keeping the total parameter count manageable.

HuggingFace PEFT config

target_modules = [
  "q_proj", "k_proj",
  "v_proj", "o_proj",
  "gate_proj", "up_proj",
  "down_proj"

QLoRA: Quantized Base + LoRA

Key Innovations

  • NF4 quantization — 4-bit NormalFloat, optimal for normally distributed weights
  • Double quantization — quantization constants themselves quantized to 8-bit
  • Paged optimizers — offload optimizer state to CPU on GPU OOM
  • Base weights: 4-bit, LoRA adapters: BF16

Memory Impact

  • 65B model fine-tuning on a single 48 GB GPU
  • 7B model: only 6–8 GB needed
  • Quality gap vs BF16 LoRA: <1–2% on benchmarks
  • Democratized LLM fine-tuning for consumer hardware

Other PEFT Methods

Prefix Tuning / Prompt Tuning

Prepend learnable continuous vectors to key/value sequences (prefix) or input embeddings (prompt). Extremely parameter-efficient (<0.1% params) but underperforms LoRA on complex tasks.

Adapters

Bottleneck layers (down → nonlinearity → up) inserted between transformer sublayers. Can match LoRA quality but add sequential latency to inference, limiting adoption in production.

IA3

Learns element-wise rescaling vectors for keys, values, and FF activations. 10–100x fewer params than LoRA with negligible overhead. Best for tasks close to base model capabilities.

DoRA

Decomposes weights into magnitude + direction, applies LoRA to direction only. Outperforms LoRA by 1–3% on instruction-following at equivalent param count.

DoRA: Weight-Decomposed Low-Rank Adaptation

Full fine-tuning tends to make large directional changes with small magnitude adjustments. Standard LoRA struggles to replicate this pattern.

# DoRA decomposition
W = m · (W_0 + BA) / ||W_0 + BA||

where:
  m     = learnable magnitude vector (d_out)
  W_0   = frozen pre-trained weights
  B, A  = standard LoRA matrices

The magnitude vector m is initialized to ||W_0|| column norms. LoRA handles direction; m handles scale.

Benchmark Results

  • MT-Bench: +1.2 pts over LoRA
  • AlpacaEval: +2.8% win rate
  • Commonsense reasoning: +1.5% avg
  • Extra overhead: ~5% training time
  • Merges cleanly into base weights

Multi-LoRA Serving

In production, one base model serves many adapters for different use cases. Naive per-adapter instances waste memory and prevent cross-user batching.

S-LoRA

  • Unified paging for adapter memory pool
  • Custom CUDA kernels: different LoRA per request within a single batch
  • On-demand adapter loading from host/disk

LoRAX (LoRA eXchange)

  • Production-ready multi-adapter serving
  • Dynamic loading + LRU cache eviction
  • HuggingFace ecosystem integration

Merging Adapters

Single Adapter Merge

W_new = W + (α/r) · B·A

Simple additive merge. Zero inference overhead. Irreversible but ideal for single-purpose deployment.

Multi-Adapter Merge

W_new = W + Σ(λ_i · ΔW_i)

Linear combination of task deltas. Tune λ_i on validation set for optimal mixing.

Advanced Merging Strategies

  • TIES-Merging — prune conflicting parameter updates, resolve sign conflicts, then merge
  • DARE — randomly drop delta elements before merging, reducing interference
  • Task Arithmetic — treat deltas as vectors: add to combine capabilities, negate to remove them

Example: Multi-Skill Merge

  • ΔW_code — coding adapter
  • ΔW_math — math reasoning
  • ΔW_instruct — instruction following
  • Merged model gains all three capabilities

Practical Guidelines

Starting Config

  • Target: all attn + MLP projections
  • Rank: 16
  • Alpha: 32
  • LR: 1e-4 to 3e-4
  • Framework: HF PEFT / Axolotl

Memory-Constrained

  • Use QLoRA + NF4
  • Enable gradient checkpointing
  • Paged optimizers for overflow
  • 7B model: 6–8 GB VRAM
  • Quality gap: <2%

When to Scale Up

  • Val loss plateaus early → raise rank
  • Large domain shift → r=64 or 128
  • Watch: params vs tokens/1000
  • If adapter params > tokens/1000 → overfitting risk

    task_type="CAUSAL_LM"
)
model = get_peft_model(model, config)
model.print_trainable_parameters()  # ~0.5-2% of total

Training Tips & Common Pitfalls

Do

  • Use BF16 for LoRA params (not FP16 — avoids loss scaling issues)
  • Apply LoRA dropout (0.05–0.1) for regularization
  • Save adapter-only checkpoints (tiny vs full model)
  • Use cosine LR schedule with warmup
  • Evaluate on held-out set every N steps
  • Tune alpha and LR together on a small sweep

Avoid

  • Rank too high for small datasets → overfitting
  • Forgetting to freeze base model → OOM + instability
  • FP16 with QLoRA → NaN gradients from loss scaling conflicts
  • Ignoring chat template → model learns wrong format
  • Training too long → 1–3 epochs usually sufficient
  • Mixing adapter + base ckpts from different commits

Key Takeaways

1. LoRA is the Default

Low-rank adaptation provides 95–100% of full fine-tuning quality at 1–5% of the parameter cost. It should be your starting point for any LLM adaptation task.

2. QLoRA Democratizes Access

4-bit quantized base + BF16 LoRA makes fine-tuning 7B–70B models feasible on consumer GPUs, with minimal quality loss.

3. Adapters are Composable

LoRA adapters can be merged, combined, swapped, and served concurrently, enabling flexible multi-tenant production deployments.

4. The Ecosystem is Mature

From training (PEFT, Axolotl) to serving (vLLM, LoRAX) to merging (mergekit), the toolchain for parameter-efficient fine-tuning is production-ready.

5. Know When Full FT Wins

For continued pre-training on large corpora or fundamental capability changes, full fine-tuning still provides the highest quality ceiling. Use LoRA for everything else.

6. DoRA is the Next Step

When you need that extra 1–3% quality improvement, DoRA's magnitude-direction decomposition offers a near-free upgrade over standard LoRA.
