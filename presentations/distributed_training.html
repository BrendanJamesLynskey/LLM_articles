The Scale Problem

Training large language models requires computational resources that vastly exceed any single accelerator.

  • Hundreds of billions of parameters
  • Tens of terabytes of memory for weights, gradients, optimizer states, and activations
  • Millions of GPU-hours of computation
  • Workload must be split across hundreds to thousands of accelerators

Four Pillars of Parallelism

Data Parallelism

Replicate model, split data across devices. Synchronize gradients via all-reduce.

Tensor Parallelism

Split individual layers across devices. Each GPU computes part of every layer.

Pipeline Parallelism

Assign groups of consecutive layers to different devices. Micro-batches flow through stages.

Expert Parallelism

Distribute MoE experts across devices. All-to-all communication routes tokens to experts.

Combined as 3D parallelism (or 4D with expert parallelism) for large-scale training

Data Parallelism

How It Works

  • Each accelerator holds a complete copy of the model
  • Each processes a different micro-batch of training data
  • Gradients aggregated via all-reduce after each step
  • All replicas remain perfectly synchronized

Properties

  • Effective batch size scales linearly with device count
  • Per-step throughput scales near-linearly for compute-bound workloads
  • Simplest and most widely used form

Limitations of Data Parallelism

Memory Redundancy

Each GPU stores a full copy of weights, gradients, and optimizer states.

  • 70B param model with Adam in mixed precision
  • ~840 GB per replica
  • Far exceeds any single GPU memory (80 GB H100)

Communication Bottleneck

Gradient all-reduce becomes a bottleneck at scale.

  • Communication volume grows with model size
  • Network bandwidth doesn't scale with compute
  • Synchronization overhead increases with device count

These limitations motivated ZeRO and complementary parallelism strategies.

ZeRO: Zero Redundancy Optimizer

Developed by Microsoft DeepSpeed. Eliminates memory redundancy by partitioning instead of replicating model states.

Stage 1

Partition optimizer states (Adam momentum & variance)

~4x memory reduction

Stage 2

Additionally partition gradients

~8x memory reduction

Stage 3

Also partition model weights themselves

Full sharding — no per-GPU limit

With ZeRO-3, each GPU stores only a fraction of the total model; aggregate memory holds everything.

FSDP: Fully Sharded Data Parallelism

PyTorch's native implementation of ZeRO-3-style sharding.

Key Features

  • Shards parameters, gradients, and optimizer states across workers
  • Gathers parameters on demand for computation
  • Releases after use to free memory
  • Mixed precision training support
  • Activation checkpointing integration
  • Flexible sharding granularity control

Now the standard approach for memory-efficient data-parallel training in PyTorch.

Tensor Parallelism

Splits individual layers across multiple GPUs, each computing a portion of the layer output.

Transformer Splitting Strategy

  • Q, K, V projections partitioned column-wise
  • Output projection partitioned row-wise
  • Only one all-reduce per layer
  • Feed-forward layers similarly split

Best Within a Node

  • Requires NVLink / NVSwitch (400-900 GB/s)
  • Frequent all-reduce makes it sensitive to latency
  • Typical TP degree: 2 to 8 GPUs

Pipeline Parallelism

Assigns consecutive groups of layers to different accelerators.

  • Device 0: layers 1..N
  • Device 1: layers N+1..2N
  • Only inter-stage activations communicated (point-to-point)
  • Suitable for cross-node distribution (lower bandwidth OK)

Pipeline Bubble Problem

Bubble fraction ≈ (P-1)/M where P = stages, M = micro-batches.

  • GPipe: many micro-batches reduce bubbles
  • PipeDream / 1F1B: start backward before all forwards complete
  • State-of-the-art: bubble overhead down to a few percent

Expert Parallelism (MoE)

For Mixture-of-Experts models, a fourth dimension of parallelism distributes experts across devices.

How It Works

  • Different experts placed on different devices
  • All-to-all communication routes tokens to the correct expert
  • Each token dispatched to its assigned expert(s)
  • Results gathered back after expert computation

Challenge

Combining expert parallelism with TP + PP + DP creates complex communication patterns requiring careful orchestration for high efficiency.

Strategy Comparison

Strategy What is Distributed Communication Pattern Best Suited For
Data Parallel Data (batches) All-reduce (gradients) Scaling batch size
ZeRO / FSDP Model state + data All-gather + reduce-scatter Memory-limited training
Tensor Parallel Individual layers All-reduce per layer Intra-node (high BW)
Pipeline Parallel Groups of layers Point-to-point activations Inter-node (lower BW)
Expert Parallel MoE experts All-to-all (token dispatch) MoE architectures

In practice, large-scale systems combine all applicable strategies simultaneously.

Communication Optimization

At scale, communication efficiency is as important as compute efficiency.

Overlap Compute & Comms

Pipeline gradient all-reduce with backward pass computation so they run concurrently.

Gradient Compression

Reduce communication volume by compressing gradients before transmission.

Hierarchical All-Reduce

Exploit network topology: intra-node reduction before inter-node, matching hardware hierarchy.

Avoid Serialization

Careful placement of communication primitives to prevent bottlenecks from blocking the pipeline.

Fault Tolerance & Checkpointing

At thousands of GPUs running for weeks, hardware failures are not exceptional but routine.

The Reality

Mean time between failures at the largest scales can be measured in hours. Fast recovery is essential.

Solutions

  • Periodic checkpointing saves model state
  • Asynchronous checkpointing avoids blocking training
  • Elastic training adapts to changing GPU availability
  • Seamless resume after failures

Determinism & Reproducibility

Achieving deterministic results across distributed configurations is difficult.

Floating-Point Non-Associativity

Collective operations (e.g., all-reduce) yield different results depending on reduction order due to float math.

Non-Deterministic Kernels

GPU kernel execution can vary between runs, producing slightly different numerical results.

Communication Ordering

Variations in timing and ordering of network operations introduce subtle non-determinism.

Practical Approach

Perfect bit-level reproducibility is often impractical. Instead, aim for approximate reproducibility via controlled random seeds, fixed communication ordering, and numerical precision management. Essential for debugging and scientific rigor.

Future Directions

Sequence & Context Parallelism

  • Sequence parallelism distributes the sequence dimension for long-context training
  • Context parallelism distributes attention computation across devices for very long sequences
  • Essential as context windows grow beyond single-device capacity

Improved Interconnects

  • NVLink 5.0 — higher bandwidth intra-node
  • CXL — memory-coherent interconnect
  • Custom network topologies for large clusters
  • Reducing communication as the core bottleneck

Automatic Parallelism

Compiler-driven tools like XLA GSPMD and Alpa aim to discover optimal parallelism strategies automatically, given a model and hardware configuration — reducing the need for manual parallel configuration.

Key Takeaways

Distributed training is the engineering backbone of LLMs

Without it, models beyond a few billion parameters would be physically impossible to train.

Hybrid parallelism is the standard

TP within nodes, PP across nodes, DP across replica groups — each strategy at its optimal communication scale.

Memory optimizations are essential

ZeRO / FSDP make it possible to train models whose state far exceeds any single device.

Systems challenges grow with scale

Communication efficiency, fault tolerance, and automated parallelism configuration are active and consequential areas of research.
