Why GPUs Are Suboptimal

Memory Bandwidth Bottleneck

  • Autoregressive decode reads full model weights per token
  • H100: 3.35 TB/s HBM3, 989 TFLOPS FP16
  • 70B FP16 = 140 GB weights → ~42 ms per read
  • Upper bound: ~24 tok/s single-stream
  • Arithmetic intensity < 1 FLOP/byte during decode

Sources of Unpredictability

  • Cache misses → unpredictable stalls
  • Dynamic warp scheduling → variable timing
  • Memory controller contention across SMs
  • PCIe / NVLink traffic variance
  • Compute sits largely idle during decode

The mismatch between GPU architecture and inference workloads creates an opening for purpose-built hardware.

TSP Architecture Overview

Spatial Dataflow Processor

Data flows through a pipeline of functional units in a predetermined order. The compiler plans exactly where every byte will be at every cycle.

Key Specifications

  • Die size: 25 mm × 29 mm
  • Process: GlobalFoundries 14nm
  • Clock: 900 MHz nominal
  • Density: >1 TeraOp/s per mm²
  • SRAM: ~230 MB on-chip

~24x Bandwidth Advantage

80 TB/s vs 3.35 TB/s per chip. Directly translates to faster weight reads during decode.

~350x Capacity Disadvantage

230 MB vs 80 GB. Even a 7B model in FP16 (14 GB) requires ~60 chips. A 70B model needs 300+.

Deterministic Execution

Every operation mapped to a specific functional unit at a specific clock cycle at compile time.

GPU: Runtime Decisions

  • Branch prediction hardware
  • Dynamic warp scheduling
  • Speculative execution
  • Runtime arbitration for shared resources
  • Variable latency per token

TSP: Compile-Time Planning

  • No branches — control flow fully unrolled
  • No scheduler — pre-planned instruction stream
  • No speculation — every op known needed
  • No contention — conflict-free by construction
  • Fixed cycles per operation, always

p99 = Median

Every token takes the same time. The p99 latency on a Groq system is essentially identical to the median — critical for real-time voice, robotics, and financial services.

Software-Scheduled Hardware

Complexity Moved to Compiler

GPUs invest billions of transistors in scheduling logic, cache controllers, branch predictors, and MMUs. The TSP eliminates all of these, using freed silicon for more SRAM and compute.

The Groq Compiler

  • Graph optimization — fusion, folding, DCE
  • Memory allocation — every tensor to a specific SRAM bank
  • Instruction scheduling — exact cycle per operation
  • Multi-chip partitioning — tensor parallel splits
  • Quantization — INT8 / FP8 post-training

Historical Analogy

Similar to CISC → RISC and OoO → VLIW: moving intelligence from hardware to software yields efficiency when workloads are predictable.

Multi-Chip Scaling

Plesiosynchronous Interconnect

  • Each chip has its own crystal oscillator
  • "Nearly synchronous" protocol compensates for clock drift
  • Software-initiated periodic synchronization
  • Hundreds of LPUs operate as one logical processor
  • Compiler predicts exact arrival time of data at each chip

Tensor Parallelism

  • Weight matrices partitioned across chips
  • Each chip holds a slice in local SRAM
  • Partial results exchanged with neighbors
  • Communication scheduled as part of static instruction stream
  • Computation and communication interleaved

FEC, Not Retransmission

Inter-chip links use Forward Error Correction rather than retransmission. Retransmission would introduce non-deterministic delays, violating the architecture's core guarantee.

GPU Contrast

GPU clusters use asynchronous NVLink/PCIe communication with variable latency. Groq extends determinism from chip to system.

Scaling Target

Architecture designed for >10,000 chips with minimal latency degradation.

Trade-Offs & Limitations

Inference Only

No training support. Static scheduling incompatible with SGD, dynamic loss scaling, and backward passes.

No Flexibility

Changing architecture requires recompilation (hours). No dynamic control flow. No adaptive computation.

Narrow Ecosystem

No equivalent to CUDA's thousands of libraries. Limited operator support compared to GPU frameworks.

Cost & Batch Economics

Capital Cost: 70B Model

  • GroqCard: ~$20,000 per chip
  • 300 chips × $20K = ~$6M chip cost
  • Excludes servers, networking, power, cooling

GPU Comparison

  • H100: $25K–$40K per GPU
  • 2–8 GPUs for 70B = $50K–$320K
  • ~20–120x lower upfront

Where Groq Wins on Cost

  • Cost per token (low batch): competitive — higher tok/s amortizes hardware
  • Cost per unit of latency: wins by wide margin
  • GroqCloud pricing competitive with GPU providers

Where GPUs Win on Cost

  • Cost per token (high batch): batching amortizes weight reads across many sequences
  • Offline processing, batch embeddings
  • High-concurrency API where latency is secondary

Power: 300 chips × 240W avg = ~72 kW vs ~1.4 kW for 2× H100

  • May integrate TSP concepts into future GPUs
  • Or offer as separate product line

  • 405B+ models: 1,700+ chips in FP8
  • Path forward: aggressive quantization (FP4, sub-4-bit)
  • Potential hybrid architectures: SRAM for weights, off-chip for KV-cache
  • Market may fragment by workload profile

The Broader Question

Will LLM inference converge on GPUs with better software, or fragment into specialized hardware? Historical precedent (networking ASICs, video encoders, crypto miners) suggests sufficiently large, stable workloads eventually justify purpose-built hardware. Most likely: coexistence — GPUs for training + flexible inference, Groq for latency-critical serving, Cerebras for extreme on-chip capacity.

Key Takeaways

Architecture Innovation

  • Trade memory capacity for bandwidth (80 TB/s SRAM)
  • Trade hardware flexibility for determinism
  • Trade runtime scheduling for compile-time planning
  • Result: ~10x single-stream token speed vs GPU

Best-Fit Workloads

  • Latency-critical: voice, chat, coding assistants
  • Low-batch / single-stream serving
  • Deterministic requirements: robotics, finance
  • Models in the 7B–70B range

Clear Limitations

  • Inference only — no training
  • 230 MB/chip → 300 chips for 70B
  • ~$6M chip cost for a single 70B deployment
  • Static compilation; narrow software ecosystem

Market Position

  • NVIDIA acquisition validates the approach
  • Cerebras WSE-3 is the strongest competitor
  • LPU v2 (4nm) could transform economics
  • The memory bandwidth wall, not compute, is the barrier worth breaking

The LPU's contribution: demonstrating that the design space for inference hardware is far larger than the GPU-centric ecosystem assumed.
