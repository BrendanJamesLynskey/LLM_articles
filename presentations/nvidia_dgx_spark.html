ARCHITECTURE

GB10 Grace Blackwell Superchip

Multi-Die SoC

  • Co-designed by NVIDIA and MediaTek
  • Fabricated on TSMC 3nm process
  • Arm CPU die + Blackwell GPU die on single package
  • Connected via NVLink chip-to-chip (NVLink-C2C)

NVLink-C2C Interconnect

  • Approximately 5× bandwidth of PCIe Gen 5
  • Enables coherent unified memory address space
  • No explicit data transfers between CPU and GPU
  • Both processors share same address space transparently

CPU

Arm big.LITTLE CPU Configuration

Performance Cores

  • 10 × Cortex-X925 cores
  • Clock speed up to 4 GHz
  • Single-threaded performance comparable to Apple M4
  • Off-the-shelf Arm designs (not Neoverse)

Efficiency Cores

  • 10 × Cortex-A725 cores
  • Clock speed up to 2.8 GHz
  • Power-efficient for background tasks
  • Reflects desktop power envelope design

Design Trade-off

Unlike NVIDIA's data center Grace CPUs that use Neoverse cores, the GB10 uses consumer-class Arm cores to fit a desktop thermal and power envelope. Despite this, the system serves as a capable general-purpose workstation even without GPU acceleration.

GPU

Blackwell GPU — 1 PFLOP AI Compute

Compute

  • 48 SMs, 6,144 CUDA cores
  • 5th-gen Tensor Cores (FP4/FP8)
  • 4th-gen RT cores
  • 1 PFLOP sparse FP4

Cache & Engines

  • 24 MB L2 cache
  • 16 MB L4 side cache
  • 2 copy engines for bidirectional transfers
  • 1 NVENC + 1 NVDEC

Positioning

  • Tensor throughput between RTX 5070 and RTX 5070 Ti
  • SM12x variant (consumer Blackwell)
  • Lacks data center SM100 features
  • Separate code paths needed for B200/B300

Compute Capability Divergence

Data center Blackwell (SM100) includes TMEM, tcgen05, and 2-SM cooperative MMA not present in the Spark's SM12x. Kernel developers targeting both platforms must maintain separate code paths, though Tile IR and cuTile aim to bridge this gap.

MEMORY

Unified Memory Architecture

128 GB LPDDR5x

  • 8 memory chips, 16 channels
  • 256-bit interface at 8533 MHz
  • 273 GB/s bandwidth (shared CPU + GPU)
  • Coherent address space via NVLink-C2C

Inference Advantage

  • No system-to-VRAM transfer bottleneck
  • Entire pool available to GPU without partitioning
  • Runs models up to 200B parameters
  • Far exceeds any single consumer GPU's VRAM

Contrast: Discrete GPU Setup

An RTX 5090 has 32 GB GDDR7 at 1,792 GB/s, but data exceeding VRAM must swap across PCIe at ~64 GB/s. The DGX Spark eliminates this cliff: the entire 128 GB is GPU-accessible at full memory bandwidth, enabling much larger models despite lower peak bandwidth.

The Bottleneck

Autoregressive token generation streams full model weights per token. At 273 GB/s, this constrains single-user tokens/sec for large models.

The Sweet Spot

DGX Spark excels at loading very large models at modest per-user throughput, and at batched inference where bandwidth cost is amortized across requests.

PORTABILITY

Tile IR & cuTile

The Problem

  • DGX Spark GPU (SM12x) differs from data center Blackwell (SM100)
  • Kernels optimized on Spark need rewriting for B200/B300
  • Teams face friction prototyping locally then deploying to cloud

The Solution (GTC 2026)

  • Tile IR: portable intermediate representation for GPU kernels
  • cuTile: Python DSL for writing kernels once
  • TileGym: pre-optimized transformer kernels
  • Write on Spark, deploy to B200/B300 with minimal changes

Roofline Scaling

Roofline analysis shows kernels scale effectively relative to each platform's theoretical peak. Optimizations made locally on DGX Spark translate meaningfully to cloud deployments, validating the desktop-to-data-center promise.

BENCHMARKS

Inference Performance Profile

Strengths

  • DeepSeek-R1 14B (FP8, batch 8): 2,074 total tok/s
  • Per-user throughput: ~83.5 tok/s
  • No thermal throttling under sustained load
  • Excels with smaller models + batching

Bottlenecks

  • Large models (70B+) constrained by 273 GB/s bandwidth
  • Llama 3.1 70B, GPT-OSS 120B: prototyping-tier speed
  • Memory-bandwidth-bound for autoregressive generation
  • Not suited for production serving of large models

Performance Strategy

The system favors memory capacity over memory throughput. It is well-suited for loading very large models at modest per-user throughput, and batching amortizes bandwidth costs across concurrent requests to maximize total throughput.

FP4 & NVFP4 Support

NVFP4 Format

  • Native hardware support in Blackwell Tensor Cores
  • Near-FP8 accuracy with <1% degradation
  • Halves memory footprint vs. FP8

Impact on DGX Spark

  • Directly reduces bytes-per-token through memory subsystem
  • Nemotron Nano 2 in NVFP4: 2× throughput vs. FP8
  • Especially valuable for bandwidth-constrained system
  • Negligible accuracy loss in practice

LIMITATIONS

Limitations & Considerations

Hardware Constraints

  • 273 GB/s bandwidth limits single-user token gen for large models
  • Cannot match per-request latency of dedicated GPU servers
  • ConnectX-7 adds idle power (~35W) and cost even for standalone users
  • GPU compute capability differs from data center Blackwell

Software & Platform

  • DGX OS only — no Windows, no alternative OS
  • Arm64 architecture: not all x86 Linux software available
  • Some Docker containers need ARM-compatible versions
  • Optimized kernels for B200/B300 require adaptation

Mitigations

NGC provides ARM64-native containers for common AI workflows. The Arm ecosystem has improved rapidly since launch. Tile IR / cuTile bridge the SM12x-to-SM100 kernel gap. FP4 quantization partially compensates for bandwidth limits by reducing bytes-per-token.
