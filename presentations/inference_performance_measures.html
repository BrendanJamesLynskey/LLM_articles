OVERVIEW

Why one number is never enough

The scalar trap

  • "Tokens per second" hides batch size, concurrency, input length, and tail behaviour.
  • Two systems with identical peak tok/s can feel radically different to users.
  • Performance is at least two-dimensional: per-user speed × concurrency.

What we need to separate

  • User-facing latency vs operator-facing throughput.
  • Prefill vs decode — different phases, different bottlenecks.
  • Average behaviour vs tails (p50 vs p99 vs p99.9).
  • Idle benchmarks vs loaded production.

BACKGROUND

Two phases, two cost models

Prefill

  • All input tokens processed in a single parallel forward pass.
  • Compute-bound; scales with prompt length.
  • 4K-token prompt on H100: 50–200 ms depending on model / TP.
  • Target for TTFT optimization.

Decode

  • Output tokens generated autoregressively, one at a time.
  • Memory-bandwidth-bound; weights reloaded per step.
  • Per-request decode rate is flat in batch size — until arithmetic intensity flips.
  • Target for TPOT / interactivity optimization.

USER-FACING

TTFT: Time to First Token

What TTFT measures

  • Wall-clock from request submission to first response token on the wire.
  • Includes network ingress, admission queue, tokenization, prefill, first decode, egress framing.
  • The latency users feel before anything happens.

Typical targets

  • Chat / streaming assistant: < 500 ms.
  • IDE autocomplete: < 200 ms.
  • Voice agent (first audio): < 300 ms.
  • p99 TTFT is dominated by queueing, not compute.

USER-FACING

TPOT, ITL, and interactivity

TPOT / inter-token latency

  • TPOT = average time between successive output tokens once streaming starts.
  • ITL = same quantity reported per step rather than averaged.
  • 25 ms TPOT ≈ 40 tok/s — reads faster than most humans.
  • Rises with batch size — batching interference.

Interactivity

  • Per-user tok/s sustained under production concurrency.
  • 100 tok/s at batch=1 but 12 tok/s at batch=32 = poor interactivity.
  • TTFT is the intercept; interactivity is the slope.
  • Keep both low/steep — and hold them as load grows.

USER-FACING

End-to-end latency and jitter

End-to-end

  • E2E ≈ TTFT + (N − 1) × TPOT for an N-token response.
  • The right metric for non-streaming JSON APIs.
  • The only metric for an agent waiting to parse a tool-call argument.

Stream smoothness

  • Two systems, same TPOT: steady 40 ms cadence vs bursts of 8 then 320 ms pause.
  • Measure as std(ITL) or P(ITL > 200 ms) < 1%.
  • Stall rate: fraction of requests with a mid-stream gap past SLA.
  • Preemption and batch re-composition are the usual culprits.

OPERATOR

Throughput measures

RPS and tokens/sec

  • RPS is ambiguous alone — 50/50 request ≠ 100K/2K request.
  • Input tokens/sec: prefill throughput (long prompts, RAG).
  • Output tokens/sec: decode throughput (long generations, reasoning).
  • Report them separately — different bottlenecks, different fixes.

Goodput

  • Raw throughput counts every token, including the ones the user no longer wants.
  • Goodput = tokens delivered while the SLA was being met.
  • Bigger batches can double raw throughput and halve goodput.
  • Goodput is the operator metric that aligns with user experience.

OPERATOR

Utilization — honest and dishonest

nvidia-smi lies

  • "GPU utilization" just reports whether any kernel is resident.
  • A decode loop saturating HBM can show 99% GPU util with 5% MFU.
  • It is a liveness signal, not an efficiency one.

Better signals

  • MFU — achieved / peak FLOPs. Prefill: 40–60%. Decode: 5–15%.
  • HBM bandwidth utilization — the right metric for decode. 60–85% in well-tuned stacks.
  • Concurrency distribution — matters as much as latency percentiles.

TAILS

Percentiles and why p99.9 matters

PercentileWhat it capturesWho reads it
p50Typical happy pathCapacity sizing
p95Bad-day experienceMost customer SLAs
p99Near-worst-caseTight interactive SLAs, voice agents
p99.9Tail events that compoundMulti-step agents

Agent tail amplification

  • 10-step agent, per-step p99 of 2 s → P(any step ≥ p99) ≈ 10%.
  • Tighten p99.9, not p50, if you run agents.

Report context

  • "p99 = 800 ms" is meaningless alone.
  • Always cite window, load, and input-length distribution.

PITFALLS

Common mis-measurements

What to avoid

  • Benchmarking on an idle machine.
  • Using average latency instead of percentiles.
  • Conflating input and output token counts.
  • Reporting TTFT without p99.
  • Quoting single-user tok/s as "interactivity".

What to do instead

  • Measure at production concurrency.
  • Report p50, p95, p99 — and p99.9 for agents.
  • Separate prefill throughput from decode throughput.
  • Always pair latency with the load level.
  • Report tok/s/user sustained under N concurrent users.

CONCLUSION

From per-call to end-to-end

What we've established

  • No scalar captures inference performance.
  • User-facing and operator-facing metrics are coupled but not equivalent.
  • Tails matter more than means — especially for agents.
  • Goodput is the metric that aligns everyone.

What comes next

  • Each measure here is a per-call property of the serving layer.
  • An agent composes many such calls with tools, memory, and network in between.
  • See the companion article — Agentic Performance — for how these measures compose and what couples to CPU, network, and storage.
