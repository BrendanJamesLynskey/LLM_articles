CONTEXT

The Kernel Bottleneck

For most of CUDA’s history, writing high-performance GPU kernels needed deep expertise: thread blocks, shared memory, register pressure, coalescing, bank conflicts. The gap between expert and naive CUDA was routinely 10×.

The Old World

  • CUDA C++ for serious kernels
  • CUTLASS templates — tens of thousands of LoC
  • Months to write a new fused operation
  • Researchers blocked on CUDA expertise

What Triton Offers

  • Python-embedded DSL
  • Block-level programming model
  • Compiler handles most low-level details
  • 50 lines of Python ≈ CUTLASS matmul perf

MODEL

Block-Level Programming

No threads. No shared memory. No warp shuffles. The programmer writes code that operates on tiles of data. The compiler handles the rest.

What You Specify

  • BLOCK_SIZE for each tile dimension
  • Per-instance grid identifier
  • Load/store offsets within the block
  • The actual math

What Compiler Decides

  • Threads per block, warps per SM
  • Shared memory layout
  • Pipelining and double-buffering
  • Tensor core dispatch
  • Coalescing

Vector add in Triton:

@triton.jit
def add_kernel(x_ptr, y_ptr, out_ptr, n,
               BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(axis=0)
    block_start = pid * BLOCK_SIZE
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    tl.store(out_ptr + offsets, x + y, mask=mask)

EXAMPLE

Matmul in Triton

Notable Features

  • Operates on (BLOCK_M, BLOCK_K) and (BLOCK_K, BLOCK_N) tiles
  • tl.dot emits tensor core mma
  • Loop is auto-pipelined by compiler
  • Accumulator stays in registers across iterations
  • Within ~10% of CUTLASS performance
  • ~50 lines vs thousands for CUTLASS
@triton.jit
def matmul(a_ptr, b_ptr, c_ptr, M, N, K,
           sa_m, sa_k, sb_k, sb_n, sc_m, sc_n,
           BM: tl.constexpr, BN: tl.constexpr,
           BK: tl.constexpr):
  pm = tl.program_id(0)
  pn = tl.program_id(1)
  offs_m = pm * BM + tl.arange(0, BM)
  offs_n = pn * BN + tl.arange(0, BN)
  offs_k = tl.arange(0, BK)
  a_p = a_ptr + offs_m[:,None]*sa_m + offs_k[None,:]*sa_k
  b_p = b_ptr + offs_k[:,None]*sb_k + offs_n[None,:]*sb_n
  acc = tl.zeros((BM, BN), dtype=tl.float32)
  for k in range(0, K, BK):
    a = tl.load(a_p)
    b = tl.load(b_p)
    acc += tl.dot(a, b)
    a_p += BK * sa_k
    b_p += BK * sb_k
  c_p = c_ptr + offs_m[:,None]*sc_m + offs_n[None,:]*sc_n
  tl.store(c_p, acc)

Practical Lesson

For new kernels, Triton is almost always the right starting point. Only drop to CUDA when you need the last 5–10% on a heavily-used kernel and you have the engineering budget.

triton / 07
LIMITS

Where Triton Falls Short

Bleeding-Edge HW Features

  • TMA on Hopper (still maturing)
  • Warp specialization patterns
  • FP4 tensor cores on Blackwell
  • Compiler catches up but with a lag

Irregular Patterns

  • Sparse / ragged tensors awkward
  • Data-dependent control flow tricky
  • Dynamic shapes still rough
  • Better in CUDA for these

JIT Compile Time

  • Several seconds first-call overhead
  • Cache helps but invalidation is messy
  • Annoying for short scripts

Hardware Quirks

  • Same source, different perf NVIDIA vs AMD
  • Tuning still needed cross-vendor
  • Portability is structural, not perf-equivalent

SUMMARY

Why Triton Won the Researcher Mindshare

The Pitch

  • 10× less code than equivalent CUDA
  • Within a few percent of expert CUDA performance
  • Cross-vendor portability without source changes
  • Compiler improves over time — old kernels get faster
  • Transparent benefit through torch.compile

Practitioner’s Lesson

If you find yourself wanting to write a custom GPU kernel for an ML workload, start with Triton. Only drop to CUDA when you have a specific reason. The productivity gain is enormous, and the resulting code is dramatically more maintainable.
