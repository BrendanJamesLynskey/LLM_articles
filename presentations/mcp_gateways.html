PROBLEM

Why MCP Gateways Are Needed

Direct Connection at Scale

  • 20 agents × 30 servers = 600 connections
  • Each needs separate auth, credentials, monitoring
  • Adding a server means updating every agent
  • Rotating a credential means touching every client
  • No single place to answer “who calls what, how often?”

With a Gateway

  • Agents connect to one endpoint
  • Gateway manages all upstream servers
  • Centralized auth, rate limits, audit logging
  • Unified tool catalog with access control
  • Add/remove servers without touching clients

Security Gap

MCP spec left auth, rate limiting, audit logging to implementations. Most servers omit them. A gateway provides a single enforcement point.

Context Window Problem

Naive aggregation of all tools wastes 30–50% of context tokens on tool definitions. Gateways curate and filter the tool catalog per user/task.

ARCHITECTURE

Five Architecture Patterns

Centralized Reverse Proxy

  • Single gateway all clients connect to
  • Aggregates tool catalogs from all servers
  • Most common pattern
  • Microsoft, Docker, IBM, Traefik, Kong

Sidecar Proxy

  • Lightweight proxy per MCP server
  • Natural for Kubernetes environments
  • Isolation: misconfigured proxy affects one server
  • Kuadrant (Red Hat / Envoy-based)

Mesh

  • Sidecar + inter-gateway communication
  • Analogous to a service mesh
  • Cross-server discovery and tracing
  • agentgateway (Linux Foundation, Rust)

Micro-MCP

  • Decompose large servers into single-purpose servers
  • One tool per server, composed behind gateway
  • Granular scaling and fault isolation
  • Inspired by microservices

Dual-Plane

  • Data plane: runtime request routing with session affinity
  • Control plane: lifecycle management via REST APIs
  • Mirrors Kubernetes networking
  • Microsoft MCP Gateway

PROTOCOL

JSON-RPC Message Interception

How It Works

  • Gateway parses every JSON-RPC 2.0 message
  • Inspects method, tool name, and arguments
  • Applies routing, authorization, and logging decisions
  • Automatically permits initialize and notifications/initialized

List Response Filtering

  • Intercepts tools/list responses from upstream
  • Filters tools based on client identity
  • Developers see dev tools; analysts see analytics tools
  • Improves security and reduces context bloat
# Traefik Hub policy example
policies:
  - match:
      Equals(`mcp.method`,
        `tools/call`) &&
      Equals(`mcp.params.name`,
        `get_weather`) &&
      Contains(`jwt.groups`,
        `weather-users`)
    action: allow

  - match:
      Equals(`mcp.method`,
        `tools/call`) &&
      Prefix(`mcp.params.name`,
        `admin_`) &&
      !Contains(`jwt.groups`,
        `admin`)
    action: deny

defaultAction: deny

TRANSPORT

Transport Bridging

stdio

  • Local servers as child processes
  • JSON-RPC over stdin/stdout
  • Zero network overhead
  • Inherently local only

HTTP+SSE (deprecated)

  • POST for client → server
  • SSE for server → client
  • Two separate connections
  • Still widely deployed

Streamable HTTP

  • Standard remote transport (June 2025)
  • POST with optional SSE response
  • GET for server-initiated notifications
  • Mcp-Session-Id header for sessions

Gateway as Transport Translator

A critical gateway function is converting stdio-based local servers into remotely accessible Streamable HTTP endpoints. The gateway spawns the server as a child process, wraps stdin/stdout with JSON-RPC framing, and exposes it over HTTP. This makes any existing MCP server available to remote agents without code changes. Docker MCP Gateway launches servers in containers, communicates via stdio internally, and exposes them through its gateway endpoint.

PROTOCOL

Session Management

The Challenge

  • MCP is stateful — unlike REST APIs
  • Initialization handshake establishes capabilities
  • Tool calls depend on session context
  • All requests in a session must route to the same server instance
  • Standard HTTP load balancers cannot handle this

Gateway Solutions

  • Microsoft: Kubernetes StatefulSets + headless services for session-aware routing
  • Traefik Hub: HRW consistent hashing — deterministic client-to-server routing by IP, automatic failover
  • Envoy AI Gateway: Full Streamable HTTP with Last-Event-ID reconnection and multiplexed streams
  • Gateway manages session lifecycle: expiry detection, state cleanup, reconnection

DISCOVERY

Tool Registry & Aggregation

Dynamic Registration

  • Gateway maintains live registry via periodic tools/list
  • Refreshes on notifications/tools/list_changed
  • Cached aggregation (e.g., 60s TTL) with notification-based invalidation
  • Never cache tools/call — side-effecting operations

Namespace Conflict Resolution

  • Two servers both expose create_issue
  • Gateway auto-prefixes: github_create_issue, jira_create_issue
  • Stacklok allows custom prefix names
  • Microsoft routes via Tool Gateway Router

Context Window Optimization

  • Selective exposure: cherry-pick tools per user/team
  • Progressive disclosure: show categories first, reveal specific tools on demand (AIRIS claims 97% token reduction)
  • Meta-tool abstraction: LLM writes code to orchestrate tools in sandbox (50%+ token savings)
  • Profile-based filtering: Docker uses --profile flag to control tool access

SECURITY

Authentication & Authorization

The Multi-Hop Challenge

User → AI Host → MCP Client → Gateway → MCP Server → Downstream API. Each hop raises authorization questions. The gateway provides a single enforcement point.

Kuadrant Three-Layer Auth

  • Layer 1: Identity-based tool filtering — validate OAuth2 token, extract roles, create signed JWT “wristband” with allowed tools
  • Layer 2: Token Exchange (RFC 8693) — swap broad token for narrowly-scoped per-server token
  • Layer 3: Vault Integration — fetch PATs/API keys from HashiCorp Vault per user per server

Credential Vaulting

  • Raw credentials never reach the client or LLM
  • Gateway injects credentials at execution time
  • Agents receive only scoped, time-limited tokens
  • Prevents credential leakage via prompts or logs
  • Peta: “1Password for AI Agents”

Sequence-Aware Authorization

  • Individual actions may be authorized…
  • …but chained together by a compromised agent, they become attacks
  • Read file + send email = data exfiltration
  • Gateway evaluates accumulated operation sequence

SECURITY

Threats & Gateway Defenses

Prompt Injection at the Gateway

  • Malicious instructions in tool outputs hijack the LLM
  • Lasso Gateway: plugin-based guardrails
  • Presidio plugin: PII detection and masking
  • Injection detection: analyzes prompts and outputs
  • Secret masking: 12 built-in patterns for API keys
  • Sanitize tool outputs before they reach the model

Tool Poisoning

  • Direct tampering: modified tool descriptions with hidden instructions
  • Shadowing: malicious tool mimicking a legitimate name
  • Rug pulls: normal during eval, malicious later
  • Gateway defense: registration approval workflows, signed version-locked tool definitions

Triple-Gate Security Pattern

  • Gate 1 — Client ↔ LLM: prompt injection filtering, PII detection, input sanitization
  • Gate 2 — LLM ↔ MCP Server: tool authorization, parameter validation, argument sanitization (the gateway’s natural position)
  • Gate 3 — MCP Server ↔ External API: rate limiting, auth enforcement, response filtering

Centralized Logging

  • Single point for all MCP interactions
  • Structured fields: mcp.method, mcp.session.id, mcp.target, mcp.resource.name
  • Integrates with ELK, Prometheus, Grafana
  • EU AI Act compliance: tamper-proof audit trails

Distributed Tracing

  • OpenTelemetry with W3C Trace Context
  • Parent span: agent request
  • Child spans: each tool call
  • Grandchild spans: downstream API calls
  • End-to-end visibility: prompt → reasoning → tools
  • Traefik: mcp.client.operation.duration histograms

Key Metrics

  • Tool invocation count by server and tool
  • Latency: p50, p95, p99 per tool
  • Error rates with failure mode labels
  • Token usage: input/output per call
  • Session metrics: active count, duration, errors

PERFORMANCE

Rate Limiting & Caching

Multi-Dimensional Rate Limiting

  • Per-user: prevent monopolization
  • Per-tool: respect downstream API limits
  • Per-team: fair resource allocation
  • Per-agent: constrain autonomous agents
  • In-memory enforcement: sub-millisecond overhead
  • Zapier: 80/hr, 160/day, 300/month reflecting external constraints

Layered Caching

  • In-memory: fast per-instance cache for tool lists
  • Distributed: Redis/Memcached for multi-instance
  • TTL-based: time-to-live expiration
  • Notification-based: refresh on server change signals
  • Safe to cache: tools/list, resources/list, prompts/list
  • Never cache: tools/call (side effects!)

Load Balancing & Resilience

Round-robin, least-connections, or consistent-hashing with session affinity. Circuit breakers detect unhealthy servers. Retry logic with restrictions on side-effecting operations. Kong: health checks + circuit breakers + session-aware balancing.

COMPARISON

MCP Gateway vs API Gateway

AspectTraditional API GatewayMCP Gateway
Session modelStateless HTTP request/responseStateful sessions with context across calls
DiscoveryRoute to known endpointsAggregate tool catalogs, resolve conflicts, filter by auth
CommunicationUnidirectional (client → server)Bidirectional (server push, notifications, sampling)
Protocol awarenessURL-based routingJSON-RPC body parsing — method, tool, arguments
Context managementNot applicableContext-window aware — token budgets, tool filtering
Load balancingRound-robin, least-connectionsSession-affinity required (StatefulSets, HRW hashing)

Can You Adapt an Existing API Gateway?

Kong (AI MCP Proxy plugin), Traefik Hub, and Apache APISIX have added MCP support. Basic proxying works, but deep features (session management, tool aggregation, context-window optimization) require significant MCP-specific code regardless of starting point.

CHALLENGES

Open Challenges

Performance Overhead

  • Every request adds latency
  • Simple policy: sub-millisecond
  • Token exchange + Vault + PII scan: meaningful latency
  • Compounds over dozens of sequential tool calls

Specification Gaps

  • MCP spec does not formally define gateway behavior
  • Each gateway implements its own proxy semantics
  • Client behavior varies behind different gateways
  • Migration between gateways is nontrivial

Context Window Tension

  • Comprehensive tools vs. context budget
  • Progressive disclosure adds complexity
  • No protocol mechanism for budget constraints
  • No standard for clients to request tool subsets

Debugging Complexity

  • Failures can originate in client, gateway, server, or downstream
  • Tracing helps but gateway debug tools are immature
  • Traefik returns HTTP 403 — Claude Desktop sees broken session

Configuration Portability

  • Every gateway has its own config format
  • K8s CRDs, YAML, JSON, web consoles
  • No standard for “this group gets these tools at this rate”
  • MCP roadmap mentions portability as a goal

GUIDANCE

Deployment Recommendations

Start Early

  • Implement a gateway before production deployment
  • Retrofitting is substantially harder
  • Centralize credentials and audit logging from day one
  • Architect for multi-tenancy even with a single team

Choose Wisely

  • Match security model to regulatory requirements
  • Existing API gateway? Consider MCP plugin (Kong, Traefik)
  • Greenfield? Purpose-built may be worth it
  • Kubernetes? Stacklok CRDs or Microsoft dual-plane
  • Security-first? Lasso, Peta, or Kuadrant

CONCLUSION

Key Takeaways

What MCP Gateways Provide

  • Centralized security: auth, credential vaulting, prompt injection defense
  • Unified tool catalog: aggregation, conflict resolution, context optimization
  • Enterprise observability: OTel tracing, structured logging, compliance audit trails
  • Operational efficiency: rate limiting, caching, load balancing, transport bridging
  • Multi-tenancy: isolation, RBAC, per-tenant visibility

The Landscape

  • 40+ implementations across open-source and commercial
  • API vendors extending (Kong, Traefik, Envoy) and MCP-native building from scratch
  • MCP spec catching up: gateway patterns, enterprise auth, registry planned for 2026
  • Just as microservices needed API gateways, MCP servers need MCP Gateways

Bottom Line

MCP Gateways represent the natural maturation of the MCP ecosystem from developer-scale experimentation to enterprise-scale production. They provide the security, observability, and governance infrastructure that production agent deployments require — implement one early, centralize from the start, and choose a gateway whose security model matches your regulatory requirements.
