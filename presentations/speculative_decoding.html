CONTEXT

The Autoregressive Bottleneck

Why Decoding Is Slow

  • Each token requires a full forward pass
  • Strictly sequential: token N+1 depends on token N
  • GPU reads full weight matrix but does minimal arithmetic
  • Decode is memory-bandwidth-bound, not compute-bound

Distributed Penalty

  • Each step triggers 2L all-reduce operations (L = layers)
  • Tensor parallelism adds collective communication
  • Pipeline parallelism adds point-to-point transfers
  • Communication can consume 60–70% of decode time on PCIe

INSIGHT

Prefill vs. Decode Asymmetry

Prefill Phase

  • Processes many tokens in parallel
  • Compute-bound — GPU fully utilized
  • Cost of K tokens ≈ cost of 1 token

The Speculative Idea

  • If K candidate tokens can be verified simultaneously, cost is near one forward pass
  • Draft cheaply with a small model, verify in bulk with the target
  • Result: 2–3x fewer target-model forward passes

ALGORITHM

The Draft-Then-Verify Loop

1. Draft

Small model generates K candidate tokens autoregressively. Typically 10–50x fewer parameters than target.

2. Verify

Target model runs a single forward pass over all K candidates in parallel, producing distributions at each position.

3. Accept / Reject

Tokens accepted left-to-right via stochastic criterion. First rejection triggers resample; subsequent tokens discarded.

Key Property

When the draft is correct, multiple tokens advance in one target call. When wrong, the system falls back gracefully with no quality loss.

speculative-decoding / 04
MATH

The Acceptance Criterion

Stochastic Acceptance Rule

For each position i, accept the draft token with probability:

P(accept) = min(1, p_target(x_i) / p_draft(x_i))

If rejected, sample corrected token from a modified distribution that accounts for the rejected mass.

Lossless Guarantee

The output distribution is provably identical to what the target model would produce on its own. No quality degradation.

Intuition

When p_target ≥ p_draft, always accept. When p_target < p_draft, accept proportionally. The draft over-predicts tokens the target would not favor.

STRATEGIES

Draft Model Approaches

Independent Small Model

  • Separate smaller model from same family
  • E.g., Llama 3 8B drafting for Llama 3 70B
  • Requires matching vocabulary & tokenizer
  • Runs on dedicated GPU

Self-Speculative Decoding

  • Target model in degraded mode
  • Layer skipping, early exit, fewer attention heads
  • No second model to load — saves memory
  • Acceptance rate: 0.7–0.85

Medusa (Multi-Head)

  • Lightweight prediction heads on target model
  • Predict positions +1, +2, +3 in parallel
  • No autoregressive draft steps needed
  • 2–3x speedup, negligible memory

N-gram / Prompt Lookup

  • Match n-grams from the input prompt
  • Zero additional parameters or compute
  • Ideal for copy-heavy tasks (code edit, reformat)
  • Acceptance rate can exceed 0.9

STRATEGY 1

Independent Draft Models

The most straightforward approach: pair a small model from the same family with the large target.

  • Draft model is 10–50x smaller than target
  • Runs on a single small GPU
  • Vocabulary and tokenizer must match for valid token-level comparison
  • Can be pipelined with target verification
speculative-decoding / 08
STRATEGY 2

Medusa: Multi-Head Drafting

  • Adds small prediction heads on top of the target model
  • Each head predicts a future token position (+1, +2, +3, ...)
  • Heads are lightweight: a few linear layers each
  • Generates candidate tree of continuations without autoregressive steps
  • Target verifies tree in a single pass using tree attention

Why Medusa?

Negligible memory overhead. Ideal when no spare GPU is available for a separate draft model.

STRATEGY 3

Prompt Lookup & N-gram Drafting

How It Works

  • Look up n-gram matches in the prompt context
  • Zero parameters, zero compute for drafting
  • Acceptance rate highly workload-dependent

Ideal Workloads

  • Code editing & refactoring
  • Document reformatting
  • Extractive summarization
  • Acceptance rate can exceed > 0.9

Example: Code Editing

// Input prompt contains:

  return a + b;

Subsequent tokens matched directly from prompt context — near-free drafting.

DISTRIBUTED

Communication Reduction

Without Speculative Decoding

  • Each decode step: 2L all-reduce ops
  • 70B on 8 GPUs via PCIe
  • Communication = 60–70% of decode time
  • 100 tokens = 100 rounds of sync

With Speculative Decoding

  • Accept avg. 3 tokens per step
  • All-reduce rounds drop by ∼3x
  • 100 tokens ≈ 33 rounds of sync
  • Substantial wall-clock improvement

DEPLOYMENT

Draft Model Placement

Dedicated GPU

Draft runs on a separate GPU outside the tensor-parallel group. Allows pipelining draft generation with target verification.

Shared GPU

Draft shares a GPU with one shard of the target. Runs during idle intervals (e.g., pipeline bubbles). No extra hardware needed.

Self-Draft / Medusa

Avoids placement question entirely. No second model. Draft via layer skip or lightweight heads on the target itself.

SERVING

Batch Interaction & Continuous Batching

  • Different sequences have different acceptance rates
  • Sequences advance by varying numbers of tokens per step
  • Frameworks (vLLM, TensorRT-LLM) handle via padding + masking
  • Efficiency improves at smaller batch sizes where decode is most memory-bandwidth-bound

Practical Implication

Speculative decoding is most beneficial for latency-sensitive, low-batch workloads. At high batch sizes, the decode phase becomes more compute-bound, reducing the benefit.

Key Takeaway

Draft quality matters more than draft speed. A slightly slower draft with higher acceptance rate often outperforms a faster but less accurate one, because each rejected token wastes a target-model verification slot.

ADVANCED

Tree-Based Verification

Linear vs. Tree Verification

  • Linear: verify a single sequence of K tokens
  • Tree: verify a tree of candidate continuations
  • Increases probability of accepting deeper paths
  • Cost: wider (but still single) forward pass

Benefit

Even if individual branches have moderate acceptance rates, the probability that at least one path extends far is much higher with tree search.

SYNERGY

Composing with Other Optimizations

Quantization

  • Draft model can be aggressively quantized (INT4)
  • Predictions only need to be approximately correct
  • Combined with quantized target: reduces memory + communication

PagedAttention

  • Efficient KV cache management
  • Speculative tokens share paged blocks
  • Rejected tokens: discard pages, no fragmentation

Continuous Batching

  • New requests slot in during verify steps
  • Spec decoding composes naturally with dynamic scheduling
  • Supported in vLLM, TensorRT-LLM

KV Cache Compression

  • Reduces memory per sequence
  • More room for speculative candidates in cache
  • Allows larger K values

SUMMARY

Key Takeaways

Core Benefits

  • 2–3x reduction in target-model forward passes
  • Lossless — output distribution provably unchanged
  • No retraining of target model required (for independent draft / n-gram)
  • Directly addresses the memory-bandwidth bottleneck

When to Use

  • Deploying large models on resource-constrained hardware
  • Latency-sensitive, low-batch serving
  • Distributed systems where communication dominates
  • Composes with quantization, PagedAttention, continuous batching

Bottom Line

For any practitioner deploying large models on resource-constrained hardware, speculative decoding should be among the first techniques evaluated. It is one of the most impactful inference optimizations available for distributed small-GPU systems.
