Why Batching Matters

The Core Insight

GPUs are designed for massively parallel computation. A single inference request leaves most compute capacity idle. Batching amortizes fixed overheads — memory transfers, kernel launches — across many requests.

Memory-Bound Problem

Autoregressive decoding reads the full model weights every forward pass but performs little arithmetic per byte loaded. Batching increases compute per memory access, shifting toward a compute-bound regime.

Two Phases of LLM Inference

Prefill Phase

  • Processes all input tokens in parallel
  • Compute-intensive — resembles a training forward pass
  • Benefits from high-FLOPS hardware
  • Also called "prompt processing"

Decode Phase

  • Generates tokens one at a time autoregressively
  • Memory-bandwidth bound — reads entire model weights per token
  • Benefits from high-bandwidth memory
  • Dominant cost in long-generation tasks

Key Takeaway

Effective batching strategies must account for these fundamentally different computational characteristics. Mixing prefill and decode in the same batch can cause interference.

Static (Naive) Batching

The simplest strategy: collect a fixed number of requests and process them together. All sequences begin and end processing together.

Limitations

  • Shorter sequences must be padded to match the longest
  • Entire batch blocked until the longest sequence completes
  • GPU resources idle as individual sequences finish
  • Poor memory efficiency due to over-allocation

Continuous Batching

Introduced by Orca, adopted by vLLM and TensorRT-LLM. Sequences enter and leave the batch dynamically at each iteration.

Advantages

  • No wasted cycles on completed sequences
  • New requests fill slots immediately
  • Dramatically improved GPU utilization
  • De facto standard in production serving

Disaggregated Prefill & Decode

Separate prefill and decode onto different hardware pools, each optimized for its workload profile.

Prefill Pool

  • Compute-heavy → high-FLOPS accelerators
  • Large batch of prompt tokens processed in parallel
  • Optimized for throughput

Decode Pool

  • Memory-bandwidth bound → high-HBM configs
  • Latency-sensitive token generation
  • Optimized for low latency

Systems: Splitwise & DistServe

By disaggregating, each phase can be batched and optimized independently, avoiding interference between compute-heavy prefill and latency-sensitive decode operations. Demonstrated significant efficiency gains in practice.

Speculative Decoding

How It Works

Converts multiple sequential decode steps into a single parallel verification step. Incorrect tokens are rejected; the system falls back to the target model's distribution.

Best When

The draft model has high acceptance rates. Particularly effective for predictable text patterns, code generation, and structured outputs.

PagedAttention & Memory Management

Introduced by vLLM, PagedAttention borrows from OS virtual memory to manage the KV cache.

Traditional Approach

Pre-allocates contiguous memory for each sequence's maximum possible length → severe fragmentation and waste.

PagedAttention

  • Stores KV cache in non-contiguous pages
  • Allocates memory on demand
  • Shares pages across sequences (e.g., common system prompts)
  • Enables much larger effective batch sizes

Dynamic Scheduling & Priority Queues

Scheduling Factors

  • Arrival time
  • Priority level
  • Estimated generation length
  • Current system load

Balancing Act

  • Throughput: process as many requests as possible
  • Latency: keep response times low per request
  • These goals are often in tension

Preemption

  • Low-priority or long-running requests can be temporarily evicted
  • Makes room for higher-priority requests
  • Requires saving/restoring KV cache state

Quantization & Reduced-Precision Batching

Reducing precision from FP16/BF16 to INT8, INT4, or FP8 shrinks memory requirements and increases effective memory bandwidth.

Impact on Batching

  • Smaller model footprint → more KV cache space
  • More sequences fit in memory simultaneously
  • Directly increases achievable batch sizes
  • Particularly impactful for decode phase

Production inference engines combine quantization with continuous batching to maximize throughput on a given hardware budget. The memory-bandwidth bottleneck during decode is directly alleviated.

Application: API Serving & Cloud Inference

Production Stack

Applications: Offline & Edge Inference

Offline / Batch Processing

  • Document summarization, extraction, classification
  • Latency less critical than throughput & cost
  • Requests sorted by length to minimize padding
  • Large static/semi-static batches maximize utilization
  • Providers offer cost discounts (e.g., Anthropic's Message Batches API)

Edge / On-Device

  • Limited memory and compute
  • Concurrent requests are rare on a single device
  • Internal batching: speculative decoding, chunked prefill
  • Batch speculative tokens or prompt chunks to improve single-user latency
  • Better time-to-first-token and tokens/sec

Challenges & Trade-Offs

Memory Pressure

  • Larger batches need more GPU memory for KV caches
  • Tension between batch size and max sequence length
  • Requires PagedAttention or KV cache eviction

Latency vs Throughput

  • Requests may wait for batch assembly
  • Shared compute resources across many sequences
  • Individual request latency can increase when optimizing for throughput

Fairness & QoS

  • Multi-tenant environments with different SLAs
  • High-priority requests must not be starved by batch workloads
  • Priority scheduling & admission control required

System Complexity

Interaction between batching and tensor parallelism, pipeline parallelism, and expert parallelism (MoE models) adds significant engineering complexity to production systems.

Future Directions

MoE Architectures

  • Different tokens routed to different expert subnetworks
  • Requires load-balancing across experts
  • Managing sparse compute patterns
  • New batching challenges and opportunities

Hardware Advances

  • HBM3E and beyond — higher bandwidth memory
  • Increased on-chip SRAM
  • Dedicated inference accelerators
  • Shifts compute-vs-bandwidth balance → larger batches

Software Innovations

  • Prefix caching for shared prompt prefixes
  • Radix-tree-based KV cache sharing
  • Adaptive scheduling algorithms

Longer Context Windows

  • 4K → 128K+ tokens
  • Enormous pressure on KV cache memory
  • Makes memory-efficient batching essential

Key Takeaways

1. Batching is Foundational

Transforms underutilized, memory-bandwidth-bound GPU workloads into efficient, high-throughput serving systems.

2. Continuous Batching Wins

Dynamic slot filling has become the de facto standard in production, replacing wasteful static batching.

3. Memory is the Bottleneck

PagedAttention, quantization, and KV cache management are critical enablers for larger batch sizes.

4. Disaggregation Unlocks Scale

Separating prefill and decode phases onto specialized hardware pools avoids interference and maximizes efficiency.

5. Trade-Offs Are Real

Latency vs. throughput, memory vs. batch size, and fairness across tenants require careful engineering.

6. Evolution Continues

MoE models, longer contexts, and new hardware will keep pushing batching techniques forward.
