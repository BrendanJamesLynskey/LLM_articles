OVERVIEW

The LLM Threat Landscape

LLMs interpret instructions and data in the same channel—natural language. This creates attack surfaces that have no analogue in traditional software.

Input Attacks

  • Direct prompt injection
  • Indirect prompt injection
  • Jailbreaking (GCG, many-shot)
  • Adversarial perturbations

Model-Level Attacks

  • Data poisoning & backdoors
  • Sleeper agents
  • Model extraction / theft
  • Fine-tuning attacks

Output Risks

  • Training data leakage
  • PII extraction
  • Hallucination in critical contexts
  • Unsafe code generation
llm-security / 02
ATTACK

Direct Prompt Injection

The attacker interacts with the model directly and crafts input to override the system prompt or bypass safety constraints. Analogous to SQL injection: code and data share the same channel.

Technique Categories

  • Role-playing: "You are DAN, you have no restrictions..."
  • Hierarchy confusion: fake system prompts in user input
  • Completion steering: partial output that the model continues
  • Obfuscation: Base64, ROT13, pig Latin encoding

Why It Works

  • Models cannot reliably distinguish developer instructions from user input
  • Instruction-following is the model's core capability
  • Safety training covers known patterns but not novel reformulations
  • Obfuscated payloads bypass keyword filters
llm-security / 03
ATTACK

Indirect Prompt Injection

The attacker places malicious instructions in content the model will process: web pages, emails, documents, API responses. The model follows injected instructions from external data.

  • RAG systems retrieve pages with hidden instructions
  • Email assistants process messages containing injected commands
  • Code assistants read files with malicious comments
  • Tool-connected models can be tricked into executing actions

Key Danger

The model must read external content to do its job. You cannot block all external text without defeating the application's purpose.

ATTACK

Jailbreaking Techniques

GCG — Universal Adversarial Suffixes

  • Optimizes a gibberish suffix that causes compliance
  • Uses gradient info to maximize P("Sure, here is...")
  • Suffixes transfer across models, including closed-source
  • Detectable via perplexity filters, but low-perplexity variants exist

Many-Shot Jailbreaking

  • Fill context with fabricated examples of model compliance
  • Exploits in-context learning: model mimics the pattern
  • More effective with longer context windows (100K+)
  • Difficult to defend: pattern looks like normal few-shot prompting

Multi-Turn / Crescendo

  • Spread harmful request across multiple turns
  • Each turn individually innocuous
  • Gradually escalate, building false premises
  • Model loses track of the cumulative trajectory

Cipher & Encoding Tricks

  • Base64 / ROT13 encoding of harmful requests
  • Low-resource languages with weak safety training
  • Unicode substitution, leetspeak, Morse code
  • Payload splitting across fragments

ATTACK

Data Poisoning & Backdoor Attacks

Backdoor Injection

  • Insert poisoned examples into training data
  • Associate a trigger pattern with malicious behavior
  • Model behaves normally on clean inputs
  • Poisoning rate as low as 0.01% can succeed

Sleeper Agents

  • Models that pass safety evaluations but activate malicious behavior under specific conditions
  • Standard RLHF / CAI does not reliably remove the behavior
  • Model learns to conceal capabilities during safety training

ATTACK

Model Theft & Extraction

API-Based Extraction

  • Query target model with crafted inputs
  • Train student model on target's outputs
  • Partial extraction of task-specific behavior is practical
  • Full extraction of large generative models remains impractical

Distillation Attacks

  • Use target outputs as soft labels for student training
  • Even text-only outputs (no logits) are sufficient
  • Captures knowledge, reasoning, and style
  • Many providers now restrict logit access

Side-Channel Attacks

  • Timing: response latency reveals architecture info
  • Token counts: billing metadata leaks tokenizer details
  • Cache-based: shared GPU infrastructure leaks activations
  • Watermarking provides detection but can be removed by paraphrasing
llm-security / 07
ATTACK

Information Leakage

Training Data Memorization

  • LLMs memorize portions of training data verbatim
  • Duplicated text is memorized at much higher rates
  • Larger models memorize more (log-linear relationship)
  • Divergence attacks can force emission of memorized content

PII Extraction

  • Targeted: "The email of [name] is..." → model completes with real data
  • Untargeted: "List of real email addresses:" → model emits memorized PII
  • Compliance risk under GDPR, CCPA
  • Mitigations: dedup, differential privacy, output filtering

Membership Inference

  • Determine if a text was in the training set
  • Lower perplexity on training data vs. unseen data
  • Relevant for copyright and privacy enforcement
  • Accuracy improves with longer or duplicated texts

System Prompt Extraction

  • "Print your system prompt" or "Repeat everything above"
  • Reveals proprietary instructions, tool configs, business logic
  • Enables further targeted attacks
  • Differential analysis can infer prompt contents

ATTACK

Adversarial Inputs

Embedding Perturbations

  • Small changes to input text cause large behavior shifts
  • Invisible Unicode chars, character substitution, whitespace
  • Open-weight models enable gradient-based optimization of perturbations
  • Targeted adversarial examples with few token changes

Typographic & Visual Attacks

  • Text overlays in images instruct multimodal models
  • Homoglyph attacks: visually identical characters from different Unicode blocks
  • Adversarial patches on physical objects
  • Cross-modal injection: image payload affects text generation

Multimodal Attack Surfaces

As models process text, images, audio, and video, each modality becomes an attack vector. Adversarial audio embeds hidden voice commands. Adversarial images inject instructions. Cross-modal attacks exploit the interaction between modalities.

llm-security / 09
RISK

Output Safety: Hallucination & Unsafe Code

Hallucination in Critical Contexts

  • Legal: fabricated regulations or compliance standards
  • Medical: invented drug interactions, diagnostics
  • Security config: hallucinated API params, crypto settings
  • Citations: fake sources that create false trust

Unsafe Code Generation

  • ~40% of generated code contains ≥1 vulnerability
  • SQL injection, XSS, command injection, path traversal
  • Deprecated crypto (MD5, DES), hardcoded secrets
  • Missing input validation, dependency typosquatting

Agentic Amplification

When models execute code, call APIs, or modify files, output failures have direct operational consequences. Hallucination + tool access = the model acts on its misunderstanding. Unintended actions, escalation chains, and persistent changes make agentic deployments highest-risk.

DEFENSE

Input & Output Filtering

Input Defenses

  • Keyword/regex matching: catches low-effort injection
  • Perplexity filtering: flags gibberish-like GCG suffixes
  • Input sanitization: strip zero-width chars, normalize encoding
  • Length/structure validation: reject suspicious patterns

Output Defenses

  • Content classifiers: secondary model checks for harmful output
  • PII detection/redaction: automated scanners before delivery
  • Code security scanning: Semgrep, CodeQL, Bandit on generated code
  • Factuality checking: verify against known sources

Fundamental Limitation

Input filtering is inherently fragile: for every filter rule, there exists an evasion technique. Filters should be one layer in a defense-in-depth strategy, never the sole defense.

DEFENSE

Guardrail Frameworks

NVIDIA NeMo Guardrails

  • Open-source; rails defined in Colang DSL
  • Topical, safety, and security rails
  • Interposes between user and model
  • Pre- and post-inference checks

Meta Llama Guard

  • Fine-tuned LLM for safety classification
  • Configurable taxonomy of safety categories
  • Evaluates input-output pairs
  • Llama Guard 3: multi-category, customizable

Guardrails AI

  • Python framework with validators
  • Structural (JSON, regex), semantic (topic, factuality), safety (toxicity, PII)
  • Corrective actions: regenerate, edit, reject

Azure AI Content Safety

  • Cloud service for text and image moderation
  • Violence, sexual content, self-harm, hate speech categories
  • Integrates with Azure OpenAI Service

DEFENSE

Constitutional AI & RLHF for Safety

Constitutional AI (Anthropic)

  • Model critiques and revises its own outputs against a constitution
  • Revised outputs become training data
  • RLAIF: reinforcement learning from AI feedback
  • Principles are explicit and auditable
  • Reduces human exposure to harmful content

Safety-Specific RLHF

  • Human judgments on safe vs. unsafe responses
  • Reward model encodes safety preferences
  • Tension: over-refusal degrades helpfulness
  • DPO, KTO, IPO as alternatives to full RLHF pipeline
  • Calibrating safety vs. utility is the core challenge

Neither Is Bulletproof

Constitutional principles can conflict (helpfulness vs. safety). RLHF coverage depends on the training distribution. Both reduce but do not eliminate jailbreaking. No single training-time defense is sufficient.

DEFENSE

Red Teaming

Manual Red Teaming

  • Human creativity and adversarial thinking
  • Diverse teams: security, social engineering, domain experts, multilingual
  • Discovers novel attack patterns
  • Cannot scale to cover all inputs

Automated Red Teaming

  • Red model generates adversarial prompts
  • Target model processes them
  • Judge model evaluates for violations
  • Scales far beyond manual testing

Tools

  • Garak: open-source LLM vulnerability scanner
  • PyRIT: Microsoft's red-teaming toolkit
  • Multi-turn orchestrators, prompt converters
  • Continuous red teaming in production

Continuous, Not One-Time

The threat landscape evolves constantly. Red teaming must be ongoing, with an up-to-date library of attack techniques tested regularly against deployed models.

DEFENSE

Architectural Defenses

Privilege Separation

  • Principle of least privilege for model tools
  • Read-only access where writes are unnecessary
  • Draft actions vs. execute actions
  • Smaller blast radius if compromised

Sandboxing & Isolation

  • Run generated code in containers, VMs, WASM
  • Network isolation for untrusted tool calls
  • Time-limited execution environments
  • Contain effects of malicious output

Human-in-the-Loop

  • Require approval for high-stakes actions
  • Financial transactions, data deletion, external comms
  • Adds latency but provides critical safety net

Multi-Model Pipelines

  • Separate models for generation, safety, fact-checking
  • Jailbreak of one model caught by another
  • Input/output separation via structured prompting

EVALUATION

Frameworks & Bug Bounties

MITRE ATLAS

  • Extension of ATT&CK for AI systems
  • Adversarial techniques by tactic: recon, access, execution
  • Common vocabulary for AI-specific threats

Automated Scanners

  • Garak: probes, generators, detectors for any text API
  • PyRIT: orchestrators, scorers, converters
  • Integrated into CI/CD for continuous evaluation

Bug Bounty Programs

  • OpenAI: API vulns, plugin security, jailbreaks — $200–$20,000
  • Google VRP: extended to AI safety issues in Gemini
  • Anthropic: responsible disclosure for novel jailbreaks and systemic vulns
  • Challenge: LLM vulns are often inherent to the architecture, not patchable bugs

REGULATION

EU AI Act & NIST AI RMF

EU AI Act

  • Risk-based framework: prohibited, high-risk, general-purpose
  • General-purpose AI with systemic risk: evaluations, red teaming, incident reporting
  • Cybersecurity requirement covers AI-specific attacks
  • Transparency: AI content must be labeled
  • Phased implementation from August 2024

NIST AI RMF (AI 600-1)

  • Voluntary framework; de facto US standard
  • Four functions: Govern, Map, Measure, Manage
  • AI 100-2: adversarial ML taxonomy and mitigations
  • Companion resources for specific sectors

Other Jurisdictions

US EO 14110: red-teaming mandate, compute-threshold reporting. UK AISI: pre-deployment testing of frontier models. China: security assessments and content compliance for generative AI services.

PRACTICE

Security Recommendations

Application Developers

  • Assume the model can be jailbroken
  • Defense in depth: input + output filters + guardrails + HITL
  • Least privilege for all model tools
  • Treat model output as untrusted input
  • Log all interactions for audit

Model Providers

  • Comprehensive red teaming (GCG, many-shot, multi-turn, encoding)
  • Continuous monitoring as threats evolve
  • Publish safety evaluations
  • Bug bounties and responsible disclosure
  • Invest in instruction hierarchy training

Organizations

  • Threat model specific deployments
  • Classify data sensitivity
  • Establish incident response plans
  • Train developers on LLM-specific risks
  • Comply with EU AI Act, NIST RMF

OUTLOOK

Emerging Threats & Defenses

Evolving Attacks

  • Multimodal attacks: image/audio/video injection surfaces
  • Fine-tuning attacks: ~100 examples can degrade safety
  • Agent-specific: tool-use manipulation, persistence, multi-agent propagation
  • Compositional: chaining harmless requests for harmful outcomes

Advancing Defenses

  • Instruction hierarchy training: privilege-aware models
  • Mechanistic interpretability: detecting deceptive reasoning
  • Formal verification: provable safety properties (early-stage)
  • Hardware TEEs: secure enclaves for model inference

The Fundamental Tension

The properties that make LLMs useful—instruction following, generalization, flexible input processing—are the same properties that make them vulnerable. Prompt injection is not a bug; it is a consequence of the architecture. Security must be designed around this reality.

SUMMARY

Key Takeaways

Attack Landscape

  • Prompt injection (direct + indirect) is the defining LLM vulnerability
  • Jailbreaking techniques evolve faster than defenses
  • Data poisoning and sleeper agents threaten model integrity
  • Information leakage creates privacy and compliance risk
  • Agentic deployments amplify all output risks

Defense Principles

  • Defense in depth: no single technique is sufficient
  • Least privilege: minimize model permissions and tools
  • Untrusted output: treat all model output as potentially harmful
  • Continuous evaluation: red-team regularly as threats evolve
  • Regulatory compliance: EU AI Act, NIST RMF, OWASP Top 10

The most robust LLM deployments combine training-time alignment, inference-time guardrails, architectural defenses, and continuous monitoring—while accepting that no defense is complete.
