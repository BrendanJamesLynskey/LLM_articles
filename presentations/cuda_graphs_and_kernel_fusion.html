CONTEXT

The Decode Overhead Problem

LLM decode runs the entire model for one token. The compute density is very low, and per-launch overheads add a constant cost that dominates at small batch sizes.

Per-Launch Cost

  • CPU dispatch: a few µs
  • PCIe transmission
  • GPU command processor setup
  • 5–50 µs overhead per kernel

Per-Token Reality

  • 500–1000 kernel launches per token
  • 10–50 ms of pure overhead
  • Comparable to actual compute time
  • Eats your latency budget

GRAPHS

What CUDA Graphs Are

The Mechanism

  1. Capture sequence of operations once
  2. Instantiate the graph
  3. Replay with single launch call

Per-replay cost is one API call instead of hundreds.

The Speedup

  • 5–10× lower per-step CPU overhead
  • 20–40% lower per-token latency in low-batch decode
  • Free win for repeated workloads

Capture Modes

  • Stream capture: put stream into capture mode, run code as normal — easiest
  • Explicit construction: build graph node-by-node — more powerful, more code

Available Since

CUDA 10 (2018). Mature, well-supported across all modern GPUs.

cuda-graphs / 03
CHALLENGE

Why Inference Makes It Hard

Dynamic Properties

  • Variable sequence lengths
  • Continuous batching changes batch size
  • KV cache grows by 1 per step
  • Each shape needs its own graph

The Workaround

  • Capture multiple graphs — one per common shape
  • Batch sizes 1, 2, 4, 8, 16, 32, 64
  • Pad requests up to nearest captured size
  • Trade some wasted compute for dispatch overhead savings

FUSION

Kernel Fusion

The Idea

  • Combine multiple small kernels into one larger kernel
  • Single launch instead of many
  • Intermediate values stay in registers/shared memory
  • Eliminates memory round-trips

Classic Examples

  • LayerNorm + linear projection
  • Attention + softmax + output projection (FlashAttention)
  • FFN gate + up + activation + down
  • RMSNorm + residual

Static vs Dynamic

  • Static: compile-time fusion (TensorRT, ONNX RT)
  • Dynamic: JIT fusion (torch.compile, XLA)
  • Both common in production

The Triton Path

Most modern fused kernels are written in Triton. The compiler-driven block-level model makes fusion natural and the resulting kernels are competitive with hand-CUDA.

RECIPE

The Standard Inference Stack

1. Library GEMMs

  • cuBLAS / CUTLASS for largest matmuls
  • Hand-tuned for the hardware
  • Last few % matter at scale

2. Triton Fused Kernels

  • Everything else
  • Aggressive fusion of small ops
  • FlashAttention, RMSNorm, RoPE, etc.

3. CUDA Graphs

  • Amortize dispatch overhead
  • Per-batch-size graphs captured at startup
  • Single replay per token

PITFALLS

Common Bugs

Capture-Replay Mismatch

  • Stream capture records pointers
  • Pointers updated between replays cause wrong results
  • Use placeholder + node parameter updates

Over-Capture

  • Including host computation in graph
  • Including host synchronization
  • Defeats the entire purpose

Bad Fusion

  • Fusing memory-bound and compute-bound kernels
  • Register pressure causing spills
  • Worse than the unfused version

ADJACENT

Beyond Graphs & Fusion

Complementary Techniques

  • Persistent kernels — run for entire workload, no per-step overhead
  • Speculative decoding — reduce step count
  • Continuous batching — amortize over many requests

The Trend

  • Algorithmic optimizations becoming dominant
  • Kernel-level work mostly done
  • Inference engine differentiation moving up the stack

SUMMARY

Table Stakes

What’s True in 2026

  • CUDA Graphs and kernel fusion are standard in every serious inference framework
  • Together they often deliver 1.5–3× throughput vs unoptimized
  • Cost nothing in model quality
  • The big wins are captured; remaining gains are algorithmic

Practitioner’s Lesson

Use a framework that supports both natively (vLLM, TensorRT-LLM, SGLang). Trust the defaults. The optimization decisions that matter at the application level are about batching, caching, and request scheduling — not kernel-level concerns. Those are solved at the framework layer, and that’s where they should stay.
